# 決済システム定義一覧

最終更新: 2026-03-19

各決済システムの公式定義、ステータス、CSVカラムをまとめる。
決済データ（収集）の共通カラム定義を設計するための基礎資料。

---

## 共通カラム定義（確定版 19列）

全決済ソースのCSVを、この19列に正規化して `【アドネス株式会社】決済データ（収集） / 決済データ` タブに書き込む。

| # | カラム名 | 意味 | なぜ必要か | 入るソース |
|---|---|---|---|---|
| 1 | 課金ID | 各システム内の取引一意キー | 重複判定、返金との突合、経理追跡 | 全ソース |
| 2 | 参照システム | どの決済システムか | ソース判別、障害時の切り分け | 固定値で全ソース |
| 3 | イベント | 処理の種類（売上/返金/チャージバック等） | UnivaPayで同一課金IDの複数行を区別。他ソースは空欄 | UnivaPayのみ |
| 4 | イベント日時 | その処理が実行された日時 | 日別集計の基準、広告費との期間突合 | 全ソース |
| 5 | イベント金額 | その行の金額（返金なら返金額） | 売上/返金/チャージバック各々の金額把握 | 全ソース |
| 6 | 課金ステータス | 取引全体の最終状態 | 有効な取引の判定 | 全ソース |
| 7 | 商品名 | 購入された商品（CSVの生の値） | 商品別売上、ファネル分析、LTV | UnivaPay/MOSH/INVOY/UTAGE |
| 8 | メールアドレス | 顧客キー（最優先） | CDP突合、全視点での顧客特定 | UnivaPay/日本プラム/MOSH/CREDIX/UTAGE |
| 9 | 姓 | 顧客の姓 | 顧客マスタとの整合 | 分割できるソースのみ |
| 10 | 名 | 顧客の名 | 同上 | 同上 |
| 11 | 名前（原本） | CSVの生の人名（分割前） | 姓名分割の検証、元データ保全 | 全ソース |
| 12 | フリガナ | カタカナ表記 | きらぼし銀行での顧客突合、読み仮名 | 日本プラム/きらぼし |
| 13 | 電話番号 | 顧客キー（補助） | メールがないソースでの顧客特定 | UnivaPay/CREDIX/UTAGE |
| 14 | カードブランド | visa/master等 | 決済分析 | UnivaPayのみ |
| 15 | 課金タイプ | 一括/リカーリング | 継続課金の追跡、MRR | UnivaPayのみ |
| 16 | 支払回数 | 分割の場合の回数 | 分割払いの進捗管理 | 日本プラム/MOSH/CREDIX |
| 17 | 定期課金ID | サブスク・リカーリングの追跡キー | 同一契約の複数課金を紐付け | UnivaPayのみ |
| 18 | 返金日時 | 返金が成立した日時 | 返金追跡、純売上計算 | UnivaPay/CREDIX |
| 19 | メタデータ（JSON） | 各システム固有の追加情報 | 後から必要になる情報の保全 | UnivaPayのみ |

### 設計方針

- **取引単位**で持つ。UnivaPayだけイベント列で売上/返金/チャージバック等を区別する
- **収集段階ではフィルタしない**。全イベント・全ステータスをそのまま入れる
- **姓名分割**: スペースで分割可能なら姓・名に入れる。不可なら空欄で名前（原本）のみ。正確な分割は加工/CDP突合の責務
- **きらぼし銀行の法人振込**（ユニバペイキャスト等）のみ対象外リストで除外
- **UTAGE売上一覧の金額はUnivaPayと重複する**。登録経路は `UTAGE補助` タブの補助データとして扱う
- **日本プラムに商品名はない**。メールアドレスで顧客マスタと突合して補完するのは加工の責務

### 書き込み先

- シート: `【アドネス株式会社】決済データ（収集）`
- タブ:
  - `決済データ`（19列）
  - `UTAGE補助`（8列）
  - `データソース管理`（12列）
  - `データ追加ルール`（3列）
  - `運用監査`（日次到着確認 / 未解決異常 / 月次照合 / 容量監視を集約）

### 運用ガードレール

- 書き込み前に `決済データ` と `UTAGE補助` のヘッダーを検証し、列ずれがあれば取り込みを停止する
- `file_id / file fingerprint / 行 signature` の3段で重複を防ぐ
- 既知の対象外ファイル（受講生サイト登録CSV、`3/18なし`、`データなし.txt` など）は main に入れず state にだけ残す
- ヘッダーは正しいがデータ行がない CSV は `0件データ` として正常処理する
- `取込失敗` は再試行対象に残し、成功後に異常ログから解消する
- `データソース管理` には `ステータス / 最終同期日 / 最終同期行数 / エラー数` を反映する
- `recent_scanned_folders` に直近フォルダの実観測を保存し、`その日のCSVが入っているか` を実観測で判定する
- `運用監査` には最新日付の到着確認、直近7日の処理状況、未解決異常、容量、月次照合をまとめて出す
- `運用監査` の月次照合は `フォルダ月 x ソース` 単位で `converted行数 = 追加行数 + duplicate除外行数` を確認する
- `UTAGE補助 / データソース管理 / データ追加ルール / 運用監査` の書式は同期時に再適用し、1行目は列幅拡張を優先しつつ必要な範囲だけ文字サイズを下げて1行表示を維持する

## 1. UnivaPay

### 概要
- クレジットカード決済代行サービス
- アドネスのメイン決済システム（全体の約88%）
- PCI DSS準拠

### 公式用語定義（公式ドキュメントより）

| 用語 | 公式定義 |
|---|---|
| 課金 | 商品やサービスの購入（利用）料金を課すること |
| 都度課金 | 一度きりの決済を行う最も基本的な課金方式 |
| 定期課金 | 定期的かつ自動的に課金する機能 |
| リカーリング課金 | リカーリングトークンに対して課金を行うこと |
| オーソリ | 指定した金額の決済が可能なクレジットカードに対し、与信枠を仮押さえすること |
| 仮売上 | オーソリと同義 |
| キャプチャ | オーソリ（仮売上）処理を行った課金情報に対して、売上確定処理（実売上）を行うこと |
| トークン | カード情報を復号できないように置き換えた文字列 |
| リカーリングトークン | 繰り返し課金に利用するために、消費者のクレジットカード情報をトークン化したもの |
| チャージバック | カード利用者が利用代金の支払に同意しないため、カード会社が加盟店に対して支払いの取り消しまたは返金を要求すること |
| 返金 | 消費者にお金を返す処理 |

### 課金ステータス（課金全体の最終状態。1つだけ）

| ステータス | 公式定義 |
|---|---|
| 成功 | 正常に決済が完了した |
| 処理待ち | カード結果待ち / オンライン決済アプリ処理待ち / コンビニ・銀行振込は入金待ち |
| オーソライズ済 | 仮売上（決済金額の与信枠をおさえている） |
| 失敗 | 決済が失敗している |
| エラー | 決済サーバーやデータベースエラーなどの例外的なエラー |
| キャンセル済 | クレジットカードは仮売上キャンセル、その他は課金申込キャンセル |

### イベント（個別処理の履歴。1つの課金に複数紐づく）

| イベント | 公式定義 |
|---|---|
| 売上 | 課金処理成功 |
| 売上失敗 | 課金処理失敗 |
| キャンセル | 加盟店による決済中止 |
| 返金 | 返金処理成功 |
| 赤伝返金 | （公式ドキュメントに定義なし。実データには存在） |
| チャージバック | カード利用者が支払いに同意せず、カード会社が取り消し/返金を要求 |
| オーソリ | 仮売上処理成功 |
| キャップチャー | 仮売上から実売上へ変更 |
| 3-Dセキュア認証 | 認証処理成功 |
| 3-Dセキュア認証失敗 | 認証処理失敗 |
| 3-Dセキュア認証タイムアウト | 認証タイムアウト |
| 3-Dセキュア認証処理待ち | 認証処理待ち |
| CVVオーソリ | CVV認証成功 |
| CVVオーソリ失敗 | CVV認証失敗 |
| リカーリングトークン発行 | トークン発行 |
| 赤伝返金失敗 | 赤伝返金処理失敗 |

### 重要な構造的注意

- **CSVの1行 = 1イベント**（1決済ではない）
- 同じ課金IDに対して「売上 → 返金」のように複数行が出る
- **イベント金額**: 各イベント時点での金額（返金なら返金額）
- **課金金額**: 課金作成時の元の金額
- 返金の場合、イベント金額 = 返金額、課金金額 = 元の決済額
- 収集シートではシンプルさ優先で **イベント金額だけ保持** し、`課金金額 / 返金金額 / 返金ステータス / 返金理由` は共通カラムに残さない

### 定期課金ステータス

| ステータス | 説明 |

| 待機中 | 初回課金待機状態 |

| 永久停止 | 完全停止・再開不可 |
| 作成失敗 | 初回課金失敗で非稼働 |
| 完了 | 指定回数終了または分割払い成功 |

### 返金ステータス

| ステータス | 説明 |
|---|---|
| 成功 | 返金処理完了 |
| 失敗 | カード会社側で調査中 or チャージバック確定 |
| エラー | 金額超過/通信エラー/一部返金非対応/期限超過 |
| 保留 | 海外接続時は1〜2週間かかる場合がある |

- PayPay Online: 1年

## 2. MOSH

### 概要
- オンラインサービス販売プラットフォーム
- 予約・決済・サブスクリプション対応
- Stripe経由の決済

### 送金ステータス（管理画面の絞り込みで確認）

MOSHからアドネスへの送金状態:

| ステータス | 説明 |
|---|---|
| 送金待ち | まだ送金されていない |
| 送金済み | アドネスの口座に送金完了 |
| キャンセル済み | キャンセルされた |
| 支払い待ち | 顧客からの支払いを待っている |
| 支払い期限切れ | 支払い期限を超過 |

### 売上ステータス（ヘルプFAQ公式定義）

サービス提供の進捗状態:

| ステータス | 公式定義 |
|---|---|
| 予約済み | レッスン日時の決済済みで、レッスンがまだ未完了な状態 |
| 開催済み | レッスンが完了し、売上が立った状態 |
| キャンセル | レッスン日時の決済済みで、キャンセル（返金）となった状態 |

### 売上の反映スケジュール
- 申し込み時点で顧客のカード決済は完了
- 売上管理トップの「売上」はサービス開催後1時間程度で反映
- 月額サブスク・回数券は購入後10分以内
- 売上履歴一覧は申し込みから10分程度で反映
- 当月出金予定額はサービス開催後、決済日から4日経過した売上から反映

### 入金スケジュール
- 入金日: 毎月25日（土日祝は翌営業日）
- 送金開始日: 入金日のおよそ2日前（毎月23日頃）
- 含まれる売上: 前月23日前後〜当月23日前後の開催済み売上
- 振込名義: 「●● ストライプジャパン（カ」

### CSVカラム（公式ヘルプ + 実データ）

公式ヘルプの項目:

1. サービスID
2. サービスタイトル（= 商品名）
3. ゲストID
4. E-mail
5. お名前
6. クリエイターID
7. 申し込み日
8. 支払い方法
9. クーポン名
10. 決済金額
11. 売上総額
12. 決済ステータス（支払い済み/キャンセル）
13. 分割払い決済金額
14. 分割回数
15. 分割ステータス
16. キャンセル日時（月をまたぐキャンセル時の正確な会計処理用）
17. サービス提供価格（税込）（クーポン適用前の本来の価格）
18. クーポン割引額
19. 決済額（税抜・消費税）
20. 対象期間（「2025年7月分」等）

実CSVの追加カラム:

- 注文内訳

## 4. 日本プラム

### 概要
- ショッピングローン（分割払い）サービス
- クレジットカード決済ではなく、信販会社による立替払い
- Webクレジット申込システム「ピクレット」
- 商品名はCSVに存在しない（金額と支払回数のみ）

### フロー
1. 申込入力（タブレットでペーパーレス）
2. 書類撮影（売買契約書・身分証明書）
3. 申請確認（バリデーション機能）
4. 審査・回答（30分〜1時間。Webでチャットも可能）
5. ベリフィケーション（本人確認電話）
6. 立替実行（日本プラムがアドネスに立替金を送金）

### 状態（CSVの「状態」列。全期間CSVから確認）

| 状態 | 説明 |
|---|---|
| 新規（送信済） | 申込が送信された初期状態 |
| 受付済 | 申込が受け付けられた |
| 受付済（要確認） | 申込受付だが確認事項がある |
| 審査依頼済 | 審査に出された |
| ベリ予定 | ベリフィケーション（本人確認電話）の予定がある |
| ベリ予定(新規受付) | 新規受付でベリ予定 |
| 最終承認 | 審査通過・立替確定 |
| (客)再申込み待 | 顧客の再申込みを待っている |
| キャンセル YYYY/MM/DD | キャンセル（日付入り） |
| お申込キャンセル YYYY/MM/DD | 申込段階でのキャンセル（日付入り） |

### 審査結果（管理画面で確認）
- 可決
- （他の値は未確認。否決等がある可能性）

### 契約詳細で見える項目（管理画面）
