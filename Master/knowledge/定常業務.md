## 秘書のChrome MCPテスト（初回・再設定後）

日報自動入力は Mac Mini 上で秘書（Claude Code CLI + Chrome MCP）がブラウザで Looker Studio を開いて実行する。以下を満たしたうえで1回動作確認する。

### 前提条件（Mac Mini）

| 項目 | 確認方法 |
|------|----------|
| `~/.claude-secretary/` が存在 | `ls -la ~/.claude-secretary` |
| Chrome が起動可能 | 手動で「Google Chrome」を起動できること |
| Claude Code CLI に `--chrome` 対応 | Orchestrator が日報・Looker CSV 実行時に `--chrome` を付与（scheduler.py で use_chrome=True） |

### テストのやり方

1. **自動実行で確認**: 翌朝 08:40 の `daily_report_input` で成功すればOK。失敗時は LINE に通知が届く。
2. **手動で試す**: Mac Mini で秘書（LINE local agent）が動いている場合、LINE で「日報入れて」と送ると Claude Code + Chrome MCP で日報入力が実行される。成功すれば Chrome MCP テスト完了。

### よくある失敗と対処

- **Chrome MCP に接続できません** → Chrome を起動し、Claude in Chrome 拡張が有効か確認。必要なら Chrome を再起動。
- **Looker Studio の Google ログインが切れています** → 自動復旧あり。秘書は `kohara.kaito@team.addness.co.jp` を第1候補としてログインを試行。2段階認証はLINE通知→甲原のiPhoneで承認。認証情報: `System/credentials/kohara_google.txt`。
- **セッション維持（予防）** → 毎朝07:00に `looker_session_keepalive` がCDP経由でLooker Studioを開いてセッションをリフレッシュ。これにより通常はログイン切れが発生しない。

### Looker Studio運用ルール

- 正本アカウントは `kohara.kaito@team.addness.co.jp`
- 既に開いている認証済みタブを再利用する。ページを何度も開き直さない
- `u/1` など account index 付き URL を新規に踏まない。必要なら account index を外した URL を使う
- `アクセスが拒否されました` やアカウント選択画面が出たら、再ログインを繰り返す前にアカウント文脈ズレを疑う
- 同じレポート内の別ページ移動は、同一タブ内のページ遷移を優先する

## 日報入力

### 自動実行（毎朝8:40）

Orchestrator `daily_report_input` タスクが毎朝8:40に自動実行。秘書がClaude Code経由でLooker Studioからデータ取得→日報シート書き込み→LINE報告まで一気通貫で行う。手動操作不要。

- **実行時刻**: 毎朝 08:40（config.yaml `daily_report_input`）
- **事前チェック**: 08:25 に `oauth_health_check` が Google + Claude Code OAuth トークンを検査・自動リフレッシュ
- **実行者**: 秘書（Claude Code CLI + Chrome MCP）
- **対象日**: 前日分
- **タイムアウト**: 10分（テキスト抽出ベースで高速化、正常時2-4分で完了）
- **最大ターン数**: 20（事前計算でターン消費を半減、正常時10-12ターン）
- **事前計算（Python側で実行、CLIの外）**: サブスク新規会員数のカウント、日報シートの対象列特定
- **取りこぼし対策**: cron の短時間 misfire は 30分まで補足。さらに Orchestrator 再起動直後にその日の重要定常を即チェックし、取りこぼしがあれば待たずに補走。平常時も `health_check` が1分ごとに未実行を見て、定刻を過ぎても走っていなければ当日中に1回だけ自動補走
- **リトライ**: 失敗時 60秒後に自動リトライ（最大2回）
- **失敗時**: LINE に通知（エラー種別ごとに具体的な復旧手順を案内）
- **完了検証**: 09:20 に `daily_report_verify` がシートの実データを自動確認。未入力があれば LINE で通知
- **日報リマインド**: 平日 12:00/19:00 に `daily_report_reminder` がチームメンバーの未記入を検出 → 赤ハイライト + **広告チーム全体** LINEグループに通知

- **ハイライト**: 未記入セル → 赤背景、記入済み → 白に戻す（チェック対象セルのみ操作）

### プリフライトチェック（自動実行前に確認される項目）

| チェック項目 | 失敗時の動作 |
|---|---|
| Claude Code CLI 存在 | LINE で通知、タスク中止 |
| `~/.claude-secretary` 存在 | LINE で通知、タスク中止 |
| Claude Code OAuthトークン | 期限切れ or 1時間以内 → CLI起動で自動リフレッシュ。**このとき `ANTHROPIC_API_KEY` は除外して OAuth 専用で起動すること**。残っていると API key 認証に切り替わり、日報・Looker CSV・DS.INSIGHT などの定常業務が失敗する |
| Claude Chrome MCP bridge | Claude の feature flag が remote bridge に戻っていたら、起動前に `tengu_copper_bridge=false` / `tengu_ccr_bridge=false` へ補正してローカル native host 経由に固定する。これがずれると `--chrome` タスクが拡張未接続扱いで失敗する |
| Chrome CDP + Claude in Chrome 拡張 | `127.0.0.1:9224` の CDP 応答確認後、拡張の OAuth を `~/.claude-secretary/.credentials.json` と同期して `https://clau.de/chrome/reconnect` を叩く。ここで失敗したら本実行前に中止 |
| Claude ブラウザ権限 | `-p --chrome` の自動実行では、navigate / screenshot などが許可待ちで拒否されることがある。定常業務では `--allow-dangerously-skip-permissions --dangerously-skip-permissions` を付けてブラウザ操作を自動許可する |
| Claude Chrome ログインアカウント | Chrome 拡張の OAuth が `~/.claude-secretary` と別アカウントになると、bridge 上で拡張一覧が空になり `No Chrome extension connected.` で失敗する。起動前に拡張 storage の `accessToken` / `refreshToken` / `tokenExpiry` を秘書用 credentials に同期し、`https://clau.de/chrome/reconnect` を踏んで再接続する |
| Chrome 起動中か | 未起動なら自動起動（10秒待機）。起動失敗なら通知 |

#### DS.INSIGHTメール配信スケジュール（公式仕様）

| 種別 | 配信日 | 内容 |
|------|--------|------|
| **Basic メールアラート** | 毎週火曜（変化がある場合のみ） | 過去に分析したKWの検索ボリューム変化・属性変化・共起KW変化 |

- **BasicはKW指定不可**: 過去の分析履歴から自動選定。登録なしで自動配信

## Looker Studio CSVダウンロード

### 自動実行（毎日11:30）

Orchestrator `looker_csv_download` タスクが毎日11:30に自動実行。秘書がClaude Code経由でLooker Studioを開き、CSVをダウンロード。12:00の `kpi_daily_import` がこのCSVを自動処理する。

- **実行時刻**: 毎日 11:30（config.yaml `looker_csv_download`）
- **対象日**: 直近7日間の不足分を自動バックフィル（前々日だけでなく、過去の失敗分も自動リカバリー）
- **タイムアウト**: 8分（複数日ダウンロード時は自動延長）
- **バックフィル**: CSVフォルダをスキャンし、不足日付を検出 → 1セッションでまとめてダウンロード → ダウンロード後に残った不足分はLINE通知

## 自動修復・予防監視

定常業務（日報入力、CSV取得等）の失敗を自動検知し、必要なら日向にコード修正を渡す仕組み。

現在は 2026-03-13 から日向を一時停止しているため、異常検知は継続するが、日向への修復タスク投入は止めている。優先順位は `Codex -> 秘書 -> 日向`。

### フロー（イベント駆動）

```
定常業務タスク実行完了
  → _execute_claude_code_task 内でメトリクス記録（SQLite task_log）
  → _check_task_health() で健全性チェック
  → 異常あり
    - 日向稼働中: hinata_tasks.json に command_type="repair" で投入
    - 日向停止中: repair 投入はせず、各タスクの通常通知とログで扱う
  → 日向稼働中のみ、日向が修復実行（Claude Code, --chrome なし）→ git push → Slack 事後報告
```

正常時の API 消費はゼロ（旧 `repair_check` の定期スキャン方式は廃止済み）。

### 検知条件

| 条件 | 閾値 | 動作 |
|------|------|------|
| 連続失敗 | 2回以上 | 修復タスク投入 |
| 実行時間超過 | duration > timeout × 0.8 | 予防修復タスク投入（次回タイムアウトの予兆） |

### 修復対象タスク

日報自動入力、Looker CSVダウンロード、KPI日次インポート、日報入力検証、シート同期、KPI夜間キャッシュ

### 修復対象外（コード修正では直らない）

認証エラー、Chrome MCP 接続エラー、Googleログイン切れ、APIレート制限、Claude Code クレジット不足
→ これらは Slack で「甲原さんの確認が必要です」とエスカレーション

### 安全策

- **修復対象ファイルの制限**: scheduler.py, tools.py, sheets_manager.py, notifier.py のみ
- **構文チェック必須**: `py_compile.compile(path, doraise=True)` 通過後にコミット
- **重複防止**: hinata_tasks.json の pending チェック + 24時間クールダウン
- **エスカレーション**: 修復不可なら Slack で「甲原さんの確認が必要です」と報告

## 完了報告（LINE）

**定常業務（🔄 ステータス）を実行・完了したら、必ずLINEで完了報告を送信する。これは省略不可。**

### 対象タスク

Addnessの甲原海人の定常業務（🔄）。代表例:

- 報告シート記入/確認する
- 全体数値報告
- スキルプラス広告コースに来ている質問回答
- メール確認/返信
- 日報入力

上記に限らず、定常業務を完了した際は全て報告対象。

### 報告の実行方法

```bash
# 完了時刻を含める場合は事前に変数に代入する
current_time=$(date +%H:%M)
python3 System/line_notify.py "メッセージ...完了時刻: ${current_time}"
```

**注意**: heredocを使う場合は `<<EOF`（クォートなし）を使うこと。`<<'EOF'` だと `$(date)` 等のコマンド置換が展開されずそのまま文字列として送信される。

### 報告内容のフォーマット

```
✅ 定常業務完了: {タスク名}
━━━━━━━━━━━━
{主要な数値（あれば）}
{異常値フラグ（あれば）}
━━━━━━━━━━━━
完了時刻: {HH:MM}
```

#### 数値がある場合の例

```
✅ 定常業務完了: 日報入力
━━━━━━━━━━━━
集客数: 1,234人
個別予約数: 89件
着金売上: ¥12,345,678
会員数: 5,432人（解約: 12人）
サブスク新規: 8人
━━━━━━━━━━━━
⚠️ 集客数が前日比 -30% で要確認
━━━━━━━━━━━━
完了時刻: 14:30
```

#### 数値がない場合の例

```
✅ 定常業務完了: メール確認/返信
━━━━━━━━━━━━
対応件数: 5件（うち要フォロー: 1件）
━━━━━━━━━━━━
完了時刻: 09:15
```

### 異常値の検知基準

以下に該当する場合、報告に `⚠️` フラグを付ける:

- 集客数が前日比 ±20% 以上の変動
- 解約数が通常の2倍以上
- 着金売上が前日比 -30% 以上
- その他、明らかに通常と異なる数値

### 報告のタイミング

- タスク完了の直後に送信する
- 複数の定常業務を連続で実行した場合は、まとめて1通で報告してよい
