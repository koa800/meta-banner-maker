# 実行主体別の閲覧・出力ルール

最終更新: 2026-03-17

## 情報ラベル

- 所有元: internal
- 開示レベル: role-limited
- 承認必須: conditional
- 共有先: 僕 / 上司 / 並列 / 直下

このファイルは、実行主体ごとに「何を読めて、どう出力し、どこまで送信できるか」を定義する正本です。
ここでいう実行主体は human も AI も含みます。AI はオンライン上の人として扱います。

## 目的

- 共通の脳を前提にしつつ、実行主体ごとに閲覧範囲と出力範囲を制御する
- `Master/` の知識を誰がどこまで使ってよいかを、運用ルールとして固定する
- 将来的に `System/registries/agent_registry.json` や shared context 分離へ反映できる形にする

## 共通構成

全員に対して、出力の骨組みは共通にする。

- `結論`
- `理由 / 背景`
- `次アクション`

## 伝え方の判断順

構成は共通にし、表現は相手区分で機械的に固定しない。
その都度、次の順で伝え方を決める。

1. 相手は今、何を知りたいか
2. 相手は次に何をする必要があるか
3. 誤解すると何がまずいか
4. その相手は今どこまで前提を持っているか

## 情報ラベル

AI 内部では英語キーで持ち、会話や文書では日本語で説明する。
優先するのは「AI が誤解しないこと」です。

### `owner` = `所有元`

- `self`
  甲原さん専用の脳資産。外に出す前提なし
- `internal`
  社内運用用。社内ルールに従って共有する
- `external-ready`
  外部に渡してよい前提で整えた情報

### `disclosure_level` = `開示レベル`

- `self-only`
  自分のみ
- `role-limited`
  相手固定共有
- `task-limited`
  目的限定共有
- `shareable`
  共有可

### `approval_required` = `承認必須`

- `always`
  毎回、甲原さん承認が必要
- `conditional`
  通常は扱えるが、外部共有や重要連絡では承認が必要
- `none`
  既定ルールの範囲なら承認なしで扱える

### `audience` = `共有先`

- `僕`
- `上司`
- `並列`
- `直下`
- `外部`

1つの情報に対して、`["上司", "並列"]` のように配列で持つ。

## ラベル継承

- 原則は `ファイル単位で明示ラベルを持つ`
- ただし legacy が大きい領域では、`path 単位の既定ラベル` を暫定的に認める
- `path 単位の既定ラベル` は `Master/output/README.md` と `Master/addness/README.md` を正本にする
- `learning` と `sheets` のように JSON 構造を変えにくい領域も、README に既定値を置いて path 単位で監査してよい
- ファイルに明示ラベルがある場合は、必ずそちらを優先する
- 高リスク資産と正本ファイルは、既定値だけで済ませず個別ラベルを持つ

## shared context の分離

### `self-context`

- 未整理の思考途中
- 対人の深い見立て
- 未公開の戦略や評価メモ
- 承認前の危険情報
- まだ外に出す形に整っていない判断材料

### `internal-context`

- 社内で共有してよい判断基準
- 社内で共有してよい背景事情
- 実務に必要な進行状況
- 社内で使う操作手順
- 社内で共有してよい数値や方針

### `actor-context`

- 特定の相手との直近やり取り
- その相手にだけ有効な約束や注意点
- その相手への送信権限レベル
- その相手に対する、承認済みの伝え方メモ
- その相手との関係でだけ使う文脈

## 送信権限

送信権限は `相手区分` ではなく `個人` ごとに持つ。
`上司 / 並列 / 直下 / 外部` は参考情報であり、最終的には個人単位で管理する。

### 初期原則

- `僕以外への送信` は原則 `提案 -> 甲原さん承認 -> 送信`
- 権限は固定ではなく、承認率と安定度を見ながら段階的に移譲する
- AI は承認履歴を記憶し、任せてもよさそうな相手に対して自分から昇格提案を出す
- 上司への AI 直接返信は原則しない

### 権限レベル

- `Lv0`
  送信禁止。下書きのみ
- `Lv1`
  毎回承認後に送信可
- `Lv2`
  定型連絡だけ自動送信可
- `Lv3`
  通常連絡まで自動送信可。重要連絡は承認必要

### 重要連絡

次のどれかに当たれば `重要連絡` として扱う。

- 謝罪や訂正が入る
- 相手の感情に強く影響する
- 約束、期限、金額、条件を確定させる
- 対外的な印象や関係性に影響する
- 甲原さん本人の評価や意思として受け取られる
- 一度送ると取り返しがつきにくい

### 昇格ルール

- 昇格は自動ではしない
- AI が昇格提案を出し、甲原さんが `はい` と承認したら、その範囲だけ昇格する
- `定型連絡` は 5 回連続でほぼ修正なしなら昇格提案の目安
- `通常連絡` は 10 回連続で問題なしなら昇格提案の目安
- 事故や大きなズレが出たら、昇格保留か必要なら 1 段階戻す

## 実行主体ごとの基本方針

### `僕`

- 全 context を参照してよい
- 送信権限の制限は置かない
- ただし対外送信や不可逆変更は、別ルールの stop 条件に従う

### `上司 / 並列`

- `internal-context` と、その相手に必要な `actor-context` を参照候補にする
- `self-context` は原則そのまま出さず、必要な場合だけ要約して使う
- 出力は `結論 -> 理由 / 背景 -> 次アクション`
- どこまで具体化するかは、相手が今ほしい粒度に合わせる

### `直下`

- `internal-context` と、その相手に必要な `actor-context` を参照候補にする
- `self-context` は参照しても、そのまま出力しない
- 実行に必要な情報だけを渡す
- 情報を渡しすぎて混乱させない
- 生の `目的.md` `判断軸.md` `BRAIN_OS.md` をそのまま渡す前提にはしない。必要な判断を短く要約した実行用出力へ変換して渡す

### `外部`

- `external-ready` の情報を優先する
- `internal-context` や `self-context` は原則そのまま出さない
- 状況と必要条件だけを渡す
- 具体的な内部事情、人物見立て、未公開の数値や戦略は出さない

## 実装に反映するときの最低要件

- 実行主体ごとに `基本区分` を持つ
- 実行主体ごとに `送信権限レベル` を持つ
- 実行主体ごとに `参照できる context 範囲` を持つ
- 情報ごとに `所有元 / 開示レベル / 承認必須 / 共有先` を持つ
- 曖昧なときは、一番厳しい区分で仮置きしつつ、甲原さんに提案して承認をもらう
