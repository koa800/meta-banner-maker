この `rules.md` は、それらを置き換えるものではなく、成果物運用から増えていくルールの受け皿です。

## ルール追加の基準

- 再発させたくない失敗は、短く禁止形でここに書く
- 勝ち筋が2回以上再現したら、知識ではなく判断ルールとしてここに引き上げる
- 3回以上、人や案件をまたいで使えるなら `Skills/` に昇格する
- 単発事例や仮説が多いものは、`output` や `knowledge` に留める

## ルール化の前提

- まず `事実 -> 観察 -> 解釈 -> 確度` の順で残す
- `確定` と `強い推定` を主に rules 化する
- `仮説` は output や knowledge に残し、次の検証対象として扱う
- 分からないことを、分かったこととして rules に上げない

## 昇格判断の基準

- `output 止まり`: 単発の成果物、または仮説中心で再利用条件がまだ見えていない
- `knowledge 候補`: 事実と観察は残せるが、まだ禁止や推奨に圧縮できない
- `rules 候補`: `確定` または `強い推定` が 2件以上あり、次回の判断を変える禁止・推奨に圧縮できる
- `Skills 候補`: 人や案件をまたいで 3回以上再利用でき、条件と手順を独立して説明できる

迷ったら 1段階下に留める。

## 運用ループ

1. 新しい成果物を `output/` に残す
2. 結果と観察を `knowledge/` に残す
3. 勝ち筋・NGを `rules.md` に抽出する
4. 汎用原理になったら `Skills/` に移す

## 初期ルール

### 推奨

- 言葉の定義が曖昧なまま企画・LP・コピーを作らない
- output には成果物だけでなく、目的・仮説・結果まで必ず残す
- 成功理由は「なぜ効いたか」まで knowledge に圧縮して残す
- 面談定性の勝ち負け比較では、まず `過去の解決策` の埋まり率を見る
- 同一顧客に複数の収録URLがあるなら、1本だけで判断せず行単位で統合して読む
- 教育の10要素は固定順に縛られず、カスタマージャーニー全体で満たす
- 序盤は `発信者の信頼` と `理想の明確化` を重く見る
- 良いオファーは `新規性 / ベネフィシャルさ / 問題解決の信頼` で設計する
- 商品の機能的価値だけで押さず、情緒的価値で心を動かしてから論理で納得させる

### NG

- 成果物だけを残して、結果を残さない
- 失敗を個別案件のまま埋もれさせる
- 汎用化できる原理を `self_clone/` に閉じ込める
- `現在の悩み` の頻出語だけで高LTV顧客像を定義する
- `人生を変えよう` のような抽象度が高すぎるコンセプトを主軸にしない
- 既視感の強い訴求を、そのまま主訴求として繰り返さない
- 商品の機能的価値だけで売ろうとしない
- `必要だが今ではない` を放置したままオファーしない

## Skills 昇格条件

- 甲原固有ではなく、他者にも再利用できる
- 判断条件とやり方の両方が言語化されている
- 3回以上の再利用実績がある
- `knowledge` の事例参照なしでも最低限機能する
