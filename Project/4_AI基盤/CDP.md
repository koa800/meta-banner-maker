## 概要・目的

アドネスの顧客データが複数のGoogleスプレッドシートに散在している状態を解消し、**1つのマスターシートに統合**する。

**目的:**
- 導線分析（どの広告→どのセミナー→どの商品で購入したか）
- セグメント別ナーチャリング（流入経路・ファネル進捗・LTVでグルーピング）
- 顧客解像度の向上（定性情報の一元管理）

**CDPスプレッドシート:** [CDP](https://docs.google.com/spreadsheets/d/1qjU279OVD0i4h2AdQzkYIsZCfA1BeiUKLHNg7i2a2fk/edit)

## 2026-03-13 再設計メモ（次回再開用）

今回の再設計議論では、目的を `CDP構築 -> セグメント分け -> マーケティング施策実行` をシームレスにつなぐことに置いた。最小工数かつ精度の高い運用を優先し、`媒体別数値` `CR別数値` `ユーザーごとの数値` `導線内数値` の4主体で持つ案は採用しない。

### 採用方針

- 正本は `顧客マスタ` `媒体×CR×LPデータ` `導線データ` の3タブに絞る
- `媒体別` と `CR別` は別主体にしない。`媒体×CR×LPデータ` の見方を変えた集計ビューとして扱う
- `顧客マスタ` は今後も施策実行の中心に置く。`今この人に何を打つべきか` がすぐ分かる状態を正とする
- `媒体別タブ` が必要な場合は `raw_Meta` `raw_Google` のような取込入口としてだけ持つ。正本にはしない
- `導線データ` は 1人1行ではなく `1イベント1行` を正とする

### 使う言葉

- `元データ`: 外部システムや入力口から、そのまま入ってくる一覧
- `加工データ`: 元データを、重複除外・整形・用途別に使いやすくした一覧
- `会議で使う数`: 経営会議や日報で最終的に見る数
- `判定ルール`: 何を数えるか、何を除くか、重複をどう扱うか

### KPI基盤づくりの進め方

最初から完成版のスプレッドシートを作るのではなく、各領域ごとに `元データ` `加工データ` `会議で使う数` `判定ルール` を先に固める。

| 順番 | 領域 | 先に決めること | 完了条件 |
|------|------|------|------|
| 1 | 集客 | メール登録 / LINE登録の元データと加工データの役割分担 | `集客数` と `ユニーク集客人数` の定義が言える |
| 2 | 個別予約 | 予約の元データ、重複判定、予約確定の定義 | `個別予約数` の数え方が固定される |
| 3 | 着金売上 | 決済元データ、返金の扱い、着金日基準 | `着金売上` の日次集計ルールが固定される |
| 4 | 広告費 | 媒体ごとの元データ、route の持ち方、税込税抜 | `広告費` の日次集計ルールが固定される |
| 5 | 会員 | 入会、クーリングオフ、中途解約の元データ | `会員純増` の計算式が固定される |
| 6 | 統合 | 上の5領域を束ねて最終的なKPI基盤を作る | 5KPIを同じ基準日で再集計できる |

### 3タブの役割

| タブ | 粒度 | 役割 |
|------|------|------|
| `顧客マスタ` | 1人1行 | 現在地・購入状況・配信可否・施策セグメントを持ち、施策実行の起点にする |
| `媒体×CR×LPデータ` | 1日 × 1 route_key × 1行 | どの入口が伸びているかを見るための集計正本にする |
| `導線データ` | 1イベント1行 | その人が導線内で何をしたかを時系列で持つ |

### 連携キー

- `user_id`: `顧客マスタ` と `導線データ` をつなぐ主キー
- `route_key`: `媒体×CR×LPデータ` と `導線データ` をつなぐ共通キー
- `route_key` の基本形は `媒体|ファネル|CR|LP`
- 例: `Meta|AIメイン|CR_202603_A|LP_240812_A`

### 次回の最初の着手順

1. 既存 `顧客マスタ` を起点にし、追加すべき列を確定する
2. `媒体×CR×LPデータ` の正本カラムを確定する
3. `導線データ` の `event_type` と必須列を確定する
4. `raw_*` タブから `媒体×CR×LPデータ` へ正規化する取り込み方針を決める
5. `導線データ` から `顧客マスタ` の `current_stage` と各日付列を更新するルールを決める

### この時点で決めたこと

- `顧客マスタ` は作り直さない。既存資産を中心に使い、不足列を足す
- `媒体ごとにタブを分ける` のは取込時だけなら許容する
- 分析・セグメント・施策実行に使う正本は、媒体横断で1枚に統合する
- `CPA` や `LP到達に対するオプトイン率` のような計算値は、正本テーブルではなくビュー側で算出する

### 5KPIの正本定義（経営会議・日報・Cursor共通）

現時点では `信頼できる単一のKPI正本` はまだ存在しない。したがって、最初に揃えるべきなのは会議用の清書シートではなく、以下5指標の一次データと正本粒度である。

| KPI | 正の定義 | 正本として数える粒度 | 一次データ | 正本にしないもの |
|------|------|------|------|------|
| `集客数` | 導線上で `lead` が成立した有効件数。オプトイン完了、LINE登録完了、フォーム登録完了など「導線に入った」時点を数える | `導線データ` の `event_type=lead` を 1イベント1行で保持し、集計時に一意化 | UTAGE / Lステップ / Mailchimp / フォームなどの登録イベント | Looker Studio、報告シート、日報、集計済みCSV |
| `個別予約数` | 個別相談の予約が確定した件数 | `導線データ` の `event_type=booking` を 1イベント1行で保持 | UTAGE、Lステップ、面談DB、予約システムなどの予約確定イベント | Looker Studio、報告シート、日報、集計済みCSV |
| `着金売上` | 実際に着金した金額の合計。返金は別管理にし、純額は派生指標で出す | `決済データ` の 1取引1行 | Univapay / MOSH / Invoy / 銀行振込の決済履歴、CS返金管理 | Looker Studio、報告シート、日報、着地予想値 |
| `広告費` | 媒体で実際に消化した費用 | `媒体×CR×LPデータ` の `1日 × 1 route_key × 1行` | Meta / Google / TikTok / X など媒体の生データ | Looker Studio、報告シート、日報、媒体別集計済みシート |
| `会員純増` | `入会 - クーリングオフ - 中途解約` の純増数 | `会員状態データ` の 1会員1状態変化、または `導線データ` の membership event | 会員管理の元データ、商品管理、解約管理、返金管理 | Looker Studio、報告シート、日報、月次まとめ表 |

**列構成（14列）:**

| 列 | 内容 |
|----|------|
| CDPカラム | 外部ソースから同期するカラム名（自動生成/自動計算/手動は含まない） |
| グループ | カラムのグループ名 |
| ソース元 | データソース（プルダウン選択） |
| 優先度 | 同一カラムに複数ソースがある場合の優先順位（1が最優先） |
| スプレッドシートURL | 参照先のURL |
| タブ名 | 参照先タブ |
| 参照先列 | 参照先の列名 |
| 正規化 | 取り込み時の変換ルール |

| ステータス | 正常/未同期/停止（自動更新） |
| 最終同期日 | 同期スクリプトが自動記録（`YYYY/MM/DD HH:MM`） |
| 更新数 | ソーススプレッドシートの参照先列における実データ件数（同期時に自動カウント） |

| メモ | 備考 |

**複数ソース対応:** 同じCDPカラムに複数行を追加し、優先度で順序を指定。同期時は優先度1から順にチェックし、最初に値が見つかったものを採用する。

### CR別データの流入情報同期

`CR別データ` は通常のメール1キー同期ではなく、`cdp_sync.py` の専用ロジックで同期する。`データソース管理` には監視用に3行を持ち、実際の取り込みは以下の条件で処理する。

| CDPカラム | 参照先列 | 突合キー | 入力条件 |
|------|------|------|------|
| `CR名` | D列 `CR名` | メールアドレス → 電話番号 → 名前 | 一意一致した顧客のみ。`登録日時` が新しい値を採用 |
| `CRリンク` | G列 `CRリンク` | メールアドレス → 電話番号 → 名前 | 一意一致した顧客のみ。空欄は上書きしない |
| `遷移先リンク` | I列 `遷移先リンク` | メールアドレス → 電話番号 → 名前 | 一意一致した顧客のみ。空欄は上書きしない |

- 名前一致は `お名前` と CDP の `姓+名` を正規化して比較し、複数候補にぶつかる場合は採用しない
- `CR別データ` の `電話番号` は現状ほぼ空のため、実運用では `メールアドレス > 名前` が中心になる
- `CR別データ` は `sync-cr-route` コマンドと通常の `sync` の両方から継続反映される
- `データソース管理` の監視行は末尾に追記しない。既存の `CR名` 行の直後に `CR名` / `CRリンク` / `遷移先リンク` をまとめて置き、同じ項目を近くに保つ
- `CR名` は既存ソースが優先度 `1` を持つ場合、`CR別データ` 側を自動で優先度 `2` にする。`CRリンク` / `遷移先リンク` は単独ソースのため優先度 `1`

### メールなしヒアリングシート同期

`日程調整後_面談前ヒアリングシート` のようにメールアドレス・電話番号を持たないソースは、通常のメール同期ではなく `cdp_sync.py` の `LINE名 -> 姓+名` 一意一致ロジックで既存顧客へだけ反映する。

- 対象カラム例: `LINE名` / `姓` / `名` / `職業` / `理想の未来`
- 新規顧客は追加しない。既存の `CDP` 行に一意一致した場合だけ更新する
- `LINE名` と `職業` のような優先度 `2` 以降のカラムは、既に値が入っている場合は上書きせず空欄だけ補完する
- `姓` / `名` / `理想の未来` のような優先度 `1` のカラムは、既存顧客へ通常更新する

### CS管理シートの返金額同期

`返金額` は通常のメール1キー同期ではなく、CS管理シート専用ロジックで同期する。`データソース管理` には監視用に3行を持ち、実際の取り込みは `cdp_sync.py` が以下の条件で処理する。

| 優先度 | タブ | 返金額参照列 | 突合キー | 入力条件 |
|------|------|-------------|---------|---------|
| 1 | `管理用_クーオフ・中途解約以外` | P列 `返金額` | I列アドレス → H列LINE名 → G列本名 | F列=`完了` |
| 2 | `管理用_2025.1.25-クーオフ` | T列 `返金額` | K列アドレス → J列LINE名 → I列本名 | X列 `返金予定日` が入っている行のみ |
| 3 | `管理用_20250125-中途解約` | X列 `返金額/請求額` の**赤文字のみ** | K列アドレス → J列LINE名 → I列本名 | AB列 `返金日` が入っている行のみ |

- 突合優先順位は `メールアドレス > LINE名 > 本名`
- `LINE名` fallback は短すぎるキー（2文字以下、数字のみ）を除外して誤突合を防ぐ
- `中途解約` タブは Google Sheets API で書式を読み、赤文字セルのみ返金額として採用する
- `返金額` 更新時は `LTV = 着金売上 - 返金額` も同時に再計算する

**同期対象外のカラム（Python自動計算 or システム管理のため、データソース管理に含まない）:**
顧客ID / 作成日 / 最終更新日 / 総流入経路数 / セミナー予約合計 / 個別予約合計 / 累計購入回数 / 購入商品 / LTV

**集計カラムの自動計算（ARRAYFORMULAからPython計算に移行済み）:**
Google SheetsのARRAYFORMULA計算限界（約53,000行）を超えたため、cdp_sync.pyが直接値を書き込む方式に変更。
- セミナー予約合計 = セミナー予約日のカンマ区切り数
- 個別予約合計 = 個別予約日のカンマ区切り数
- 累計購入回数 = 購入商品のカンマ区切り数
- LTV = 着金売上 - 返金額
- 同期時にソースカラム更新と同時に自動計算。一括再計算は `backfill-computed` コマンド

**ステータス判定ルール（自動更新）:**
- 正常: 直近の同期が成功した（同期成功時に自動セット）
- 未同期: まだ一度も同期されていない（新規登録時にセット）
- 停止: 直近の同期でエラーが発生した（同期エラー時に自動セット）
- `最終同期日` には状態更新時刻を `YYYY/MM/DD HH:MM` で入れる
- 状態が変わったときだけ、`System/data/cdp_source_status_history.json` に `source_url / tab / 変更時刻 / 変更前後ステータス / error_count / detail` を追記する

**鮮度チェック（ステータスとは別の監視レイヤー）:**
- 7日以上未同期のソースを `check_source_staleness()` で検出
- Orchestrator経由でLINE通知（ステータスは変更しない）
- 確認が必要な通知では、`CDP / データソース管理` のリンクと、該当ソースのリンクを付ける
- 正常終了時の通知は更新件数だけを短く出し、ユーザーが介入できない内部処理の成功は通知しない

**ソース元（プルダウン選択肢）:**
Lステップ / UTAGE / Mailchimp / Univapay / MOSH / Invoy / 決済システム / 銀行振込 / 講義保管庫 / スプレッドシート

## 登録・名寄せルール

- **登録条件**: メールアドレスまたは電話番号が必須（LINE名のみでは登録しない）。さらにイベント発生（アンケート回答・セミナー予約・購入等）が必要。LINE登録のみ・メール登録のみの人は集客データシートに格納
- **名寄せ（自動統合）**: メールアドレス一致（メールアドレス2も含む） or 電話番号一致 → 統合
- **LINE名のみ**: 自動統合しない（重複リスクが高い）。後からメール/電話番号が判明したら統合
- **重複排除**: 複数ソースで同じメールアドレスがあれば既存レコードに統合（新規追加しない）
- **除外リスト**: CDPマスタの除外リストタブが Single Source of Truth。メール・電話番号のいずれかに一致したら除外。集客データシートの同期でも同じ除外リストを参照。LINE名は重複が多いため除外キーに使わない
- **スパム自動検知**: 除外リストに加え、同期時にパターンマッチでテスト/いたずらメールを自動除外（`is_spam_email()`）
  - test系: `test@`, `test123@` 等
  - 同一文字繰り返し: `aaaa@gmail.com` 等（フリーメールのみ）
  - テストドメイン: `example.com`, `test.com` 等
- **更新方式**: 名寄せで既存行を更新。新規顧客のみ最下部に追加
- **複数ソース**: 同一カラムに複数ソースがある場合、優先度順に値を探し最初のヒットを採用

### 同期の競合ルール

| 状況 | ルール |
|------|--------|
| 新規顧客 | 優先度1のソースから値を取得。なければ優先度2を確認 |
| 既存顧客（値が空） | 優先度順にソースを確認し、最初に見つかった値を記録 |
| 既存顧客（値が入っている・優先度1） | **優先度1ソースのみ** ソース側の値で上書きする |
| 既存顧客（値が入っている・優先度2以降） | 上書きせず空欄補完だけ行う |
| カンマ区切り追記カラム | セミナー予約日・個別予約日・個別着座日・購入商品は上書きではなく追記（重複スキップ） |

## データ保護

### 電話番号の保護
- 顧客マスタの電話番号列は **Plain Text形式** に設定済み
- Googleスプレッドシートが数値に自動変換して桁が壊れるのを防止
- 同期時にバリデーション実行（科学的記数法検出・桁数チェック）

### メールアドレスの重複検知
- 新規メール登録時にレーベンシュタイン距離で類似メールを検出（閾値: 距離2以下）
- 同一ドメインのメールのみ比較（パフォーマンス最適化）
- 警告ログを `System/data/cdp_email_warnings.json` に記録

### 入力バリデーション（同期・インポート時に自動適用）
- **メールアドレス**: RFC準拠の形式チェック（`is_valid_email()`）
- **電話番号**: 10〜11桁チェック・科学的記数法検出（`validate_phone()`）
- **年齢**: 数字1〜3桁 + optional 代/歳 のみ許可（`is_valid_age()`）
- **URL系（収録URL / サポートURL / CRリンク / 遷移先リンク）**: http/https で始まる形式のみ許可（`is_valid_url()`）

### 変更ログ
- 同期時に全変更の before/after を `System/data/cdp_change_log.json` に記録
- タイムスタンプ・行キー・カラム・旧値・新値・ソースを保持
- 最大10,000件を保持（古い方から自動削除）

### シート保護
- `顧客マスタ` と `データソース管理` はシート全体を保護し、通常の共同編集者は編集できない
- 編集を許可するのは AI / 同期で使う `kohara.kaito@team.addness.co.jp` と `gwsadmin@team.addness.co.jp` のみ
- 保護を変えるときは、先に同期アカウントを保護対象の編集者に残せることを確認する。AIが書けなくなる可能性がある設定は入れない

## 同期の仕組み

### 自動同期フロー

``` 
データソース管理タブ（マッピング）
  → cdp_sync.py がマッピングを読み取り
    → ソースごとにシートを読み込み
      → 除外チェック → 名寄せ → バリデーション → 書き込み
        → ステータス・同期日・更新数をデータソース管理に記録

CR別データ
  → cdp_sync.py の専用同期がメールアドレス → 電話番号 → 名前で CDP を一意突合
    → `CR名` / `CRリンク` / `遷移先リンク` の最新値だけを書き戻す
```

- `顧客マスタ` / `データソース管理` / `除外リスト` は同期後に罫線を最終行まで自動補正する
- `顧客マスタ` は `ensure-schema` 実行時に `CRリンク` の右へ `遷移先リンク` を挿入し、W列右 / AE列右の罫線と行1グループ見出しを再調整する

### 排他制御

- 同期開始時にロックファイル（`System/data/cdp_sync.lock`）を取得
- ロック中に別の同期が実行されるとブロック（「同期がロック中です」エラー）
- 10分経過でロック自動解除（デッドロック防止）
- 同期完了 or エラー時にロック自動解除

### 増分同期

- ソースごとの最終同期状態を `System/data/cdp_sync_state.json` で追跡
- 前回同期時の行数を記録し、差分（新しい行）のみ処理
- `--full` オプションで全件同期も可能

### 面談定性の自動補完

`収録URL` が入っている顧客について、AILEAD の面談収録から定性情報を抽出し、`現在の悩み` / `理想の未来` / `過去の解決策` を埋める。

```text
顧客マスタの収録URL
  → interview_insights_sync.py が AILEAD GraphQL から transcript を取得
    → LLM で顧客の悩み・理想の未来・過去の解決策を要約
      → 空欄のときだけ CDP の既存列へ書き戻し
        → 詳細はローカルキャッシュに保存
```

- 既存列に値がある場合は上書きしない（`--force` で上書き可能）
- まず Anthropic API を試し、クレジット不足やAPI失敗時は Claude CLI に自動フォールバックする
- AILEAD GraphQL は persisted query 前提で叩く。ローカルに持つ query 本文は送らず、`operationHash` と `buildId` を正本として取得する
- `confidence=low` の結果はシートへ書き込まず、キャッシュだけ残す
- 定性列は増やしすぎず、顧客理解と広告/LP/営業分析に直結する3項目だけに絞る
- キャッシュ: `System/data/interview_insights_cache/<callResultId>.json`
- 状態管理: `System/data/interview_insights_state.json`
- 実行ロック: `System/data/interview_insights_sync.lock`。`sync` / `backfill` / `analyze` / `ensure-schema` の多重起動を防ぎ、6時間を超えた stale lock は自動で張り替える
- Orchestrator: `interview_insights_sync` が **毎時50分** に最大20件を実行
- backlog 解消: `interview_insights_backfill` が **毎日01:20** に最大240件をまとめて処理
- 比較分析: `interview_insights_analysis` が **毎日05:20** に `System/data/interview_insights_analysis.json` を再生成
- 成功通知は送らず、止まったときだけ `CDP / 顧客マスタ` のリンク付きで通知する

### ソース変更検知

同期前にソースの構造変化を自動検出し、知らない間に違うデータを取り込むことを防ぐ。

| 検知パターン | 動作 |
|-------------|------|
| マッピング対象の列名が消失 | まず列名揺れを自動修復（記号差分・表記ゆれを吸収）。直せない場合のみ **同期を中断** |
| 列の追加・削除（マッピング外） | アラート表示して続行 |
| 行数が50%以上急減（100行以上のソース） | アラート表示して続行 |

- ヘッダーのスナップショットを `cdp_sync_state.json` に保存し、次回同期時に比較
- 初回同期時はスナップショットがないため検知なし（2回目以降から有効）
- 自動修復に成功した列名は `データソース管理` に書き戻し、次回以降も新しい列名を正として使う

### バッチ書き込み

- 既存行の更新: `batch_update()` で500件ずつチャンク分割
- 新規行の追加: `update()` で500行ずつ範囲指定書き込み（`append_rows()`はグリッド膨張リスクあり不使用）
- グリッドが足りない場合は自動拡張
- カンマ区切りカラム（セミナー予約日、個別予約日、個別着座日）は上書きではなく追記（重複チェック付き）

### CLIコマンド

```bash

python3 cdp_sync.py sync-leads         # 経路別タブ→集客データ同期のみ
```
