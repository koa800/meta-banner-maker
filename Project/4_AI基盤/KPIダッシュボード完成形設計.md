# KPIダッシュボード完成形設計

最終更新: 2026-03-17

## 目的

`【アドネス株式会社】KPIダッシュボード` を、経営会議のための見た目シートではなく、事業全体の判断に使う表示層として完成させる。

この設計で達成したい状態は次の3つ。

1. 集客、個別予約、着金売上、広告費、会員の数値を、同じ基準で毎日更新できる
2. 数値の元データ、加工データ、表示データの役割が混ざらない
3. 人手の入力や貼り替えを最小化し、異常時は更新を止めて検知できる

---

## このダッシュボードが答えるべき問い

ダッシュボードは、数値を並べる場所ではなく、次の判断に答える場所として設計する。

### 経営が毎日見たい問い

1. 今日は集客が伸びたのか、落ちたのか
2. 集客で取れた人は、個別予約までつながっているのか
3. 個別予約は、実施までつながっているのか
4. 実施から売上まで、どこで落ちているのか
5. 広告費に対して、売上は見合っているのか
6. 返金やクーリングオフを含めても、事業は伸びているのか
7. 会員純増は維持できているのか

### 現場が毎日見たい問い

1. どのソース、媒体、ファネルが集客を作っているのか
2. どのソースからの予約率が高いのか
3. どの商品の返金や解約が多いのか
4. どのデータ入口が止まっているのか
5. どのシートが未同期なのか

この問いに答えられない数値は、ダッシュボードには載せない。

---

## 完成形の一言定義

完成形は、`収集データ -> 統合データ -> 加工データ -> ダッシュボード` の4層で成り立つ。

- 収集データ
  外部システムや入力口から入る事実
- 統合データ
  顧客単位やソース単位で、他領域とつなげるための中核データ
- 加工データ
  経営判断や運用で使える粒度に整えた集計データ
- ダッシュボード
  人が見る場所。ここでは計算や名寄せの責務を持たない

`【アドネス株式会社】KPIダッシュボード` は最後の表示層だけを担当する。

---

## 設計原則

1. 正本は1つにする
2. ダッシュボードは見るだけにする
3. 最小粒度は `媒体×ファネル` ではなく `ソース` にする
4. 集計値より先に、元データの入口を固定する
5. 人が触るタブと自動生成タブを分ける
6. 異常時は更新しない
7. 過去の補完ルールと、今後の日常運用ルールを分ける

## 完成形の全体構造

### 1. 収集データ

ここは「事実を欠損なく受け取る場所」。

#### 集客

- `【アドネス株式会社】集客データ（メールアドレス）`
  - 役割: メール登録の入口
  - 持つもの: 登録日、メールアドレス、タブ名起点の流入経路

#### 個別予約

- `【アドネス】顧客管理シート / 個別予約集計botログ`
  - 役割: 個別予約イベントの入口
  - 現状の取得元: botログ
  - 持つべきもの: 日付、LINE名、流入経路、キャンセル、顧客名、タグ名、メールアドレス、CR名、予約アカウント名、電話番号

- `Lステップの ★【個別予約完了】★`
  - 役割: 予約完了導線が正しく発火しているかの検証と、将来の個別予約イベントログの起点
  - 現状の取得元: Lステップ
  - 確定していること:
    - ユーザーがカレンダー予約で指定コースを予約した時に付く
    - ユーザーがイベント予約で予約した時にも付く
    - 一度付いたら外さない
    - 再予約しても増えない
  - 持つべきもの:
    - 予約イベント日時
    - LINE名
    - `LSTEPメンバーID`
    - Lステップアカウント名
    - 通知リンク

#### 個別予約の正本戦略

個別予約は、現時点で `正確さ` と `運用コスト` の両立がいちばん難しい領域。

- Googleカレンダー
  - 予約イベントの事実に近い
  - ただしカレンダーが複数あり、日程変更で `キャンセル + 新規予定` になるため、全体の正本にするコストが高い
- `★【個別予約完了】★` タグ
  - 予約完了の成立条件そのものには近い
  - ただしタグ単体は累積状態で、日別イベント数を取れない
- `salon_reservation_reserved / reservation.reserved` の予約イベント通知
  - 1予約イベントに近い
  - 予約イベント数の本命候補になりやすい
- `個別予約集計botログ`
  - すでに日付つきのイベント一覧になっていて、今すぐ集計に使える
  - ただし生成過程が少しあやふやで、長期的な最終正本にはしにくい

このため、完成形は次の2段階で作る。

1. 第1版
   - `【アドネス】顧客管理シート / 個別予約集計botログ` を使って `個別予約数` を持つ
2. 第2版
   - Lステップの通知機能で `salon_reservation_reserved / reservation.reserved` の予約イベント通知を専用の通知先へ送り、1イベント1行の `個別予約通知ログ` を作る
   - `★【個別予約完了】★` のタグ通知は、正本ではなく検証用と冗長系として残す
   - この通知ログを、個別予約の高精度な正本候補にする

つまり、タグは `今は検証用`、将来も `冗長系の確認軸` として使う。通知ログの本命は予約イベント通知。

#### 継続集計体制で変えること / 変えないこと / 追加すること

この領域は、`予約イベントをどう正確に残すか` が本質。継続体制では次の3つを分けて運用する。

##### 変えないこと

- `個別予約数` は `予約イベント数` を意味する
- 同じ人が別日に再予約したら、別件数で数える
- `★【個別予約完了】★` は `予約完了の成立条件` として扱う
- `【アドネス株式会社】KPIダッシュボード` は表示層に徹し、正本を直接持たない

##### 変えること

- `個別予約数` の本命の計上元を、`個別予約集計botログ` 固定から `個別予約通知ログ` へ段階移行する
- `Googleカレンダー` は主計上元ではなく、必要時の検証手段へ下げる
- `LSTEPメンバーID` は `同一人物ID` ではなく、`予約イベントを正確に残すためのイベントキー` として使う

##### 追加すること

- `Slack #個別予約通知` を通知の受け皿にする
- `【アドネス株式会社】個別面談データ / 個別予約通知ログ` を新設し、過去の個別予約データと現在の通知を `1イベント = 1行` で保存する
- 将来の照合に備えて、通知ログへ次を残す
  - 予約イベント日時
  - LINE名
  - `LSTEPメンバーID`
  - Lステップアカウント名
  - 通知リンク
  - メールアドレス
  - 電話番号
- メールアドレスや顧客IDの照合は、通知ログのあとに別段階で行う

#### 個別予約通知ログの設計

通知ログは、`Lステップでタグが付いた瞬間` を `あとで日別集計できる形` に変えるための土台。

- まず使う通知先
  - 第1候補は `Slack の専用チャンネル`
  - 理由:
    - 既存で Slack 読み取りの仕組みがある
    - チャンネル単位でログを取りやすい
    - Webhook の bot 投稿は読み取り側で除外しやすい
- 第2候補
  - `Chatwork`
  - ただし現状の callback は `mention_to_me` 前提なので、そのままでは予約通知ログには向かない

- 通知ログで最低限持つ列
  - 予約イベント日時
  - LINE名
  - LSTEPメンバーID
    - Slack 通知内の `notification/view?type=member&id=...` から取る
  - Lステップアカウント名
  - 通知リンク
  - メールアドレス
  - 電話番号
- シート外で持つ内部状態
  - `Slack ts`
  - `最終取込日時`
  - 重複判定キー
- `データソース管理` では、`個別予約通知ログ` の最終取込日時と保存件数、ステータスを自動で反映する
- 過去分は `個別予約データ / シート1` を毎回読み直して統合する
- `LSTEPメンバーID + Lステップアカウント名` が local の友だち一覧CSVに一致した時は、`メールアドレス / 電話番号` を自動補完する

- 重複判定の考え方
  - 同一人物の `ユニーク判定` ではなく、まずは `1タグ付与 = 1予約イベント候補` として持つ
  - まず `予約イベント日時 + LSTEPメンバーID + Lステップアカウント名 + LINE名 + メールアドレス + 電話番号` を重複判定キーにする
  - `Slack ts` はシートには出さず、内部状態の継続取得だけに使う

- メールアドレス取得の考え方
  - 通知だけではメールアドレスは取れない前提で設計する
  - まずは `LSTEPメンバーID` を確実に残し、あとから `友だち情報管理` や予約関連CSVの `メールアドレス / 電話番号` と突合する
  - したがって、通知ログの目的は `まず予約イベントを正確に残すこと` であり、`その場でメールアドレスまで確定すること` ではない

- 自動化の流れ
  1. Lステップで予約イベント通知 (`salon_reservation_reserved / reservation.reserved`) を発火させる
  2. 同時に `★【個別予約完了】★` のタグ通知も検証用として残す
  3. 通知機能で Slack の専用チャンネルへ送る
  4. 定期実行で Slack チャンネルを読み取る
  5. `個別予約通知ログ` に 1イベント1行で蓄積する
  6. `日別個別予約数` と将来の `日別個別予約数（UU）` の正本候補として使う

#### Lステップ監査

- `python3 System/scripts/lstep_booking_audit.py`
  - 役割: 個別予約に関係する通知設定、タグ、メンバークエリ、カレンダーの live 状態を一括確認する
  - 目的: `★【個別予約完了】★` のタグ通知と、`salon_reservation_reserved / reservation.reserved` の予約イベント通知のどちらを本命にするかを、live 設定から判断できる状態にする
  - 出力: `System/data/lstep_booking_audit_latest.json`
- live 監査で既に見えている通知の例
  - `個別予約完了`
    - `tag_attached`
    - `★【個別予約完了】★` のタグ通知
  - `【予約数】みかみ@個別_3/16~`
    - `salon_reservation_reserved`
    - `友だち情報 [予約回数] <= 1`
  - `SNS個別予約数`
    - `reservation.reserved / salon_reservation_reserved`
    - `友だち情報 [予約回数] = 1`
  - `【共通】【予約通知】個別専用`
    - `reservation.reserved / salon_reservation_reserved`
    - 広めの予約イベント通知候補

#### 売上

- `決済履歴シート`
  - 役割: 着金売上の入口
  - 現状の取得元: 決済履歴CSV
  - 持つべきもの: 決済日、商品、メール、氏名、LINE名、着金売上

- `CS管理シート`
  - 役割: 返金、クーリングオフ、中途解約の入口
  - 持つべきもの: 対応完了日、返金額、クーリングオフ日、中途解約日

#### 広告費

- Meta / Google / TikTok / X の媒体原本
  - 役割: 広告費と配信実績の入口
  - 現状: 新しい正本は未構築
  - 完成形ではここを直接読みに行く

#### 会員

- 会員管理元データ
- `CS管理シート`
- `秘密の部屋（月額）`
- `秘密の部屋（年額）`

役割が混ざっているので、完成形では会員イベントとして切り直す。

## 何のデータを取るべきか

### 集客

#### 必須指標

- メール登録件数
- メール登録件数（UU）
- メール登録ユーザー数
- LINE登録件数
- LINE登録件数（UU）
- 集客数
- 集客数（UU）

#### 補助指標

- ソース別件数
- 媒体別件数
- ファネル別件数
- 媒体×ファネル別件数
- LP別件数
- CR別件数

#### 正本の考え方

- 重複あり件数
  - 登録という事実
- UU
  - 最初に確認された日ベースのユニーク
- ユーザー数
  - 複数アドレスユーザーを調整した人数寄りの数

---

### 個別予約

#### 必須指標

- 個別予約数
- 個別予約数（UU）
- 個別予約率
- 個別実施数
- 個別実施数（UU）
- 個別実施率

#### 補助指標

- ソース別予約数
- 媒体別予約数
- 予約から実施への移行率
- 予約日と実施日の差分

#### 正本の考え方

- 予約
  - `【アドネス】顧客管理シート / 個別予約集計botログ` のうち `日付あり and キャンセル=FALSE`
- 実施
  - 第1版ではまだ正本未接続

#### いまの課題

- 個別予約数（UU）の同一人物判定が未確定
- 個別実施数の正本がまだ未確定
- `★【個別予約完了】★` をどこまで検証軸として使うか、Lステップ実確認が残っている

#### Lステップで後から確認すること

## `【アドネス株式会社】KPIダッシュボード` の画面設計

ここでは「どんなタブを持つか」ではなく、「最初に何が見えれば判断できるか」を先に固定する。

### 画面設計の原則

1. 1画面目で経営判断に必要な情報が終わる
2. 上から下へ、`集客 -> 予約 -> 実施 -> 売上 -> 会員` の順で流れる
3. 最初の画面では、表よりカードと推移を優先する
4. 詳細内訳は別タブに逃がす
5. 色は「意味があるもの」だけに使う
6. 1つの要素に情報を詰め込みすぎない

### 1画面目に必要な要素

#### 1. ヘッダー

画面上部に必ず置く。

- ダッシュボード名
- 集計期間
- 会議基準日または最新集計日
- 最終更新日時
- 更新状態
  - 正常
  - 未同期
  - 停止

ここで分かるべきことは、`いつの数字を見ているか` と `その数字を信じてよいか` の2点だけ。

#### 2. 主要KPIカード

1行目に置く。6枚まで。

推奨順:

1. 集客数
2. 集客数（UU）
3. 個別予約数
4. 個別実施数
5. 着金売上
6. 会員純増

各カードで持つ情報は3つまでに絞る。

- 現在値
- 前週比または前月比
- 目標進捗

カードの中に率や補助指標を何個も入れない。

#### 3. 事業ファネル

主要KPIカードの下に、`集客 -> 予約 -> 実施 -> 成約 -> 着金売上` の流れを置く。

ここで見たいのは次。

- どこで落ちているか
- 前週比でどこが改善したか
- 率が悪い場所はどこか

ファネルは `件数` と `率` の両方を持つが、表示はシンプルにする。

- 上段: 件数
- 下段: 前段に対する率

#### 4. 日次推移

ファネルの下に、時系列で変化を見るグラフを置く。

ここは2つに分ける。

- 集客・予約・実施の推移
- 着金売上・広告費・粗利の推移

1つのグラフに全部載せない。

#### 5. 内訳ブロック

1画面目の下部に、`何が数字を作っているか` を見るブロックを置く。

左:

- 集客内訳 Top10
  - ソース
  - 媒体
  - ファネル
  - 集客数
  - 集客数（UU）

右:

- 売上・広告費内訳 Top10
  - ソース
  - 着金売上
  - 広告費
  - ROAS

ここは `全部を見る場所` ではなく、`今どこが効いているか` を掴む場所。

#### 6. 更新状況

最下部に、`見る価値があるか` を判定するための状態欄を置く。

- 今日止まっているシート
- 未同期のシート
- 直近の異常検知

ここは運用チーム向けだが、経営画面にも必須。

## 自動更新の完成形

### 原則

- 収集データを取り込む
- 統合データを更新する
- 加工データを再生成する
- ダッシュボードを更新する
- 異常があれば止める

### 目標の実行順

1. `【アドネス株式会社】集客データ（メールアドレス）` を取得
2. `【アドネス株式会社】顧客データ（メールアドレスのみ）` と `【アドネス株式会社】顧客データ（複数イベント）` を更新
3. `【アドネス株式会社】集客データ（メール集計）` を更新
4. `【アドネス株式会社】個別面談データ` を更新
5. `【アドネス株式会社】着金売上データ（集計）` を更新
6. `【アドネス株式会社】広告費データ（集計）` を更新
7. `【アドネス株式会社】会員データ（集計）` を更新
8. `【アドネス株式会社】KPIダッシュボード` を更新

### ガードレール

- 更新元が0件なら止める
- 前回比で急減したら止める
- 生成タブは保護する
- `データソース管理` に `正常 / 未同期 / 停止` を持つ
- `更新状況` タブで最終更新日時を見える化する

### 人的リソースを減らすための原則

1. 元データの貼り付けを日次運用にしない
2. 収集データはできるだけ API か既存連携で受ける
3. どうしても CSV が必要なものは、入口を1つに固定する
4. 加工データはすべて再生成型にする
5. 手修正で合わせる運用を禁止する
6. `停止` になった時だけ人が見る
7. ダッシュボードではなく `データソース管理` を監視対象にする

## 現時点の重要な論点

### 1. `媒体×ファネル` で持つか `ソース` で持つか

結論は `ソース`。

理由:

- 後から `媒体×CR×LP×ファネル` に拡張できる
- `媒体×ファネル` だけだと戻れない
- ダッシュボードでは `媒体×ファネル` で見ればよい

### 2. 個別予約の正本は何か

結論は、今のままだと1つではない。

- 予約: `【アドネス】顧客管理シート / 個別予約集計botログ`

## 現在の第1版状態

2026-03-17 時点で、実際に形になっているのは次の通り。

### `【アドネス株式会社】KPIダッシュボード`

- 表に出しているタブは `定義 / スキルプラス事業サマリー / 日別数値 / データソース管理`
- `日別数値` は次の14列を受け口として固定
  - `日付`
  - `集客数`
  - `集客数（UU）`
  - `個別予約数`
  - `個別予約数（UU）`
  - `個別実施数`
  - `着金売上`
  - `広告費`
  - `CPA`
  - `個別予約CPO`
  - `ROAS`
  - `会員数`
  - `中途解約数`
  - `クーリングオフ数`
- 現在接続済みなのは `集客数 / 集客数（UU） / 個別予約数`
- `スキルプラス事業サマリー` は `日別数値` を参照して期間集計する構造で、未接続項目は空欄のままにする

### `【アドネス株式会社】個別面談データ`

- 第1版の `日別個別予約数` は、`2026/03/16` 以前を `【アドネス】顧客管理シート / 個別予約集計botログ`、`2026/03/17` 以降を `個別予約通知ログ` で持つ
- 今つないでいるのは `個別予約数` だけ
- `個別予約数（UU）` は将来用のタブだけ先に持ち、未接続のままにする
- `個別実施数` は今は扱わない
- `★【個別予約完了】★` は正本ではなく、Lステップ側の整合確認用に使う

### ここまでで確定していること

- `個別予約数` は `2026/03/16` 以前を `【アドネス】顧客管理シート / 個別予約集計botログ` の `日付あり and キャンセル=FALSE`、`2026/03/17` 以降を `Slack #個別予約通知` から作る `個別予約通知ログ` の日別件数で持つ
- `★【個別予約完了】★` は一度付いたら外れず、再予約しても増えない
- したがって `★【個別予約完了】★` は日別イベント数の正本には向かない
- `Googleカレンダー` は予約事実の確認には有効だが、複数アカウント運用のため第1版の正本には向かない
