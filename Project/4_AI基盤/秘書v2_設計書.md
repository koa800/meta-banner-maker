# 秘書v2 設計書 — 「甲原さんを作る」

> キーワードマッチの命令実行型から、文脈を理解して自律的に動く会話型へ。
> 秘書の全機能を最高モデルで動かし、甲原さんの分身として機能させる。

最終更新: 2026-03-07

---

## ゴール

**「弊社を最強にする」ための秘書。** 甲原さんの思考・判断・口調を再現し、すべての対話チャネルで甲原さんの分身として動く。

## アーキテクチャ

### 全体フロー

```
甲原さん / 社内メンバー / 受講生（LINE）
  ↓ メッセージ
app.py（Render）
  ↓ メッセージ中継（現状の委譲構造を維持）
local_agent.py（Mac Mini）
  ↓
┌─────────────────────────────────────┐
│  秘書v2 コア                          │
│                                       │
│  ① 記憶ロード（長期 + 短期）           │
│  ② 送信元の判定（秘書G / 他G / Q&A）   │
│  ③ LLM に全部渡す                     │
│     - システムプロンプト（人格・OS）     │
│     - 長期記憶                         │
│     - 直近の会話履歴                   │
│     - 送信者プロファイル               │
│     - 利用可能なツール一覧             │
│     - 今回のメッセージ                 │
│  ④ LLM が自律的に判断                 │
│     - そのまま返答する                 │
│     - ツールを使ってから返答する        │
│     - 複数ツールを組み合わせる          │
│  ⑤ 返答をLINEに送信                   │
│  ⑥ 記憶の更新（覚える/忘れる）         │
└─────────────────────────────────────┘
```

### Render（app.py）の役割

Renderは中継に徹する。判断はすべてMac Mini側のv2エンジンが行う。

```
Renderの責務:
  ✅ LINEからのWebhook受信
  ✅ メッセージをMac Miniのタスクキューに追加
  ✅ Mac Miniからの応答をLINEに送信
  ❌ ツール選択（v2ではやらない）
  ❌ キーワード判定（v2ではやらない）
```

Renderのコード変更は最小限。既存の委譲構造を維持しつつ、v1のキーワード判定ロジックを外すだけ。

## 1. モデルルーター

### 設計思想

- モデルは設定ファイル1つで切り替え可能。コード変更不要
- デフォルトは最高モデル（GPT-5.4）。賢くないモデルは使わない
- Anthropic / OpenAI のAPI差分はルーターが吸収する

### 設定ファイル: `System/config/llm_router.json`

```json
{
  "default_model": {
    "provider": "openai",
    "model": "gpt-5.4",
    "max_tokens": 1024
  },
  "fallback_order": [
    {"provider": "openai", "model": "gpt-5.4"},
    {"provider": "anthropic", "model": "claude-opus-4-6"},
    {"provider": "anthropic", "model": "claude-sonnet-4-6"}
  ],
  "providers": {
    "anthropic": {
      "api_key_env": "ANTHROPIC_API_KEY",
      "models": ["claude-opus-4-6", "claude-sonnet-4-6"]
    },
    "openai": {
      "api_key_env": "OPENAI_API_KEY",
      "models": ["gpt-5.4", "gpt-4.1", "o3"]
    }
  }
}
```

### 切替方法

```
甲原さん「モデルをOpusにして」
  → 秘書: default_model を更新 → 「Claude Opus 4.6に切り替えました」

甲原さん「GPT-5.4に戻して」
  → 秘書: default_model を更新 → 「GPT-5.4に戻しました」
```

### フォールバック

API障害時は `fallback_order` に従って自動切替。GPT-5.4が落ちたら → Opus → Sonnet の順。

## 2. 会話エンジン

### v1 → v2 の変更

**v1（キーワードで分岐）:**
```python
if any(kw in text for kw in ["ROAS", "CPA", "売上"]):
    return kpi_query(text)
elif text.endswith("？"):
    return context_query(text)
else:
    return simple_reply(text)
```

**v2（LLMに全部任せる）:**
```python
response = llm.chat(
    system=system_prompt,
    messages=conversation_history,
    memory=long_term_memory,
    tools=available_tools,
    user_message=text
)
```

### ツール定義（v2）

既存の機能をtool_use形式でLLMに渡す。ツールの中身（実行コード）は既存を流用。

```python
tools = [
    # --- データ取得 ---
    {
        "name": "get_kpi_data",
        "description": "アドネスの事業KPI（集客数・CPA・ROAS・売上・広告費等）を取得する",
        "input_schema": {
            "properties": {
                "period": {"type": "string", "description": "取得期間（例: '今月', '直近3ヶ月'）"},
                "metrics": {"type": "array", "description": "取得する指標（省略時は全指標）"}
            }
        }
    },
    {
        "name": "get_calendar",
        "description": "Googleカレンダーの予定を取得・作成する"
    },
    {
        "name": "check_email",
        "description": "未読・返信待ちメールの状況を確認する"
    },
    {
        "name": "read_spreadsheet",
        "description": "Google スプレッドシートのデータを読み取る"
    },

    # --- ナレッジ ---
    {
        "name": "search_knowledge",
        "description": "Skills/やMaster/のナレッジベースから情報を検索する"
    },
    {
        "name": "search_qa",
        "description": "過去のQ&A回答データベース（Pinecone）から類似質問を検索する"
    },

    # --- 記憶 ---
    {
        "name": "update_memory",
        "description": "長期記憶を更新する（重要な情報を覚える or 不要な情報を忘れる）"
    },

    # --- 画像 ---
    {
        "name": "generate_image",
        "description": "テキストから画像を生成する（Gemini 2.5 Flash）"
    },

    # --- システム ---
    {
        "name": "switch_model",
        "description": "使用するLLMモデルを切り替える"
    }
]
```

### 送信元別の動作

v2エンジンは1つだが、送信元に応じてシステムプロンプトとツールセットを調整する。

| 送信元 | システムプロンプト | 使えるツール | 口調 |
|--------|-----------------|------------|------|
| 秘書グループ（甲原さん） | OS + IDENTITY + 全ツール | 全ツール | 秘書として（フランク） |
| 他グループ（メンション） | OS + IDENTITY + 送信者プロファイル | データ取得 + ナレッジ | 甲原さんの口調再現 |
| Q&A（受講生） | Q&A専用ルール + ナレッジ | search_qa + search_knowledge | 丁寧・分かりやすく |

## 3. 記憶システム

### 設計思想

人間と同じ。重要なことは覚え、不要なことは忘れる。
「覚えるため」でもあり「忘れるため」でもある。
LLM自身が「これは覚えるべきか」を判断する。賢いモデルだからこそ任せられる。

### 3層構造

```
┌─────────────────────────────┐
│ 短期記憶（ワーキングメモリ）    │  直近の会話。セッション内で保持
│ 保持: 直近10-20往復           │  → messages 配列としてLLMに渡す
│ 消失: 6時間無操作でリセット    │
├─────────────────────────────┤
│ 長期記憶（エピソード記憶）      │  重要な出来事・決定・好み・文脈
│ 保持: LLMが重要と判断したもの  │  → secretary_memory.md に保存
│ 消失: LLMが不要と判断したら削除 │  → システムプロンプトに注入
├─────────────────────────────┤
│ 知識（意味記憶）               │  Skills/, Master/ のナレッジ
│ 保持: 永続（git管理）          │  → search_knowledge ツールで参照
│ 更新: 手動 or 秘書が提案       │
└─────────────────────────────┘
```

### 初期データ（既存資産から引き継ぎ）

ゼロスタートではなく、これまでの蓄積を引き継いでスタートする。

| ソース | 引き継ぐ内容 | 引き継ぎ方法 |
|--------|------------|------------|
| `MEMORY.md` | 行動ルール・ユーザー設定・好み | secretary_memory.md の初期データとして構造化 |
| `IDENTITY.md` | 甲原さんの口調・価値観・人格 | システムプロンプトに組み込み（現状と同じ） |
| `contact_state.json` | 各人との会話履歴・関係性 | 送信者プロファイルの初期データとして統合 |
| `execution_rules.json` | OS（行動ルール） | システムプロンプトに組み込み（現状と同じ） |
| `Master/learning/hinata_memory.md` | 日向の学習記録 | 参考情報として長期記憶に含める |

### 長期記憶の動作

**覚える（会話のたびにLLMが判断）:**
```
甲原さん「来週から広告費を月200万に増やす」
  → LLMの判断: これは重要な意思決定 → update_memory ツールを呼ぶ
  → secretary_memory.md に追記:
    「2026-03-07: 広告費を月200万に増額する方針決定」
```

**忘れる（定期的にLLMが整理）:**
```
記憶の棚卸し（1日1回）:
  → 「3週間前の『明日の会議資料送って』はもう不要」→ 削除
  → 「先月決めた広告戦略の方針はまだ有効」→ 残す
  → 「同じ内容が3つある」→ 1つに統合
```

### 記憶ファイルの保存先: `System/data/secretary_memory.md`

git管理外（`System/data/` は .gitignore 対象）に置く。理由:
- `git_pull_sync` の fetch/reset で上書きされない
- 秘書が自由に読み書きできる
- バックアップは Orchestrator が日次で `secretary_memory.md` を git commit & push する専用ジョブで担保

### 記憶ファイルの構造: `secretary_memory.md`

```markdown
# 秘書の記憶

## 甲原さんの方針・意思決定
- （MEMORY.mdから引き継ぎ + 会話から自動追加）

## 進行中の重要事項
- （会話から自動追加・完了したら削除）

## 甲原さんの好み・スタイル
- （MEMORY.mdから引き継ぎ + 会話から学習）

## 人間関係・送信者メモ
- （contact_stateから引き継ぎ + 会話から更新）

## 覚えておくべき文脈
- （LLMが会話から自動で追加・削除）
```

### 記憶の判断基準（システムプロンプトに組み込む）

```
【覚えるべきもの】
- 方針・意思決定（「〇〇にする」「〇〇をやめる」）
- 繰り返し聞かれること（2回以上同じ話題が出たら覚える）
- 明示的に「覚えておいて」と言われたこと
- 期限付きのタスクや約束
- 人に関する情報（誰がどういう役割か、関係性の変化）

【忘れてよいもの】
- 一度きりの事務的なやり取り（「今日の予定は？」→回答）
- 完了して1週間以上経った単発タスク
- 既にナレッジファイル（Skills/, Master/）に書かれている情報
- 時間が経って状況が変わった古い情報
```

## 4. メッセージキュー（順番処理）

### 設計思想

同時に複数メッセージが来ても、キューに入れて1つずつ順番に処理する。
シンプルで壊れにくい。並列処理の複雑さを持ち込まない。

### 処理フロー

```
秘書グループ「ROASどう？」─┐
                           ├→ キュー: [①秘書G, ②他G, ③Q&A]
他グループ「@秘書 確認して」─┤      ↓
                           │   ① 処理中...完了 → LINE送信
Q&A「質問です」────────────┘   ② 処理中...完了 → LINE送信
                               ③ 処理中...完了 → LINE送信
```

### 会話セッションの分離

キューは1本だが、会話履歴（短期記憶）はチャネルごとに分離する。

```
短期記憶:
  秘書グループ  → conversations/secretary_group.json（直近20往復）
  グループA     → conversations/group_A.json（直近10往復）
  グループB     → conversations/group_B.json（直近10往復）
  Q&Aチャネル  → conversations/qa.json（質問単位でリセット）

長期記憶:
  全チャネル共通 → secretary_memory.md（1ファイル）
  ※ 書き込みはキュー処理なので排他制御不要（常に1プロセスのみ書き込む）
```

## 5. 障害対策

### Mac Mini 障害時

Mac Miniは常時稼働。停電以外で停止することは基本的にない。
万が一の障害時は、甲原さんに通知して対処アクションを指示する。

```
Mac Mini 停止を検知（MacBookの外部監視 = monitoring/check_mac_mini.sh）
  ↓
甲原さんにLINE通知:
  「Mac Miniとの接続が切れました。
   ① Wi-Fi/電源を確認してください
   ② 復旧しない場合: [手順URL]
   秘書は一時的に応答できません。復旧次第、自動で再開します。」
  ↓
Mac Mini 復旧 → launchd が自動で秘書を再起動 → 通常運転に復帰
  ↓
甲原さんにLINE通知:
  「復旧しました。通常運転に戻ります。」
```

### Render側の最低限保険

Mac Miniが応答しない場合（タスクキューに入れて60秒応答なし）、Renderが最低限の応答を返す:

```
「すみません、今ちょっと処理が追いついていません。
 少し待ってからもう一度お願いできますか？」
```

これ以上の高度な応答はRender側では行わない（中継の役割を守る）。

### API障害時

`fallback_order` に従って自動切替。全プロバイダーが落ちたら甲原さんに通知。

```json
"fallback_order": [
  {"provider": "openai", "model": "gpt-5.4"},
  {"provider": "anthropic", "model": "claude-opus-4-6"},
  {"provider": "anthropic", "model": "claude-sonnet-4-6"}
]
```

```
GPT-5.4 → タイムアウト or エラー
  → Opus 4.6 にフォールバック（自動）
  → それも失敗 → Sonnet 4.6（最終手段）
  → 全滅 → 甲原さんにLINE通知:
    「APIサービスが全て障害中です。復旧を待っています。」
```

## 6. LINE応答タイムアウト対策

### 問題

LINEのreply APIはWebhook受信から一定時間内に返す必要がある。
v2ではツール実行（KPI取得→分析等）に数十秒かかるケースがある。

### 対策: 即時受付 + 結果はpush message

```
① Webhook受信 → Render が即座に 200 OK を返す（LINE側のタイムアウト回避）
② タスクをMac Miniのキューに追加
③ Mac Mini が処理完了 → push message API で結果を送信
```

これは現状のv1と同じ構造。v2でも維持する。

重いツール呼び出し（KPI取得 + シート読み込み + 分析）が連鎖する場合:
- 10秒以内に完了 → そのまま送信
- 10秒超え → 「確認しています...」を先に送信 → 完了後に本文を送信

## 7. コンテキストウィンドウ設計

### 予算とのバランス

コンテキストが大きいほど賢いが、コストも比例して上がる。上限を設けて制御する。

### トークン配分（1メッセージあたりの上限: 20,000トークン）

| 項目 | 上限 | 備考 |
|------|------|------|
| システムプロンプト（人格・OS・ルール） | 3,000 | 固定。IDENTITY.md + execution_rules |
| 長期記憶 | 3,000 | secretary_memory.md。上限100行で制御 |
| 短期記憶（会話履歴） | 8,000 | 直近の会話。古い順に切り捨て |
| ツール定義 | 2,000 | 固定。全ツール分 |
| ユーザーメッセージ | 2,000 | 入力テキスト |
| 出力（max_tokens） | 2,000 | 応答テキスト |
| **合計** | **20,000** | |

### コストへの影響

| モデル | 20,000トークン/メッセージ × 900通/月 |
|--------|--------------------------------------|
| GPT-5.4 | 入力$0.045 + 出力$0.02 = **~$58/月** |
| Opus 4.6 | 入力$0.27 + 出力$0.15 = **~$378/月** |

GPT-5.4なら月$58、Opusでも月$378で予算内。

### 記憶ファイルの上限管理

```
secretary_memory.md: 上限100行
  → 超えたら棚卸し（日次）でLLMが統合・削除
  → 棚卸しでも収まらない場合は古い順に強制削除

会話履歴: チャネルごとに上限
  → 秘書グループ: 直近20往復
  → 他グループ: 直近10往復
  → Q&A: 質問単位でリセット（履歴を持たない）
```

## 8. デプロイ戦略

### 原則

v1とv2を並行稼働させ、設定ファイルのフラグで切り替える。

### 手順

```
Step 1: v2コードをデプロイ（v1は動いたまま）
  → local_agent.py に v2 エンジンを追加
  → config で engine_version: "v1" のまま

Step 2: テスト（本番環境で）
  → 秘書グループでテストメッセージを送って動作確認
  → 甲原さんが「OK」と判断するまでv1で稼働

Step 3: 切り替え
  → config の engine_version を "v2" に変更
  → 即座にv2で稼働開始

Step 4: 問題があればロールバック
  → engine_version を "v1" に戻す → 即座にv1に復帰
```

### 切替設定

```json
// System/config/llm_router.json
{
  "engine_version": "v2",    // "v1" に戻せば即ロールバック
  "default_model": { ... }
}
```
