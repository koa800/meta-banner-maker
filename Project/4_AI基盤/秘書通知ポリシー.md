# 秘書通知ポリシー

最終更新: 2026-03-16

## 依頼理解

秘書の通知が多すぎて、重要な通知の価値まで薄れている。
通知を `何のために送るか` で再定義し、棚卸し結果を人が読める形と、コードが参照できる形の両方で正本化する。

## 結論

秘書が通知してよいのは、`甲原さんの次の行動が変わる情報` だけ。

通知は次の 3 区分で扱う。

- `即時通知`: 今見ないと遅延・損失・顧客影響が出る。判断・承認・認証・返信など、甲原さんの介入が必要
- `まとめ通知`: 今日見る価値はあるが、今すぐの介入は不要。朝夕や週次のまとまった確認で十分
- `非通知`: 内部成功ログ。秘書が自動で吸収すべきもので、甲原さんの行動を変えない

## 判定ルール

通知前に必ず次を判定する。

1. `owner_action_required`
   甲原さんの確認・判断・承認・返信・認証が必要か
2. `time_sensitive`
   今見ないと価値が大きく落ちるか
3. `external_impact`
   顧客・社外・対外送信・重要データ更新に影響するか
4. `manual_recovery_required`
   秘書が自動復旧できず、人の介入が必要か
5. `review_today_value`
   即時ではないが、今日の判断材料としては価値があるか

判定結果は次で切る。

- `1-4` のどれかが `true` なら `即時通知`
- `5` だけが `true` なら `まとめ通知`
- すべて `false` なら `非通知`

追加ルール:

- 復旧通知は、直前に甲原さんへ対応依頼を出していた案件だけ送る
- 内部で自己回復した成功通知は送らない
- 同じテーマの通知は cooldown を入れる
- 1通知につき 1 アクションに絞る

## 文面ルール

### 即時通知

必須要素:

- 結論
- 影響
- 次にしてほしいこと
- 開く場所（UI確認が必要なとき）

基本テンプレート:

```text
[結論]

影響:
[何が止まるか / 何が遅れるか]

次にしてほしいこと:
[1アクション]

見る場所:
[画面名 or URL]
```

### まとめ通知

必須要素:

- 今日見る価値があること
- 優先順位
- 必要なら次アクション

基本テンプレート:

```text
[タイトル]

今日見る価値があるもの:
- ...
- ...

必要なら次アクション:
- ...
```

### 非通知

- LINE や Slack には出さず、ログまたは state のみ更新する

## 実装ギャップ

現時点で、ルールと実装に次のズレがある。

- `内部処理の成功通知は送らない` という原則は既にある
- ただし実装には成功通知が複数残っている
- `AI秘書作成.md` には `daily_report` の 21:00 稼働報告を停止したとあるが、`scheduler.py::_run_daily_report` はまだ送信している

## 本質的な仕組み

通知の要否をジョブごとに個別 if 文で増やさない。
送信直前に共通判定を通す。

### 正本

- 人が読む正本: `Project/4_AI基盤/秘書通知ポリシー.md`
- コードが読む正本: `System/config/secretary_notification_policy.json`

### 今後の実装方針

1. `send_line_notify()` の直前で、通知イベントを `kind` 付きで渡す
2. `kind` ごとに policy JSON を参照して `immediate / digest / silent` を決める
3. `silent` は送らずにログだけ残す
4. `digest` は即送らず、日次/週次のバッファに貯める
5. `immediate` だけをその場で送る
6. 復旧通知は `prior_human_action_requested=true` のときだけ送る

### 実装の優先順

1. 既存のノイズ通知を `silent` に落とす
2. `digest` バッファを入れる
3. ジョブごとの文面を policy テンプレートへ寄せる
4. `send_line_notify` を直接呼ばず、`notify_event()` 経由へ寄せる

## 初期実装メモ

2026-03-16 の初回実装では、次まで入れた。

- `agent_orchestrator/notifier.py` に policy 読み込みを追加
- `notify_event()` で `immediate / digest / silent` を共通判定
- `flush_digest_events()` で digest queue を朝夕のまとめから出せる形にした
- `KPI更新完了` `管理シート同期完了` `Git同期復旧` `週次プロファイル学習完了` `21:00 稼働報告` を `silent` 化
- `mail_waiting_digest` を queue 化
- `line_bot/app.py` のデプロイ成功通知を停止
- `csv_sheet_sync.py` の成功通知を停止

まだ残っている課題:

- digest に寄せる対象は一部だけで、既存の定期サマリ系はまだ直送が残っている
- `send_line_notify` の直呼びが scheduler 全体に残っている
- digest の grouping と template は次段でさらに整理する
