# 引き継ぎメモ

## 目的
Addness の現行運用を前提に、`Lステップ / UTAGE / Mailchimp / short.io / Zapier` を使った施策実装を、要件が来たら高精度で進められる状態にすること。

## 確定事項
- 応答と報告は日本語で行う
- ツール理解は `ツールそのものの役割` と `アドネス株式会社の販売導線における役割` を分けて扱う
- 100点の定義は `役割 / ゴール / 必要変数 / ワークフロー / NG / 正誤判断` を説明できること
- コンテンツの定義
  - `コミュニケーションの手段であり、ユーザーをこちらが意図した認識に変換するもの`
- 販売導線の重要概念
  - `シームレスな顧客体験をコミュニケーションコンテンツ単位で描く`
- Mailchimp
  - `Journey = evergreen`
  - `Campaign = 1回きりの segment 配信`
  - 新規実装の main CTA は `short.io` を正とする
  - tag は英語で、基本は `media_funnel_event`
- Lステップ
  - action 設定の主戦場は `流入経路 / テンプレート(Flex) / シナリオ step`
  - `友だち追加時設定` は基本的にあまり触らない
  - current 思想は `まずタグで状態を作る -> そのタグに応じてテンプレートやシナリオを適用`
- UTAGE
  - メール本線ではなく `ページ / 会員サイト / 商品 / detail / action / 動画 / メディア` の基盤
  - `登録経路` は `どこから来たかを知りたい時に切る分析軸`
  - 価格、商品名、流すシナリオ、設定構成、会員サイト解放先が変わるなら別商品に寄せる
- short.io
  - 導線リンクの正本基盤
  - 差し替えと失効対策のために使う
- Zapier
  - relay layer
  - front 体験を作るのではなく、event を次システムが読める action に変換する

## 判断とその理由（最重要）
- 今は `動線データ基盤` の構築より、施策をすぐ設定できる理解の方が優先
- 短く雑な役割定義ではズレるため、各システムは
  - 役割
  - 入力
  - 変換
  - 出力
  - 接続先
  - 制約
  - 失敗条件
  - 正誤判断
  の粒度で見る
- 動画学習は `見た` で終わらせず、
  - 章ごとに手順
  - 判断基準
  - NG
  - live UI 差分
  まで落としてから正本化する
- Mailchimp の current 実装に direct LIFF が残っていても、今後の標準は `short.io` に寄せる
  - 理由は失効対策、差し替え容易性、分析、一貫した運用のため
