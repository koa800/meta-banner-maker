# 引き継ぎメモ

## 目的
Addness の現行導線と運用システムを、要件が来たらすぐ実装できる粒度まで理解すること。
対象は主に `Lステップ / UTAGE / Mailchimp / short.io / Zapier`。

## 確定事項
- `コンテンツ` の定義は Addness 固有ではない。
- 定義は `コミュニケーションの手段であり、ユーザーをこちらが意図した認識に変換するもの`。
- この定義は販売導線だけでなく、広告、SNS、LP、メール、LINE、会員サイト、セミナー、個別相談まで適用する。
- 各ツールの理解は `ツールそのものの役割` と `Addness の販売導線における役割` を分けて持つ。
- Mailchimp は `Journey = evergreen`、`Campaign = 単発の segment 配信`。
- 今後こちらが作る Mailchimp の main CTA は原則 short.io。
- UTAGE の `登録経路` は、広告IDだから必ず切る機能ではなく、`どこから来たかを知りたい時に切る分析軸`。
- Lステップの action 設定の主戦場は `流入経路 / テンプレート(Flex) / シナリオ step`。`友だち追加時設定` は基本あまり触らない。
- 今後 Addness 株式会社の業務で新しく作る Zapier 資産は、原則 `甲原` フォルダ配下に置く。
- 既存の current Zap は無理に移動しない。`新規は甲原、既存はそのまま読む` で運用する。

## 判断とその理由（最重要）
- まず 1 ツールを 100 点にする時は、機能網羅ではなく `役割 / ゴール / 必要変数 / ワークフロー / NG / 正誤判断` を持てているかで判断する。
- `コンテンツ` は Mailchimp だけの話にするとズレる。上位定義として固定し、各システムがそのコンテンツをどう作るか、どう変換するか、どう次へ渡すかで役割を定義する。
- Addness の実装判断では、ツール一般の役割と Addness 内での役割を分けないとズレる。
- 分析基盤は後回しでよい。先に `施策をすぐ実装できる理解` を完成させる方が優先。
- Zapier の既存 current を無理に整理し直すより、新規作成先を `甲原` に固定した方が、今後の保守と棚卸しが安定する。
