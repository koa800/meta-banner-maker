## 判断とその理由（最重要）
- LP別の下流数値は誤帰属を避けることを最優先した。理由は、同じCRコードが複数LPにまたがるケースが多く、そのままLPへ配賦すると誤読が起きるため。
- Meta raw と CDP の結線は `CRコード` を正本にした。理由は、full `広告名` の完全一致は 0 件で、`CRリンク` も watch URL が多く安定キーにならなかったため。
- `見られるがクリックされない` や `クリックされるがCPAが高い` などの表現は、抽象語ではなく具体指標で読める形を維持した。理由は、ユーザーから「何を指しているか分かるように」と明示リクエストがあったため。
- `下流数値から見える仮説` は現時点では knowledge 止まり。理由は、下流数値が見え始めた段階であり、rules 化するには期間や具体CRの再現がまだ必要なため。
- `CVR` はこの文脈では `オプトイン率` に固定し、`購入者率` とは分けた。理由は、ユーザーから現文脈の意味を明示で指定されたため。
- LP を共有・判断するときは `LP1 / LP4` のようなラベルではなく `URL` を主に使う。理由は、同じラベル名が複数あり認識ズレが起きるため。
- `購入率` は曖昧なので、今の分析では `商品購入率（フロント+バック全体）` と書く。理由は、現時点の結線が商品別未分割で、フロント/バック別購入率ではないため。
- Meta広告の CR 分析では、`顧客の質` や `ターゲット層の質` という語を使わず、`ターゲット層の属性` と書く。理由は、いま切り分けたいのは良し悪しではなく、狙った属性かどうかだから。
- さらにこの原則は Meta広告に限らず全体ルールへ上げた。`顧客の質` のような抽象語を使わず、`属性 / LTV / 返金率 / オプトイン率 / 購入率` に分解して書く。
- `CVR = オプトイン率` ではない。`CVR` は抽象語で、必ず `何に対しての何率か` まで具体化する。例: `LP到達に対するオプトイン率`、`面談実施に対する購入率`。
- `運用差` を人ではなく `オーディエンス / 広告ID / 広告セットID` 側で読んだ。理由は、同一アセット・同日・同LPで `運用者名` が違って数値がズレた群が `0` だったため。
- 同一アセット比較では、`時期差` が最も群数が多く、CPA差中央値も最大だった。次に疑う順番は `時期差 -> LP差 -> 配信条件差` が自然。
- `CR別データ` シートは LP 帰属の正本として使う方が良い。理由は、Meta raw の広告名推定より `CRコード -> LP` の一意解が増え、実際に LP 解像度が上がった CRコードが `318` 件あったため。
