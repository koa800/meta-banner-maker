### 実データ照合で分かったこと

- `決済データ` には `メールアドレス / 名前 / 電話番号 / 定期課金ID / メタデータ（JSON）` があり、商品特定の補助キーとして使える
- `UTAGE補助` にも `メールアドレス / 商品名 / 金額 / 売上日時` があり、`決済データ` と照合できる
- ただし、UnivaPay の `商品名空欄 + 売上成功` の大口ブロックはかなり大きい
  - 件数: `18,582`
  - メール + 金額で `決済データ` 側の非空商品へ一意に寄るのは `42`
  - メール + 金額で `UTAGE補助` 側へ一意に寄るのは `35`
  - メールだけで一意に寄る件数は `決済データ 1,138 / UTAGE補助 2,259` だが、複数商品メールが多く自動確定には危険
- blank UnivaPay の `メタデータ（JSON）` は `charge_metadata / token_metadata` を持つが、中身は `univapay-name / univapay-link-id / univapay-phone-number` で、商品名は入っていない
- INVOY の generic 名
  - 例: `スキルプラス お支払い代金`
  - `メール + 金額` でも `決済データ / UTAGE補助` と一意に寄らない
  - 現在は甲原さんの判断を受けて、generic な `スキルプラス請求名` は `スキルプラススタンダードプラン` へ寄せる

### スキルプラス着金売上の確定定義（次に固める正本）

- 目的
  - `商品名空欄` や `プラン未特定` の成功売上を、`確定 / 候補 / 完全不明` に分ける
  - `スキルプラスっぽい` ではなく、`何を根拠に着金売上へ入れたか` を説明できる状態にする

- 決済データ（収集）で使える主要キー
  - `参照システム`
  - `イベント日時`
  - `イベント金額`
  - `課金ステータス`
  - `商品名`
  - `メールアドレス`
  - `電話番号`
  - `名前（原本）`
  - `定期課金ID`

- 証拠の強さ
  - `商品確定`
    - 決済データの `商品名` が `決済商品変換マスタ` で正式商品へ確定している
  - `事業確定`
    - 決済データの顧客キーが、スキルプラス加入後にしか発生しない一次データへ一致する
    - 候補: `入会時アンケート`、`オンボーディング回答`、`契約締結日が入った契約データ`
  - `参考一致`
    - スキルプラス導線内にいた証拠には当たるが、購入確定の証拠ではない
    - 例: `集客シート`、`個別予約シート`
  - `完全不明`
    - 上記のどれにも十分な根拠がなく、現行ロジックでは採用も除外も安全にできない

- 加工での扱い
- `商品確定` はそのまま着金売上へ入れる
- `事業確定` は `スキルプラス本体（プラン不明）` として `新規着金売上` または `継続課金売上` へ入れる
  - `参考一致` は着金売上へ入れない
  - `完全不明` は着金売上へ入れず、`売上サマリー` だけで監視する

- 参照先の優先順位
  - 正の証拠
    - `入会時アンケート / オンボーディング回答`
    - `契約締結日が入った契約データ（現状候補は 全面談合算）`
  - 補助証拠
    - `個別予約シート`
    - `集客シート`
  - 負の証拠
    - `他事業の成約リスト`

- 実装意図
  - `顧客マスタ` は統合結果なので、正の証拠の正本にはしない
  - `スキルプラス受講生データ（元）` は収集の正本に残し、`【アドネス株式会社】スキルプラス受講生データ（加工）` を決済加工の参照先に使う

## 数値管理シートとの突合結果

2025/07〜2026/02の着金売上を数値管理シートと比較検証済み。

- UnivaPayの商品名空（リカーリング2回目以降）とデザジュク系を除外して集計すると、合計で-2.8%（¥7,500万の不足）
- 不足の主因: UnivaPayの商品名が空でも高額（¥798,000等）のバックエンド商品がある。これを含めれば数字は近づく
- 結論: **収集データは揃っている。加工段階でフィルタを正しく設定すれば正確な着金売上が出せる**

---

## 判断とその理由

- `KPIダッシュボード` に元データや複雑なロジックを入れない → 表示層を軽く保つため
- `収集データ -> 加工データ` の順序を崩さない → 責務が混ざるのを防ぐため
- 収集データにはフィルタしない。全イベント・全ステータスを入れる → 加工で柔軟にフィルタできるように
- UTAGE売上一覧は決済データタブとは別タブ → UnivaPayと金額が重複するため。登録経路の補助データ
- きらぼし銀行の法人振込は除外 → 決済売上ではなく企業間の立替金振込
- UnivaPayの認証プロセス（3-Dセキュア等）は収集データから除外 → 決済の事実ではない
- デザジュク系の除外は加工の責務 → 収集には入れておく
