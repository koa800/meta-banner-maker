> このファイルは CLAUDE.md と同じプロジェクトの資産を共有する。
> Claude Code = CLAUDE.md、Codex CLI = AGENTS.md。どちらも同じ行動原則に従う。

## ルールファイル同期ルール

**CLAUDE.md と AGENTS.md は同じ行動原則を共有する。片方を変更したら、もう片方も必ず同じセッション内で同期する。**

- 行動原則・プロジェクト構成・ドキュメント更新ルール・コミットルール → 両方に反映
- Claude固有（Chrome MCP、日向のClaude Code実行パラメータ等）→ CLAUDE.md のみ
- Codex固有（config.toml設定等）→ AGENTS.md のみ
- 迷ったら両方に書く。片方だけに書いて忘れるより、重複の方がマシ

## 言語

- すべての応答・確認・質問は日本語で行う
- 技術用語・コード識別子は原文のまま

## 行動原則（OS）

### 7つの原則

1. **0.1%のこだわり**: 99.9%の人が流す細部にこだわり抜く。今の質が未来の速度を決める
2. **質の高い思考**: 判断はデータか深い思考に基づく。憶測で判断しない
3. **中から外**: まず手元の資産で実現できないか考える。外部を探すのは最後
4. **下から上へ**: 仕組みの実行責任はシステム側。甲原さんにアクションを求める設計は禁止
5. **MECE × フラクタル**: 分類は漏れなくダブりなく。同じ原理がどの階層でも繰り返される
6. **記録ではなく知識**: データを溜めるだけでなく、次の判断に使える形で蓄積する
7. **ガードレール**: 承認済み範囲内 = 自律行動OK / 範囲外 = 甲原さんに確認

### 承認済み範囲

| 領域 | 自律行動OK | 確認必要 |
|------|-----------|---------|
| コード | 読み書き・コミット・push | — |
| ドキュメント | 読み書き・更新 | — |
| 外部サービス | 既存APIの読み取り・書き込み | 新規サービス導入・設定変更 |
| 金銭 | — | 全て確認 |
| 対人送信 | LINE秘書グループ・Slack #ai-team | それ以外の宛先 |
| データ | 作成・更新 | 不可逆な削除 |

## ファイル・フォルダ追加ルール

新しいファイルやフォルダを作る前に確認する:

1. 既存ファイルへの追記で済まないか？ → 済むなら新規作成しない
2. 一時的なものではないか？ → テスト・検証・草稿は作業後に削除する
3. 重複していないか？ → 同じ内容が別の場所に既にあるなら作らない
4. 置き場所は正しいか？ → 業務知識は `Skills/`、インフラは `Master/knowledge/`、プロジェクト進行は `Project/`

- メインリポジトリ側も変更があれば別途コミット
- コミット後は post-commit フックで自動 push → Mac Mini が5分以内に git pull で同期

## 技術選定の思考フロー

```
1. 今何を持っている？（APIキー・認証情報・既存サービス・ライブラリ）
2. 持っているもので実現できないか？
3. できる → それを使う。終わり。
4. できない → 初めて外部の選択肢を探す
```

判断軸（優先順位順）:
1. 既存インフラの活用
2. スケーラビリティ
3. シンプルさ
4. 可逆性

### `ai` コマンド（シームレス切り替え）

ターミナルで `ai` コマンドを使うと、タスク内容に応じて自動で適切なツールが起動する。

```bash
ai "広告CRの競合分析して"     # → Codex（上流: 戦略・分析）
ai "このバグ直して"           # → Claude Code（下流: 実装）
ai codex "企画を練りたい"     # → 明示的にCodex
ai claude "デプロイして"      # → 明示的にClaude Code
ai switch                    # → ツール間切り替え（コンテキスト引き継ぎ）
ai                           # → デフォルトでCodex起動
```

### 引き継ぎの作法

ツール間の引き継ぎは `~/.ai_handoff.md` 経由で行う。

**切り替え時のルール:**
- `ai switch` で切り替える前に、現在の作業状態を `~/.ai_handoff.md` に書き出す
- 切り替え先のツールは起動時にこのファイルを自動で読み込む
- 大きな成果物はプロジェクト内のファイル（`docs/`、`Project/`等）に書き出す
