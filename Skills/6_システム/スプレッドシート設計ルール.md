## 設計の考え方

### 本質だけ残してシンプルに

- 同じ情報を2箇所に持たない。1つのテーブルで完結できるなら分けない
- 「見栄えのための構造」は作らない。運用で使う情報だけを列にする
- 不要な行・列・セクションは削除する。空の構造はノイズになる
- 基本的に空行で区切らない。見やすさは罫線、結合セル、背景色、配置で作る
- 使っていない行や列を残さない。`あとで使うかも` の余白は管理ノイズになる
- 同じ意味の言葉は1つの表現に統一する。`実施日` と `架電日` のような揺れを残さない
- 見出しや区分名には、分かりにくい社内用語や英語を安易に使わない。業務側が一目で理解できる言葉を優先する
- 重要なタブほど左に置く。主タブを右側へ追いやらない
- 行はできるだけ縦に伸ばさない。基本は1行、多くても2行で収める
- 縦に伸びる列は、まず列幅を広げる。それでも足りない列だけ文字サイズを少し下げる
- 見た目調整のための細いスペーサー列は作らない。列として存在する以上、どこかのセクションで実データに使われる前提で幅を決める

### 1行が何を表すか を最初に決める

- テーブルの行が「何の単位」かを明確にする（例: 1行=1顧客、1行=1カラム定義）
- 1つのソースが複数のスプレッドシートを持つケースがある。「1ソース=1行」のような単純化は避け、実態に合った粒度にする
- ダッシュボードとマッピングを無理に1つのテーブルにまとめない。粒度が違えばテーブルを分ける

### カラムマッピング設計（データ統合シート向け）

複数のデータソースを1つのマスタに統合する場合、**カラム単位で参照先を明示する**。

- 各カラムに対して「どのスプレッドシートの、どのタブの、どの列から取得するか」を1行で定義する
- マトリクス（○印を並べる形式）ではなく、具体的な参照先（URL・タブ名・列名）を記載する
- **同一カラムに複数ソースがある場合**: 行を追加して優先度列で順序を指定。同期時は優先度順にチェックし、最初に値が見つかったものを採用
- 運用列（ステータス・最終同期日・更新数・エラー数）も同じテーブルに持つ

### データの正規化ルール

取り込み時にデータ形式を統一する。ルールはシート上にも明記する。

| データ種別 | 統一形式 | 例 |
|-----------|---------|-----|
| 電話番号 | ハイフンなし、+81→0変換 | `09012345678` |
| メールアドレス | そのまま保存（照合時のみ大文字小文字を無視） | |
| 金額 | ¥ + 3桁カンマ区切り | `¥1,200,000` |
| 日付 | YYYY/MM/DD | `2026/03/05` |
| 郵便番号 | ハイフンあり（3桁-4桁） | `150-0001` |
| 姓名 | 姓と名を分割して保存。スペース区切り or 姓辞書で推測 | |

### 自動同期を前提に設計する

- シート上のマッピング情報（URL・タブ・列）を同期スクリプトが読み取り、自動実行する設計にする
- ステータスは時間ベースで自動判定（正常 / 7日超で更新なし / エラーで停止）
- 行数が足りなくなったら自動拡張する
- 自動生成タブの体裁は手作業で直さない。列幅、文字サイズ、固定行、配置、タブ色は生成スクリプトで毎回再適用する
- 監査タブのように1枚へ集約すること自体が本質な場合だけ、1タブ内に複数セクションを許容してよい。その場合は「タブ全体に1つの整列ルール」ではなく「セクションごとの列幅・配置ルール」を持つ

## 手順（ステップバイステップ）

### Step 1: 構造設計

1. 行の粒度を決める（1行=何か）
2. 必要なカラムを洗い出す
3. カラムをグループ化する（システム管理 / 基本情報 / 運用情報 など）
4. データ統合がある場合はカラムマッピングを設計

### Step 2: シート構築

書式ルール:

**ヘッダー**

- `項目` と分かる行は、全シート共通で `青背景 / 白文字 / 太字 / fontSize 12` を基本にする
- ヘッダーは `どこが項目かを一瞬で判別できる状態` を作るために整える

| 行 | 背景色 | 文字 | 配置 |
|----|--------|------|------|
| 行1（項目行 / グループ名） | 青 `rgb(66,133,244)` | 白・太字・fontSize 12 | 中央揃え |
| 行2（カラム名） | 白 | 黒・fontSize 10 | 中央揃え |

- 行1のグループ名は結合セルで範囲を明示
- 一覧・管理シートでカテゴリ列の値が連続する場合は、カテゴリセルを結合して1つの見出しとして見せる
- 逆に、ステータス・運用・担当者のように1行ごとに意味が変わる列は結合しない
- フォントは全体 Arial で統一
- セクションを分けたい時も、まずは空行ではなく罫線とカテゴリ結合で見せる。空行は本当に意味がある時だけ使う
- 説明・要件タブでタイトル直下に見出し行を置く場合、その見出し行は本文より一段大きい `fontSize 12` を基準にしてよい

**データ行**

- 交互色（ゼブラストライプ）: 奇数行=白、偶数行=薄い青 `rgb(232,240,254)`
- **全行同じ書式**（fontSize 10・太字なし）で統一。グループの区切りは値の変化で判別できる。書式で差をつけない
- 既存シートに行や列を追加したときは、追加部分だけを整えるのではなく、**テーブル全体**にヘッダー・罫線・交互色・条件付き書式を再適用して体裁を揃える
- 空白セルに不要な背景色を残さない。意味のない塗りはノイズとして扱う

**配置の判断基準**

| データの性質 | 配置 |
|-------------|------|
| 数値（金額・回数・日数） | 右揃え |
| 日付 | 右揃え |
| テキスト（名前・住所・説明） | 左揃え |
| カテゴリ・ステータス | 中央揃え |
| ヘッダー | 中央揃え |

**罫線**

データが入る領域全体に罫線をつける。ヘッダーだけでなくデータ行も含めた全領域。

- 外枠: 太線（SOLID_MEDIUM）
- ヘッダー下: 太線
- グループ境界: 太い縦線（全行を貫通）
- 内部: 細い罫線（SOLID）
- **カラム追加時**: 新しいカラムが追加されたら、そのカラムを含む全行に罫線を再適用する。罫線が途切れた状態で残さない

**その他**

- ヘッダー行はフリーズ
- ヘッダー行は保護（警告モード）
- 列幅はカラム名が切れないように調整（全角≒14px、半角≒7px + 余白20px）
- データ行が3行以上に折り返す状態は避ける。まず列幅で吸収し、必要な列だけ fontSize を1段下げる
- サマリーと明細が同じタブにある場合、列幅はサマリーではなく「最も長い実データ」を基準に決める。上段の見た目都合で下段を読みにくくしない
- 複数セクションの監査タブでは、セクション見出し行、各セクションのヘッダー行、データ行を別物として書式設計する。見出しとデータを同じ規則で塗らない

### Step 3: バリデーション・条件付き書式

- 選択肢が限られるフィールドはプルダウンにする（ステータス、ソース種別など）
- **プルダウンに新しい値を追加する際は、必ずバリデーションの選択肢リストにも追加する。** 値だけ書いてプルダウンに含まれていない状態を作らない
- ステータスに条件付き書式: 正常=緑、未同期=黄、停止=赤
- タブに色をつける（メインデータ=青、運用・監視=緑、リファレンス=グレー）

**条件付き書式の色定義:**

| ステータス | 背景色 | RGB |
|-----------|--------|-----|
| 停止 | 赤（薄い） | `{"red": 0.957, "green": 0.8, "blue": 0.8}` |
| 未同期 | 黄（薄い） | `{"red": 1, "green": 0.949, "blue": 0.8}` |
| 正常 | 緑（薄い） | `{"red": 0.851, "green": 0.918, "blue": 0.827}` |

### Step 4: 標準タブ構成

データ統合シートには、メインデータタブに加えて以下の運用タブを標準で用意する。

| タブ | 用途 | タブ色 |
|------|------|--------|
| メインデータ | 実データ | 青 |
| データソース管理 | カラムマッピング + 運用ステータス | 緑 |
| データ追加ルール | 登録・名寄せ・更新のルール | グレー |
| 除外リスト | スタッフ・テストデータの除外対象 | グレー |
| 定義 | 各カラムの定義（カラム名/グループ/定義/データソース） | グレー |

### 防止策と検知

- 自動生成タブは保護する。人が直接編集する前提にしない
- `データソース管理` には最低でも `ステータス / 最終同期日 / 更新数 / エラー数` を持つ
- `更新数` は元ソース件数ではなく、実際に対象列や対象タブへ反映した件数を入れる
- 主ソース件数が `0` の場合は `停止`
- 取得はできたが更新が一定期間走っていない場合は `未同期`
- `UU > 総件数` のような定義上ありえない値は `停止`
- 日次データは `0件の日` も並べる。日付が飛ぶ状態で完成扱いにしない
- `なし` `メアドなし` のようなプレースホルダは、データとして保存しない
- 契約系・売上系のように必須キーがあるものは、必須キーが無い行を加工シートや収集シートに残さない

**定義タブの整合性:**
- メインデータタブのカラムと定義タブのカラムは**完全一致**させる（列数・列名・順序）
- カラム追加・リネーム時は定義タブも必ず同時に更新する

## エラー時の対処

| エラー | 原因 | 対処 |
|--------|------|------|
| Google Sheets API タイムアウト | シートが大きい / API制限 | リトライ。大きいシートは分割して処理 |
| 条件付き書式が残る | タブの clear() では条件付き書式が消えない | タブを削除→再作成で完全リセット |
| データバリデーション（プルダウン）が残る | clear() では消えない | `setDataValidation` で `rule: None` を明示 |
| 列幅が合わない | 2セクション構成で列幅が競合 | 1テーブル1構造に統一。粒度が違えばテーブルを分ける |
| ヘッダーのフォントがバラバラ | 過去のスクリプトがグループ先頭を太字にしていた | 全行同じ書式で統一。書式で差をつけない |

## API実装メモ（Google Sheets API）

### 色の指定値

| 名前 | RGB |
|------|-----|
| 青（ヘッダー） | `{"red": 0.357, "green": 0.584, "blue": 0.976}` |
| 薄い青（交互色） | `{"red": 0.91, "green": 0.941, "blue": 0.996}` |
| 白 | `{"red": 1, "green": 1, "blue": 1}` |

### 金額表記

- 通貨マーク: ¥を付ける（例: ¥800,980）
- 桁区切り: 3桁ごとにカンマ（例: ¥1,200,000）

### 交互色の設定

`addBanding` リクエストを使用。行追加時にも自動で色が適用される。

### フリーズ行

`updateSheetProperties` で `gridProperties.frozenRowCount: 2` を設定。
