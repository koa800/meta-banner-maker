# 改善を含めたワークフローづくりの考え方

最終更新: 2026-03-19

このドキュメントは、**「動かすだけ」で終わらない仕組み**を設計するための思考の型をまとめたものです。新しくルール・ワークフロー・自動化を作るとき、または既存の仕組みを見直すときに参照する。

## 前提にする考え方

- **ゴール** = やりたいこと・届けたい結果。仕組みはそのためにある。
- **仕組み** = 動かす + 検知する + 改善を考える + 変える。ここまでをセットで「作る」とみなす。
- **改善を含める** = 失敗や弱い点が出たときに「どう直すか」を考え、仕組み自体に反映する流れを最初から組み込む。

## 実行の分解原則

- タスクは必ず `ゴールを決める -> ゴールの解像度を上げる（要件定義） -> ゴールを分解する -> 1つずつ実行する` の順で扱う。
- 実行できない時は止まらず、その場でさらに分解する。
- 次の判断は常に `実行する` か `分解する` の二択で持つ。
- 分解後の小タスクにも同じ原則を再帰的に適用する。
- `何となく詰まった` は停止理由ではない。`まだ分解が足りない` とみなして粒度を落とす。

## 依頼受領時の着手前ゲート

- 非自明な依頼では、最初の返答を必ず要件定義にする。
- 最初の返答では最低でも `依頼理解 / ゴール / 完了条件 / 固定条件 / 対象範囲 / 推奨手段` を短く出す。
- この要件定義を出す前に、検索、Web閲覧、ファイル探索、ブラウザ操作、実装、設定変更、対外送信、コミットを始めない。
- 調査が必要な依頼でも、先に要件定義を返してから探索に入る。
- 列挙・網羅・一覧化の依頼では、対象の母集団を会話上で固定するまで探索に入らない。
- 要件定義を出した後に、必要な前提を自分で埋め、分解と実行に進む。

## 設計時に考える3つの軸

### 1. エラー・不具合の検知

- **何を検知するか**
  - 例外・タイムアウト・想定外の結果
  - 手順の抜け（例: 成功時だけ通知して、エラー時は何も送らない）
  - 「今回は動いたが、ここが脆い」という箇所
- **どう検知するか**
  - 人が気づく（通知・ログ・ダッシュボード）
  - コード・ルールで「ここで必ず〇〇する」と縛る
  - 振り返りや事後確認のステップを入れる

### 2. 改善の思考

- **原因を短く押さえる**  
  なぜ起きたか（抜け・曖昧さ・前提の違いなど）を一言で言えるようにする。
- **ゴールに近づく変更を選ぶ**  
  再発防止・時間短縮・確実性アップのうち、今のゴールに一番効くものを優先する。
- **何を変えるか具体化する**  
  ルールの文言 / コードの分岐 / 手順の順番 / 通知の有無 など、どこをどう変えればよいかまで落とす。

### 3. ゴールに近づく変更

- **その場で直せること**  
  ルールの追記・コード修正・手順の更新として、すぐ反映する。
- **その場で直せないこと**  
  「改善案: 〇〇すべき。理由: 〇〇」とメモし、次のタスクや仕組みの改訂に回す。

## プロセス逸脱時の改善フロー

- 進め方の逸脱を検知したら、本題を続ける前に止まる。
- `事実 -> 観察 -> 原因 -> 再発防止策` を分けて整理する。
- 再発防止策は、抜けていたゲート、曖昧だった判断基準、足りなかった検知方法のどれかとして具体化する。
- その場で直せるなら、共通正本と個別ルールを同じセッションで更新する。
- 正本更新後に、元タスクへ戻る。

## ワークフローを「作る」ときのチェックリスト

新しくルール・ワークフロー・自動化を作る／書き換えるときに、以下を確認する。

1. **成功時の動き**は定義したか？（何をしたら「完了」か）
2. **失敗・エラー時の動き**も定義したか？（誰が・何で気づき、何をすべきか）
3. **通知・報告**は成功時だけでなく、エラー時も届く設計か？
4. **弱い点**（手動依存・曖昧な手順・抜け）をどこで検知し、どう改善に繋げるか？
5. **着手前ゲート**はあるか？（要件定義を返す前に実行へ滑り込めないか）
6. **対象範囲**は固定されているか？（何を含め、何を含めないかが先に決まっているか）

## いつこの考え方を使うか

- 新しく自動化・ルール・手順を作るとき → 上記の3軸とチェックリストを最初から当てる。
- 既存の仕組みで不具合や「報告が来ない」などの問題が出たとき → 検知→改善の思考→変更の流れで見直す。
- 振り返りや「仕組みをよくしたい」という依頼があったとき → この考え方に沿って改善案を出す。

## まとめ

- **作る** = 動かす + 検知する + 改善を考える + 変える。
- エラーや弱い点を見つけたら「どう改善すべきか」を必ず思考し、可能ならルール・コード・手順に反映する。
