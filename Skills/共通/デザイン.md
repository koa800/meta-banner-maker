## 0. MPT

**MPT = 単位時間あたりの意味の量。**

- 情報を減らすことが目的ではない
- 必要な意味を削らず、より短い時間で理解できる形に圧縮することが目的
- 文字だけで詰め込むのではなく、図解、色、余白、グルーピングを使って意味のまとまりを瞬時に伝える
- 「長いから悪い」「短いから良い」ではない。**意味が濃く、迷わず読めるか**で判断する

### MPT を上げる基本手段

- 関係する情報を1つの塊として見せる
- 1スクロールごとにテーマを1つに絞る
- 重複説明を減らし、同じ意味を複数回読ませない
- 文字で読むより早い部分は、図、カード、比較配置で見せる
- 最重要の結論を最初に出し、詳細は開閉や下層で追えるようにする
- 実務ページは「資料」ではなく「次の手が動くインターフェース」として作る

### MPT が低い状態

- 情報はあるが、どこから見ればいいか分からない
- 同じことを表と本文で何度も読ませる
- 文字量は多いのに、判断に必要な意味が薄い
- 強弱がなく、全部同じ重さで並んでいる

### MPT が高い状態

- 最初の数秒で「何が起きていて、どこを見ればいいか」が分かる
- 1つのカード、1つのブロックに意味のまとまりがある
- 余白と対比で、情報の優先順位が視覚で理解できる
- 詳細は残っているが、必要な人だけが深掘りできる

## 0.1 意図

**機能も装飾も、必ず意図を説明できるものだけ残す。**

- 「便利そうだから」「なんとなく今っぽいから」では採用しない
- その要素が、今回の目的達成にどう効くかを一文で説明できなければ入れない
- 目的に直接効かない機能は、どれだけ実装できてもノイズになりやすい
- 逆に、目的に必要な意味は削らない。削るのは情報ではなく、迷いと重複
- レビュー時は「これは何のためにあるか」を先に問い、答えられない要素から削る

### 意図の例

- 開閉UI: 長いページでも最重要情報を先に見せ、詳細は必要な人だけ開けるようにするため
- sticky ナビ: 現在地と次の移動先を見失わせないため
- カード分割: 表より早く、1件ごとの判断単位を切り出すため
- 色面の差: 重要度と役割の差を一瞬で伝えるため

### 意図が弱い例

- 使わなくても困らない補助ボタン
- 読み始めを遅らせるだけの演出
- 目立つが、判断速度や理解速度を上げない装飾

### 世界的に見やすいサイトの共通原理

Mercury、Stripe、Linear、Vercel、Notion、Figma、Shopify を見ると、表現は違っても骨格はかなり共通している。

- **横幅を先に固定する**  
  大画面でも本文やカードがだらけないように、ページ全体の最大幅を先に決める。広い画面ほど「広く使う」のではなく、「読みやすい密度を保つ」。

- **ブレークポイントを明示して、列数をはっきり切り替える**  
  なんとなく縮むのではなく、1列、2列、3列、4列を明確に切り替える。中途半端な列幅が一番見づらい。
  デバイス名で決めるのではなく、**内容が苦しくなる幅**で決める。これは web.dev の responsive 設計でも一貫している。

- **意味の塊ごとにカード化する**  
  世界的に見やすいサイトは、表現より先に「この塊で何を理解してほしいか」が分かれている。カード、段、面で意味単位を分離している。

- **小見出しと大見出しで同じことを繰り返さない**  
  小見出しは番号やカテゴリ、大見出しは結論や意味。役割を分けることで、読む負荷を下げている。

- **濃い色面は本当に重要な箇所だけに使う**  
  強い背景色は少ない方が効く。常用すると視認性が落ちる。特に本文量が多い画面では、薄い面と明確な境界線の方が読みやすい。

- **余白は装飾ではなく、区切りとして使う**  
  余白があるから高級に見えるのではない。情報の始まりと終わりが分かるから理解しやすい。

- **長文は行長と行間を先に整える**  
  本文が長いページでは、可読性の大半は装飾ではなく行長と行間で決まる。  
  目安として、英語圏では 66ch 前後、WCAG では CJK 40文字以下が参照値になる。  
  行間は最低でも 1.5 以上を基準にし、段落間は行間より広く取る。

- **ナビゲーションは現在地と次の移動先だけを示す**  
  Sticky な上部ナビや目次がよく使われるが、重要なのは情報量ではなく「今どこにいて、次にどこへ飛べるか」が一瞬で分かること。
  長文の共有ページでは、mobile で常時張り付く目次は邪魔になりやすい。**mobile は開くときだけ出る disclosure、desktop は常設でも可** が安定しやすい。

- **表よりも、判断単位に切ったブロックを優先する**  
  表は一覧には強いが、修正や判断には弱い。判断させたいときは、1件ごとに意味のまとまりを切った方がMPTが高い。

- **アニメーションは意味の遷移を補助するために使う**  
  目立たせるためではなく、スクロールしたときに「新しい塊に入った」と感じさせるために使う。速く、短く、邪魔をしないことが前提。  
  ただし、**最初の1画面を隠す演出は避ける**。hero や結論は、読み始めた瞬間に見えていなければならない。

- **Apple 的な遊びは、装飾量ではなく秩序の中の余韻で作る**  
  目立つ色を増やすのではなく、薄い光、柔らかい面の重なり、静かなグラデーション、精密な余白で「遊び」を入れる。  
  情報密度の高い実務ページでも、これなら MPT を落とさずに高級感を足せる。

- **長い修正ページは、一覧ではなく修正単位で見せる**  
  シナリオごと、メールごと、URLごとにカードや accordion を切る。  
  表だけで見せると読めても、直す手が遅くなる。

- **便利な UI でも、存在感が勝ったら失敗**  
  mobile の目次や固定UIは、便利でも本文の邪魔になれば負け。  
  desktop は sticky でもよいが、mobile は disclosure のように「使うときだけ出る」方が安定しやすい。

### 100サイト走査で見えた共通項

2026-03-09 に、`市場で強く残っているサービス` を中心に 100 サイトをカテゴリ横断で走査した。  
対象カテゴリは以下。

- payments / fintech
- devtools / infra
- productivity / collaboration
- design / no-code / AI
- enterprise / marketing

そのうち 98 サイトで取得できた実装上の共通項はかなりはっきりしていた。

- **92 / 98 が media query を使っている**  
  モバイルとPCを「自然に縮む」前提ではなく、明確に切り替えている。

- **90 / 98 が max-width を使っている**  
  大画面ほど横いっぱいに広げるのではなく、読みやすい横幅を固定している。

- **91 / 98 が flex、80 / 98 が grid を使っている**  
  列と塊の設計は、ほぼ全ての勝ち筋で共通していた。

- **71 / 98 が sticky を使っている**  
  常にではないが、現在地やCTAを見失わせない工夫がかなり一般的。

- **88 / 98 が CSS変数、88 / 98 が rounded、81 / 98 が gradient**  
  表現は違っても、変数化、角丸、柔らかい背景の変化は高頻度。

- **24 / 98 しか container query を使っていない**  
  高度な新機能は必須ではない。まずは max-width と breakpoints と grid で十分戦える。

#### ここから分かること

- 勝っているサイトは、最新技術よりも**基礎の徹底**で成立している
- PC対応で大事なのは、`広げること` ではなく `横幅を制御すること`
- モバイル対応で大事なのは、`縮めること` ではなく `列を切り替えること`
- 情報設計の勝ち筋は、ほぼ `max-width + grid/flex + media query + 明確な階層` に集約される

### 事例収集の正しい進め方

事例収集は、闇雲に100サイト見ることではない。**良い母集団を作ってから、良い法則と悪い法則を分けて抽出する**ことが重要。

#### 1. 発散の母集団

まずは以下の3層で事例を集める。

- **市場で強く残っているサービス**  
  有名だからではなく、長く選ばれ続けている理由があるもの。事業として勝っているサイトは、情報設計や信頼設計に優れた要素を持ちやすい。

- **デザイン面で評価されているサービス**  
  受賞歴、UI/UXの文脈で繰り返し名前が出るもの。見せ方や演出の質が高い。

- **悪い事例・崩れている事例**  
  何が読みにくいのか、なぜ信用しづらいのか、なぜ迷うのかを言語化するために必要。悪い法則を知ると、避けるべき地雷が明確になる。

#### 2. 良い事例の選定基準

- そのサービスが市場で実際に残っているか
- そのサイトが繰り返し参考例として挙げられているか
- 自分が作るページに転用できる要素があるか
- 見た目だけでなく、情報整理、現在地、CTA、信頼設計まで成立しているか

#### 3. 悪い事例を見る目的

- どこで読む気が切れるかを知る
- どこで「何を見ればいいか分からない」状態になるかを知る
- どこで不信感や素人感が出るかを知る
- 良い要素を増やすだけでなく、悪い要素を排除する基準を持つ

#### 4. 抽出すべきもの

事例収集のゴールは、サイト名を増やすことではない。以下を言語化できる状態にすること。

- 良いサイトに共通する法則
- 悪いサイトに共通する法則
- 自分が今回の制作で真似する要素
- 自分が今回の制作で絶対に避ける要素

#### 5. 制作への転用

事例を見た後は、必ず以下に変換する。

- **真似する法則**  
  例: 横幅先決、カードで意味を区切る、濃い背景色を使いすぎない

- **避ける法則**  
  例: 中途半端な強調、重複見出し、PCで列が崩れる、全部が同じ重さに見える

### 悪いサイトに共通する法則

良いサイトを見るだけでは足りない。**何が悪いのかを先に言語化しておくと、制作時に避けやすい。**

- **全部が同じ重さに見える**  
  見出し、本文、補足、リンク、カードが全部同じテンションだと、どこから見ればいいか分からない。

- **PC幅で間延びする**  
  モバイルだけ整っていて、PCで見ると横に広がりすぎて意味のまとまりが消える。

- **初期表示で内容を隠す**  
  fade-in や scroll reveal の作り方次第で、hero や要点が一瞬見えない状態になる。最初に読むべき要素を JS の初期待ちにしない。

- **中途半端な2カラムや3カラム**  
  何となく横並びにしていて、列幅が不均衡、文字量も揃っていない。結果として縦に読むより理解しづらい。

- **小見出しと大見出しが同じことを言っている**  
  重複は情報密度を落とす。番号とカテゴリで足りるなら、繰り返さない。

- **濃い背景色を使いすぎる**  
  強調のつもりが、可読性を落とす。特に長文セクションでは逆効果。

- **開閉UIを言葉で説明しすぎる**  
  プラス記号で伝わるなら、余計なラベルはノイズになる。

- **sticky が強すぎて本文を押し下げる**  
  目次や固定ボタンが有用でも、最初の画面や本文の読み始めを奪うなら逆効果。特に mobile で起きやすい。

- **英語の内部語がそのまま前面に出る**  
  実装用語は根拠としては必要でも、読む人にとっては認知負荷になる。自然言語へ翻訳すべき。

- **1画面目で何のページか分からない**  
  ヒーローが雰囲気だけで、目的、対象、今日やることが分からないと、その時点で離脱しやすい。

- **表に頼りすぎる**  
  表は一覧には強いが、修正や判断には弱い。1件ごとの意味のまとまりを切った方が理解しやすい。

- **アニメーションが意味を持っていない**  
  ただ動いているだけでは読みやすくならない。意味の切り替わりを補助しないアニメーションはノイズ。

## 1. デザインの4原則

デザインの良し悪しは感覚ではなく、4つの原則で説明できる。すべての制作物でこの4つを意識的に適用する。

### 1-1. 近接（Proximity）

**関連する要素を近づけ、関連しない要素を離す。**

- 情報のグループが視覚的に伝わるかどうかは「距離」で決まる
- 商品画像・商品名・キャッチコピーは密集させる → ユーザーが瞬時に「これは1セットの情報だ」と認識できる
- 関連のない情報との間には十分な余白を設ける
- グループの数が多すぎると逆効果。情報を整理してからレイアウトする

**判断基準**: 初見の人が「どれとどれがセットか」を0.5秒で判断できるか？

### 1-2. 整列（Alignment）

**すべての要素を意図的に揃える。何も「なんとなく」置かない。**

- テキスト・画像・ボタンなど、見えない線（グリッド）に沿って配置する
- 左揃え・中央揃え・右揃えを混在させない。1つのブロック内では揃え方を統一する
- 要素同士の繋がりが直感的に伝わり、全体に統一感が生まれる
- 少しでもズレていると「雑」「素人っぽい」という印象を与える

**判断基準**: 要素の端を線で結んだとき、きれいに1本の線になるか？

### 1-3. 反復（Repetition）

**同じルールを繰り返し使い、一貫性を作る。**

- 色・フォント・アイコン・余白のサイズ・装飾スタイルを統一ルールで繰り返す
- 反復によって得られる3つの効果:
  - **現在地認識**: 「今どこを見ているか」が分かる（サイト内のナビゲーション等）
  - **強調効果**: パターンの中で1つだけ違う要素があると、それが際立つ
  - **緩急**: 繰り返しの中にリズムが生まれ、単調さを避けられる
- 「このデザイン、統一感がある」と感じるのは反復が効いている証拠

**判断基準**: 同じ役割の要素が、すべて同じ見た目になっているか？

### 1-4. 対比（Contrast）

**重要な情報とそうでない情報に、明確な強弱をつける。**

- サイズ・色・太さ・余白で視覚的な優先順位を伝える
- 近接・整列・反復で情報を整理すると、全体が均一になりやすい。対比で意図的にメリハリをつける
- 「ちょっとだけ違う」は逆効果。やるなら大胆に差をつける
- セール広告の「●%OFF」を大きく表示するのが典型例

**判断基準**: 最も伝えたい情報が、最も目立っているか？

## 2. 4原則の適用順序

4原則は独立しているが、実務では以下の順序で適用すると崩れにくい。

```
① 近接: 情報をグループ化する
② 整列: グループ内・グループ間を揃える
③ 反復: 色・フォント・装飾のルールを統一する
④ 対比: 優先度の高い情報を強調する
```

①〜③で整理した後に④で強弱をつける。順序を逆にすると「派手だけど読みにくい」デザインになる。

## 3. デザイン基礎用語

制作物のレビューや指示で使う基本用語。正しく理解していないとコミュニケーションがズレる。

> **4原則との違い**: 4原則（近接・整列・反復・対比）はすべての制作物で必ず適用する絶対ルール。一方、以下の用語に含まれる数値や比率（配色の70:25:5等）はあくまで目安であり、制作物のコンセプトや目的によって柔軟に判断する。原則を知った上で、意図を持って崩すのはOK。知らずに外れるのはNG。

### 3-1. 余白（ホワイトスペース）

- 要素と要素の間の「何もない空間」
- 余白の役割:
  - **整理**: 情報のグループを分ける（近接の原則と連動）
  - **視線誘導**: 余白の方向に目が流れる
  - **強調**: 余白で囲まれた要素は目立つ
- 紙面やスクリーンを情報で埋め尽くすと、すべてが埋もれて何も伝わらない
- 余白は「何もない」のではなく「意図的に空けている」

### 3-2. カーニング

- 文字と文字の間隔を微調整する作業
- フォントの特性により、文字の組み合わせによって間隔が不均一に見えることがある
- 特に大きな見出しやロゴでは、カーニングの雑さが目立つ
- 本文テキストでは自動カーニングで十分。見出し・タイトルは手動調整する

### 3-3. フォントと書体

- **書体**: 文字スタイルの分類体系（明朝体、ゴシック体、セリフ体、サンセリフ体など）
- **フォント**: 同一デザイン・同一サイズの文字セット。書体の具体的な製品
- 制作物で使うフォントは2〜3種類に絞る（反復の原則）。増やすほど統一感が崩れる

### 3-4. 配色

- 色は心理的な影響を与える。選択には根拠が必要
- 基本の3要素: **色相**（何色か）・**明度**（明るさ）・**彩度**（鮮やかさ）
- 配色の比率ルール: **ベースカラー 70% : メインカラー 25% : アクセントカラー 5%**
- アクセントカラーがCTAボタンや強調箇所に使われることが多い（対比の原則と連動）
- 色数を増やすほどデザインは難しくなる。迷ったら色数を減らす

---

## 4. 制作物別の適用ポイント

4原則は共通だが、制作物ごとに特に重視すべきポイントが異なる。

| 制作物 | 特に重要な原則 | ポイント |
|--------|--------------|---------|
| バナー広告 | 対比 > 近接 | 一瞬で伝わる必要がある。情報量を絞り、最も伝えたい1メッセージを大胆に強調する |
| 動画広告 | 反復 > 対比 | テロップ・色使い・トランジションのルールを統一。変化する瞬間に対比を使う |
| LP | 整列 > 近接 | 縦長のページで情報が多い。グリッドで揃え、セクションごとにグループ化する |
| 資料・スライド | 整列 > 反復 | ページ間の統一感が重要。テンプレートで反復を担保する |
| SNS投稿 | 対比 > 反復 | フィード上で目を止めてもらう必要がある。アカウント全体で反復、個々の投稿で対比 |
| メール | 近接 > 整列 | 読み飛ばされやすい。情報をグループ化し、余白で区切る |

## 5. デザインレビューチェックリスト

制作物を確認するとき、以下の順でチェックする。

### 4原則チェック

- [ ] **近接**: 関連情報がグループ化されているか。無関係な要素が近すぎないか
- [ ] **整列**: すべての要素が見えない線に沿って揃っているか。ズレがないか
- [ ] **反復**: 色・フォント・装飾が統一ルールで使われているか。例外がないか
- [ ] **対比**: 最も伝えたい情報が最も目立っているか。強弱が中途半端でないか

### 基礎チェック

- [ ] **余白**: 情報が詰め込まれすぎていないか。余白が意図的に使われているか
- [ ] **フォント数**: 3種類以内に収まっているか
- [ ] **配色**: 色数が多すぎないか。70:25:5の比率から大きく外れていないか
- [ ] **視線の流れ**: 目が自然に重要な情報へ導かれるか

### 目的チェック

- [ ] **誰に向けたものか**が明確か（ターゲット）
- [ ] **何を伝えたいか**が1つに絞られているか（メッセージ）
- [ ] **どう行動してほしいか**が明確か（CTA）
- [ ] 初見の人が3秒で「何のことか」分かるか

## 6. よくあるNG（アンチパターン）と改善方法

| NGパターン | 原因（どの原則に反しているか） | 改善方法 |
|-----------|--------------------------|---------|
| 情報がバラバラに散らばっていて何がセットか分からない | 近接の不足 | 関連情報を物理的に近づけ、グループ間の余白を広げる |
| テキストや画像の端が揃っていない | 整列の不足 | グリッドを設定し、すべての要素を揃える |
| ページごとに色やフォントが違う | 反復の不足 | スタイルガイドを作り、全ページで統一する |
| 全体が均一で、どこを見ればいいか分からない | 対比の不足 | 最重要情報のサイズ・色・太さを大胆に変える |
| 余白がなく窮屈 | 余白の軽視 | 要素を減らすか、余白を意図的に確保する |
| 色が多すぎてゴチャゴチャ | 配色ルール違反 | 3色に絞り、70:25:5で配分する |
| 「ちょっとだけ」サイズや色が違う要素がある | 対比の中途半端 | 同じにするか、大胆に差をつけるか、どちらかに振り切る |
