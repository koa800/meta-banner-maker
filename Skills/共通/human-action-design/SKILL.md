# Human Action Design

相手を動かす時は、送り手の都合ではなく受け手の内部ロジックから設計する。

人が動く基本の流れは `意識 / 注意` → `欲求` → `目的` → `現在地とのギャップ` → `問題認識` → `解決策探索` → `行動` である。依頼文、提案文、営業文、会話スクリプト、LP、CTA、導線設計はすべてこの流れで点検する。

必要に応じて [movement-model.md](./references/movement-model.md) を読んで、概念の定義や短い例を参照する。

## Core Principles

- 人は自分で動いていると感じながら、実際には認識のされ方や環境に強く影響される。
- 相手が `どんな状況で / どの流れの中で / 何と混ざって` 情報を受け取るかを先に想定する。単独で読むのか、前後文脈があるのかで最適な表現は変わる。
- こちらの `やってほしいこと` を起点にせず、相手の `より良くなりたい` という欲求を起点にする。
- 相手が口にした表面の要望だけでなく、その裏にある `本当の目的 / 迷い / 不安 / 欲求` を取りにいく。
- 欲求を、相手が自分の言葉として持てる具体的な `目的` に変換する。
- `目的` と `現在地` の差を見せて `問題` を立ち上げる。
- 依頼、提案、商品、導線は、その問題を埋める `解決策` として提示する。
- 最後は一つの具体的な次の一歩に落とす。複数アクションを同時に求めない。
- 注意の向き先が行動を左右する。情報の順番、見出し、視線誘導も設計対象に含める。

## Workflow

1. 誰を動かしたいのか、どの行動を取ってほしいのかを一文で定義する。
2. 相手がその情報をどんな状況で受け取るかを整理する。
3. その相手が本来持っている欲求を言語化する。
4. その欲求を、相手が魅力を感じる具体的な目的へ落とす。
5. 今の状態を整理する。
6. 目的と現在地のギャップを明確にする。
7. そのギャップを相手の言葉で問題として表現する。
8. 解決策を一つに絞って提示する。
9. すぐ実行できる次の一歩を一つだけ示す。
10. 送り手目線の説明、余計な抽象論、複数の依頼を削る。

## Output Format

文章を新規作成またはリライトする時は、内部的に次の順で整理する。

- `相手の欲求`
- `相手の受信状況`
- `相手の目的`
- `現在地`
- `ギャップ / 問題`
- `提示する解決策`
- `今すぐの次の一歩`
- `最終文面`

ユーザーが完成文だけを求めていても、この順で考えてから書く。

## Channel Adaptation

- `依頼文 / マネジメント文`
  目的と期待成果を先に示し、次の一歩を一つに絞る。
- `提案文 / 営業文`
  先に問題とその重要性を見せ、その後に解決策を出す。
- `LP / CTA / 導線`
  欲求、問題、解決策、行動の順で注意を流す。
- `会話スクリプト`
  問いかけで相手自身に目的とギャップを言語化させる。

## Anti-Patterns

- 理由を作らずに `これをやってください` から入る。
- 相手が問題を認識する前に解決策を押し込む。
- 一つで十分な場面で複数の行動を求める。
- 罪悪感、圧、曖昧な緊急性で押し切ろうとする。
- 相手の内部ロジックが見えていないのに送り手目線で説明を重ねる。

## Quality Check

出力前に次を確認する。

- 相手の欲求が文中で明示または強く示唆されているか。
- 目的が具体的で、相手が自分事として想像できるか。
- 現在地とギャップが見えるか。
- 提示した解決策が飛躍ではなく橋になっているか。
- 次の一歩が一つで、今すぐ実行できるか。
- 文面が `やらされる行動` ではなく `自分のための行動` に見えるか。

文面が押しつけがましく見える時は、表現より上流を疑う。欲求、目的、ギャップのどれかが薄いことが多い。
