# Spreadsheet KPI Foundation

Googleスプレッドシートで業務基盤を作る時は、`見やすいシートを作る` ではなく `何を正本にするかを固定する` ところから始める。

この skill は、KPIシート、顧客管理シート、進行管理シートなどを `元データ / 加工データ / 表示` に分けて再設計する時に使う。

## Core Principles

- まず `母集団` を固定する。誰を数えるのか、どの行を対象にするのかを先に決める。
- `1項目につき正本は1つ` にする。登録日、売上、予約数など、項目ごとに正本を1つに絞る。
- `元データ` と `加工データ` と `会議で使う数` を混ぜない。
- `日常運用ルール` と `初回整備の補完処理` を分ける。
- `自動生成する場所` と `人が触る場所` を分ける。
- `表示用タブ` を正本にしない。見やすいことと、信頼できることは別物として扱う。
- 壊れた値で上書きするくらいなら止める。異常時は更新停止を正とする。
- 運用ルールは、現場の人が理解できる日本語で固定する。
- `項目` と分かる行は、青背景 / 白文字 / 太字 / fontSize 12 を基本にして、一瞬で見出しだと分かるようにする。
- 対象シートで重要なタブほど左に置く。主タブを右へ追いやらない。
- 行はできるだけ縦に伸ばさない。基本は1行、多くても2行で収める。
- 縦に伸びそうな列は、先に列幅を広げる。それでも足りない列だけ文字サイズを少し下げる。

## Workflow

1. 対象業務を一文で定義する。
   例: `メール集客の重複なし人数を信頼できる形で持つ`
2. `母集団` を固定する。
   例: `顧客マスター + メールのみ`
3. 主要項目ごとに `正本` を決める。
   例: `登録日の正本 = 集客データシート A列`
4. シートやデータを次の4区分に分ける。
   - `元データ`
   - `加工データ`
   - `会議で使う数`
   - `判定ルール`
5. `日常運用で使うソース` と `初回整備だけで使う補助ソース` を分ける。
6. 自動生成タブを決める。
7. 自動生成タブに対応する `データソース管理` と `データ追加ルール` を作る。
8. 異常検知条件を決める。
   例: `主キー重複 > 0`, `件数が前回比で急減`, `母集団が 0`
9. 自動更新頻度を決める。
   精度が落ちるなら無理にリアルタイム化しない。
10. 自動生成タブを保護し、手編集を防ぐ。

## 防止策

スプレッドシートを作る時は、`正しく作る` だけでなく `壊れにくくする` ところまでを完了条件にする。

- `自動生成タブ` は保護する。人が直接書き換える前提にしない。
- `データソース管理` を必ず作る。最低でも
  - `対象項目`
  - `ソース元`
  - `参照タブ`
  - `参照列`
  - `取得条件`
  - `ステータス`
  - `最終同期日`
  - `更新数`
  - `エラー数`
  を持つ。
- `データ追加ルール` を必ず作る。人が運用で迷わないようにする。
- `共通除外マスタ` や `無条件除外ルール` のような共通ルールは、説明文ではなく機械参照できる形で持つ。
- 主タブ以外の不要な列・行・タブは残さない。`あとで使うかも` の空構造は防止策にならず、むしろ事故源になる。
- 更新スクリプトは `idempotent` にする。同じ処理を再実行しても壊れない前提で設計する。

## 検知

異常検知は、`失敗したら気づく` ではなく、`壊れた値で上書きしない` まで含める。

- `ステータス` は `正常 / 未同期 / 停止` の3値で統一する。
- 次の条件では `停止` もしくは `未同期` に上げ、更新を止める。
  - 主ソース件数が `0`
  - 前回に比べて件数が異常に急減
  - 必須列が空になる
  - 定義上ありえない値が出る
    - 例: `UU > 総件数`
    - 例: `契約締結日なしで解除イベントあり`
  - 解析対象件数と保存件数の差が閾値を超える
- `更新数` は `ソース総件数` ではなく、`実際にその列やタブへ反映された件数` を出す。
- `0件の日` も、日次タブでは日付を飛ばさず並べる。欠損と `0` を混同させない。
- `未接続` の項目は空欄で残し、埋まったふりをしない。

## 実装前チェック

次の4点を満たしていなければ、実装に入らない。

1. `正本` が一文で言える
2. `壊れた時に止まる条件` が言える
3. `人がどこを触ってよくて、どこを触ってはいけないか` が言える
4. `更新数` と `エラー数` が何を意味するか説明できる

## Fixed Terms

- `元データ`
  外部システムや入力口から、そのまま入ってくる一覧
- `加工データ`
  元データを、重複除外や整形で使いやすくした一覧
- `会議で使う数`
  経営会議や日報で最終的に見る数
- `判定ルール`
  何を数えるか、何を除くか、重複をどう扱うかのルール
- `母集団`
  集計対象として含める範囲
- `正本`
  その項目について最も信頼する唯一のデータ源

## Output Format

シート設計や改善案を出す時は、最低でも次を明示する。

- `目的`
- `母集団`
- `正本`
- `元データ`
- `加工データ`
- `会議で使う数`
- `判定ルール`
- `自動生成タブ`
- `保護対象`
- `異常検知`
- `更新頻度`

## Anti-Patterns

- 顧客マスター、未転換一覧、履歴、表示を同じシートで兼ねる。
- `見やすい表` をそのまま正本扱いする。
- 例外的な補完処理を日常運用ルールに書く。
- 自動生成タブを人が普通に編集できるままにする。
- 何を数えるかを決める前にグラフや体裁から作る。
- 重複や件数急減を検知せずに毎回上書きする。
- `raw` や `canonical` など、運用者が理解しにくい言葉だけで進める。

## Quality Check

- `母集団` を一文で説明できるか。
- `この項目の正本はどこか` を項目ごとに言えるか。
- `日常運用` と `初回整備` が分かれているか。
- `表示用タブ` と `正本` が混ざっていないか。
- `自動生成タブ` が保護されているか。
- 異常時に `止まる` 設計になっているか。
- 運用者が見ても理解できる言葉でルールが書かれているか。

迷ったら、体裁より先に `母集団` と `正本` を疑う。
