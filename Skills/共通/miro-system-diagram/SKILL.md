# Miro System Diagram

Miro は `見栄えの良い図` を作る場所ではなく、`後から見ても関係性を誤読しない構造` を置く場所として使う。

この skill は、スプレッドシート群、業務フロー、データの流れ、運用ルールを、人と AI の両方が読みやすい図に落とす時に使う。

## Core Principles

- まず `何を収集データ・加工データ・統合データと呼ぶか` を揃える。図を描く前に定義を合わせる。
- 図は `今動いている仕組み` だけを描く。過去の補完、一次退避、例外的な移行経路は基本的に入れない。
- ブロックには `スプシ名 / スプシURL / 役割 / 何のデータがあるか` を入れる。
- URL はブロックの外にまとめない。対応するブロックの中に置く。
- URL はクリック可能にする。Miro API を使うなら、shape の content に HTML の `<a href=\"...\">...</a>` を入れる。
- レイヤーは上から下へ意味が分かるように置く。例: `統合データ -> 加工データ -> 収集データ`
- 線は `データの流れ` だけを示す。説明ブロックを増やしすぎない。
- 補助ブロックは必要最小限にする。スプシそのものと矢印だけで伝わるなら、それが最優先。
- 文字は小さくしすぎない。本文は `24-30` を基本にする。
- 枠を大きくしなくても済むように、URL は正規形へ短くする。不要な `gid` や共有パラメータは削ってよい。

## Workflow

1. 何を図解したいのかを一文で言い換える。
   例: `メール集客データの仕組みを、後から見ても分かる形にする`
2. 分類定義を先に固定する。
   例: `収集データ / 加工データ / 統合データ`
3. 図に入れる対象を絞る。
   入れる:
   - 現在運用中のスプシ
   - 自動同期の流れ
   - 日常運用で意味がある関係
   入れない:
   - 一時的な補完ソース
   - 過去の検証用データ
   - 例外処理のためだけの一時構成
4. レイヤー順を決める。
   例:
   - 上段: 統合データ
   - 中段: 加工データ
   - 下段: 収集データ
5. 各ブロックを同じ型で書く。
   - `スプシ名`
   - `スプシURL`
   - `役割`
   - `何のデータがあるか`
6. 線とラベルを最小限で入れる。
   ラベルは `同期`, `集計`, `反映`, `削除` のような短い動詞中心にする。
7. 人が後から見ても、AI が後から読んでも迷わないかを確認する。

## Standard Block Format

各ブロックは次の順で統一する。

- `スプシ名`
- `スプシURL`
- `役割`
- `何のデータがあるか`

`何のデータがあるか` は、タブ名や主キー列を 3-4 行で短く置く。列挙しすぎない。

## Layer Definitions

図にレイヤーを置く時は、次の意味で使う。

- `収集データ`
  外部から入ってくる事実を最初に受け取る入口データ
- `加工データ`
  収集データや統合データを、集計や一覧に使える形へ整えたデータ
- `統合データ`
  顧客単位や運用単位で連動し、全体の中核になるデータ

定義がユーザーの認識とズレそうなら、先に言葉の定義を確認してから描く。

## Miro Editing Rules

- すでに相手が位置調整した後なら、まず `位置は残して本文だけ直す` を優先する。
- 構造自体が間違っている時だけ、全消しして描き直す。
- API で更新する時は、shape の content だけ patch できるかを先に小さく試す。
- 接続が不安定な時は、一括更新より `1つ試す -> 通る -> 全体反映` の順で進める。
- 文字サイズを上げたい時は、枠を大きくする前に `文言圧縮` と `URL短縮` を試す。

## Anti-Patterns

- 図の右側に URL だけを別置きする。
- 過去の補完経路や一時データを本流と同じ重さで描く。
- 顧客マスタ中心に描きすぎて、入口からの流れが見えなくなる。
- 補助説明の大きな箱が主役になってしまう。
- 同じブロックごとに書き方が変わる。
- 線の説明が長すぎて、何が動いているのか逆に分からなくなる。

## Quality Check

- この図は `今の仕組み` だけを描いているか。
- レイヤー定義はユーザーの認識と揃っているか。
- 各ブロックに `スプシ名 / URL / 役割 / データ内容` が揃っているか。
- URL はクリック可能か。
- 人が見ても AI が見ても、入口から活用先まで追えるか。
- 補助ブロックがなくても意味が通るか。

迷ったら、説明を増やすより `分類を揃える` と `ブロックの型を揃える` を優先する。
