# shortio-link-operations

## Overview

Addness の short.io 運用を、動画の作法と current UI の両方に合わせて再現する skill です。

この skill の目的は、short.io を「短縮URL作成ツール」としてではなく、`導線のリンク正本基盤` として扱えるようにすることです。

上位定義として、コンテンツを `コミュニケーションの手段であり、ユーザーをこちらが意図した認識に変換するもの` と扱う。
short.io はコンテンツそのものを作らないが、どのコンテンツへ、どの順番で、どの安全性で送るかを支える基盤である。
つまり short.io は、Addness の導線実装においてコンテンツ OS を壊さずに実行するための link control system として扱う。

理解の順番は次の 6 つです。
- 役割
- ゴール
- 必要変数
- ワークフロー
- NG
- 正誤判断

## ツールそのものの役割

short.io は、長い URL を短くするだけでなく、リンクの作成、編集、差し替え、クリック計測を 1 箇所で管理する link management system です。

## アドネスでの役割

アドネスでは short.io を、LINE、UTAGE、Mailchimp、外部ページへ送る導線リンクの正本として使う。見た目の短縮より、リンク失効対策、差し替え容易性、台帳管理、クリック把握を優先する。

### 役割

short.io の役割は、導線のあらゆる設置箇所に散らばるリンクを 1 箇所で管理し、BAN やリンク失効が起きても、設置箇所を触らずに遷移先を差し替えられる状態を作ることです。

### ゴール

ゴールは 4 つです。
- 正しい命名で短縮URLを作る
- 正しいフォルダに置く
- URL管理シートに必ず記録する
- 差し替えは設置箇所ではなく short.io 側で行う

このゴールをより速く、安全に、再現性高く達成できるなら、ワークフローは更新してよい。
手順そのものが正本なのではなく、ゴール達成に対して最適な手順が正本です。

### 必要変数

最低限、次の変数を埋めてから作業する。
- 何の導線か
- どこに設置するか
- 最終遷移先は何か
- 流入経路名は何か
- リンクタイトルは何か
- slug は何にするか
- どのフォルダに入れるか
- URL管理シートのどこへ記録するか

Lステップ の `流入経路分析` と対で作る時は、さらに次を固定する。
- `リンクタイトル = 流入経路名`
- SNS は `（アカウント名）集客媒体_ファネル名_イベント名`
- `直リストイン = 直接LINE登録`
- `リストイン = メール登録後にLINE登録`

この skill は、上の変数が曖昧なまま作業しない。

### 判断フレーム

#### 新規作成する

次のどれかが変わるなら、新規 short.io を作る。
- 設置場所が変わる
- 流入経路名が変わる
- 分析したい単位が変わる
- ファネル名やイベント名が変わる
- 同じ遷移先でも、どこから来たかを分けて見たい

#### 既存リンクを差し替える

次の条件をすべて満たすなら、既存 short.io の `元のURL` を差し替える。
- 役割が同じ
- 設置場所が同じ
- 流入経路名が同じ
- 分析単位も同じ
- 変わるのが最終遷移先だけ

#### 既存リンクを流用してよい

次の条件をすべて満たすなら、既存 short.io をそのまま流用してよい。
- 設置場所が同じ
- 役割が同じ
- 計測を分ける必要がない
- 今後の差し替えも同じ単位で扱う

この 3 つを説明できない場合は、作業前に判断を止める。

### 設置場所の考え方

`設置場所` は、見た目上の部品名ではなく、運用上区別したい粒度で決める。

- 通常は `媒体 × 導線 × メール or ページ` を 1 単位で切る
- 同じページ内の複数CTAでも、分析を分けないなら同じ short.io でよい
- 同じ遷移先でも、媒体や導線が違えば別 short.io を作る

## 実装前の最小チェック

- `新規作成 / 既存差し替え / 既存流用` のどれかを先に切っている
- `設置場所 / 流入経路名 / 最終遷移先 / フォルダ` を決めている
- URL管理シートの記録先まで決めている

## Workflow

1. 先に流入経路名かリンクタイトルを確定する
2. short.io で短縮URLを作る
3. `リンクタイトル = 流入経路名` で揃える
4. slug を媒体や導線のルールに沿って調整する
5. 正しいフォルダへ入れる
6. URL管理シートを更新する
7. 差し替え時は既存リンクを検索して、元URLだけを差し替える

## exact 手順

1. 一覧で既存リンクを検索する
2. `新規作成 / 既存差し替え / 既存流用` を切る
3. 新規なら `ブランドリンク` で `元のURL / リンクタイトル / slug / フォルダ` を設定する
4. 差し替えなら既存リンクの edit で `元のURL` を更新する
5. 保存後に統計または preview で intended な遷移先を確認する
6. URL管理シートの `遷移先 / 更新日` を更新する

### API helper の exact 補助経路

- create:
  - `python3 System/scripts/shortio_api_client.py create-link --original-url "<url>" --path "<slug>" --title "<title>"`
- resolve:
  - `python3 System/scripts/shortio_api_client.py resolve "https://skill.addness.co.jp/<slug>"`
- update:
  - `python3 System/scripts/shortio_api_client.py update-link "<link_id>" --original-url "<url>" --title "<title>"`
- delete:
  - `python3 System/scripts/shortio_api_client.py delete-link "<link_id>"`

## 保存前の最小チェック

- `リンクタイトル = 流入経路名` にできているか
- 役割の違う設置場所へ流用しようとしていないか
- 既存差し替えなら、役割と分析単位が同じと言えるか

## 保存後の最小チェック

- 作成または差し替えした short.io を実際に開けるか
- intended な最終遷移先へ行くか
- URL管理シートの `遷移先 / 更新日` が更新されているか

## 最小 live change

既存 current を壊さずに exact 性を上げる時は、最初の 1 回を `1 link 1変更` に制限する。

最初に触る候補
- `title`
- `slug`
- `originalURL`
- `folder`

順番
1. 対象 link を 1 本に固定する
2. `新規作成 / 既存差し替え / 既存流用` を 1 文で固定する
3. 変更対象を 1 つだけ決める
4. 保存する
5. 実 click で intended な最終遷移先を確認する
6. URL管理シートの `遷移先 / 更新日` を揃える
7. exploratory なら元に戻す

最初からやらないこと
- `slug` と `originalURL` を同時に変える
- `folder` 変更とリンク差し替えを同時にやる
- 複数 link を一度に直す

## 最初の複雑パターン

最初に exact に増やす 1 本は、次のどれか 1 つに絞る。

- `新規作成 -> URL管理シート更新 -> 実 click -> final destination 確認`
- `既存差し替え -> 実 click -> 旧設置箇所の downstream 確認`
- `既存流用 -> 設置場所の意味が同一か確認 -> 実 click`

最初から
- 複数 link の一括差し替え
- folder 移動と URL 差し替えの同時実行
- 台帳未更新のまま放置
をやらない。

## 症状から最初に疑う場所

- short URL は開くが意図した場所に着かない
  - まず `originalURL`
  - 次に final destination
  - その後に設置箇所の意味づけ
- 同じ遷移先なのに実績が混ざる
  - まず `既存流用` 判断
  - 次に title と設置場所
  - その後に URL管理シートの分類
- 直したはずなのに downstream が古いまま
  - まず実 click
  - 次に browser cache
  - その後に設置箇所が別 link を踏んでいないか

## 構築精度だけを見る時のチェック

1. `新規作成 / 既存差し替え / 既存流用` を 1 文で言えるか
2. `title` が流入経路名や設置場所の意味と一致しているか
3. `originalURL` の intended destination を確認したか
4. 実 click で final destination まで押したか
5. URL管理シートの `遷移先 / 更新日` を揃えたか

## ここで止めて確認する条件

- 同じ遷移先だからという理由だけで流用したくなっている
- 設置場所は違うが、分けるべきか判断できない
- short.io の title とシート上の意味づけが一致しない
- 差し替えで済むのか新規作成すべきか 1 文で説明できない

## 完成条件

- `新規作成 / 既存差し替え / 既存流用` の理由を説明できる
- short.io 側の設定と URL管理シート更新が 1 セットで終わっている
- 実 click で最終遷移先まで確認している

## Execution Templates

### ケース1: 新規作成

使う条件:
- 設置場所が新しい
- 分析単位を新しく切りたい
- 既存 short.io を流用すると後で混ざる

埋める順番:
1. 集客媒体
2. ファネル名
3. イベント名
4. 設置場所
5. 最終遷移先
6. 流入経路名
7. リンクタイトル
8. slug
9. フォルダ

完了条件:
- short.io 作成済み
- `リンクタイトル = 流入経路名`
- URL管理シート記録済み

### ケース2: 既存差し替え

使う条件:
- 役割は同じ
- 設置場所は同じ
- 流入経路名は同じ
- 分析単位も同じ
- 最終遷移先だけ更新したい

埋める順番:
1. 対象 short.io
2. 現在の設置場所
3. 現在の役割
4. 新しい遷移先
5. 差し替え理由

完了条件:
- `元のURL` 差し替え済み
- 設置箇所は未修正で済んでいる
- URL管理シートの遷移先と更新日を更新済み

### ケース3: 既存流用

使う条件:
- 設置場所が同じ
- 役割が同じ
- 分析を分ける必要がない
- 今後の差し替えも同じ単位で扱う

埋める順番:
1. 既存 short.io
2. 流用先の設置場所
3. 分析を分けない理由

完了条件:
- 流用理由を説明できる
- 後で分けたくなる要件がない

## Guardrails

- LINE に送るリンクは原則 short.io で管理する
- Mailchimp で新規に作る Journey / Campaign の `main CTA` も原則 short.io を使う
- LIFF や LINE の長い直URLを UTAGE や Mailchimp に直書きしない
- 名前が曖昧なまま作らない
- short.io を作っただけで終わらず、URL管理シート更新までを 1 セットにする
- 既存リンクの差し替えは、設置箇所を修正する前に short.io 側で完結できないか先に見る
- 記録先は、hidden の `01_全体台帳` と visible の `集客媒体タブ` を優先する

## Good / Bad

### 正しい状態

- short.io の役割を「リンク正本基盤」として説明できる
- リンク作成前に、流入経路名と設置場所が確定している
- `リンクタイトル = 流入経路名` で揃っている
- URL管理シート更新まで 1 セットで完了している
- 差し替え時に設置箇所を触らず、short.io 側で完了している
- `新規作成 / 差し替え / 既存流用` の理由を説明できる

### 間違った状態

- とりあえず短縮して配る
- リンクタイトルと流入経路名がずれている
- どのフォルダに入れるか曖昧
- シート記録を後回しにする
- 設置箇所を先に直してしまう
- 同じ遷移先というだけで、別の設置場所に同じ short.io を流用する

## Current UI

current の画面差分や exact route は [references/workflow.md](references/workflow.md) を読む。

特に次のときは必ず読む。
- 新規作成の current UI が動画と同じか確認したいとき
- 既存リンク編集の入口が見つからないとき
- ドメイン統計や API 画面まで見る必要があるとき
