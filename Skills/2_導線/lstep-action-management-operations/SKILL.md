# lstep-action-management-operations

## Overview

Lステップの `アクション管理` を、`新規作成 -> 保存 -> 検索 -> 削除` まで exact に再現する skill です。

Addness 固有の current representative、account 選定、命名規則は `Master/knowledge/lstep_structure.md` を正本にする。この skill は、会社をまたいで再利用できる exact 手順だけを持つ。

live で触る前に、必ず `python3 System/scripts/funnel_tool_live_readiness.py` を実行し、`lstep.auth_alive=true` に加えて `lstep.account_name` が intended な公式LINE account か確認する。`auth_alive=true` でも別 account 文脈を見ていることがある。

## ツールそのものの役割

`アクション管理` は、複数の処理を reusable な 1 つの action に束ね、流入経路、テンプレート、シナリオ step などから呼び出せるようにする機能です。

## アドネスでの役割

アドネスでは `アクション管理` を、`テンプレート送信 / タグ操作 / シナリオ操作 / 表示制御` を束ねる reusable macro として使う。毎回 inline で書くとズレやすい複合処理を、再利用しやすい単位に切る時の主戦場です。

## 役割

`アクション管理` は、`送信 / 状態変更 / シナリオ制御 / 表示制御` を 1つの reusable macro に束ねるための機能です。

## ゴール

次を迷わずできる状態を作る。
- `アクション管理` の正しい入口を選ぶ
- 最小構成の action を作成する
- 保存後に一覧で検索して確認する
- テスト用なら削除する

## 必要変数

- action 名
- 何をまとめたいか
- reusable にするか inline にするか
- 動作種別
- 送信タイミング
- 条件の有無
- テスト後に残すか削除するか

## 依頼を受けたら最初に埋める項目

- `保存するアクション名`
- 何を 1 action に束ねたいか
- reusable にする理由
- 最初の 1 動作
  - `テキスト送信 / テンプレート送信 / タグ操作 / シナリオ操作`
- downstream で何が変わるか
- exploratory か、本線用か

## 実装前の最小チェック

- `lstep.account_name` が intended な公式LINE account と一致している
- 何を 1 action に束ねたいか 1 文で言える
- reusable にする理由があるか確認している
- inline action で足りない理由を説明できる
- 最小構成で検証するか、複合処理まで入れるか決めている

## 入口選択の exact な問い

- 今やりたいのは `同じ複合処理を何度も呼ぶ` ことか
  - なら `アクション管理`
- 今やりたいのは `その場限りのボタンや step の挙動` か
  - それはまず inline action を疑う
- 今やりたいのは `情報回収` か
  - それはこの skill ではなく `回答フォーム`

つまり、この skill の入口は `reusable な macro にする価値があるか` で決まる。

## 使い分け

- `アクション管理`
  - 何度も使う複合処理
- 流入経路 / テンプレート / シナリオ step の inline action
  - その場限りの処理

迷ったら次で切る。
- 使い回すなら `アクション管理`
- 入口ごとに少しずつ違うなら inline

## 10秒判断

- `アクション管理`
  - 同じ複合処理を何度も呼ぶ
- inline action
  - その場限りの挙動で足りる
- `テキスト送信`
  - まず最小構成で疎通を見たい
- `テンプレート送信`
  - 既存テンプレートを呼びたい
- `タグ操作 / シナリオ操作`
  - state を動かすのが主役

この 10 秒判断が言えない時は、`新しいアクション` を押さない。

## Workflow

1. `アクション管理`
2. `新しいアクション`
3. 動作種別を選ぶ
4. `保存するアクション名`
5. 動作の中身を入れる
6. `この条件で決定する`
7. 一覧で `検索` に action 名を入れる
8. `search` を押して存在確認
9. テスト用なら row 右端 `more_vert -> delete 削除 -> 削除する`

## exact 手順

### create

1. `アクション管理`
2. `新しいアクション`
3. 動作種別を選ぶ
4. `保存するアクション名`
5. 動作 block を埋める
6. `この条件で決定する`

代表 URL
- 一覧: `/line/action`

### 最小構成

いま exact に閉じている最小構成は `テキスト送信` です。

1. `新しいアクション`
2. `テキスト送信`
3. `保存するアクション名`
4. `1. テキスト送信`
5. `メッセージ`
6. visible `[contenteditable="true"]` に本文
7. `この条件で決定する`

重要
- `メッセージ` は visible `textarea` ではなく `[contenteditable="true"]` を主入口にする
- `検索` は入力だけでは反映されない。`search` ボタンまで押す

### 一覧で確認

1. 一覧へ戻る
2. `検索` に action 名を入れる
3. `search` を押す
4. row が 1 件に絞れることを確認する

### 削除

1. `検索` に action 名を入れる
2. `search` を押す
3. row 右端 `more_vert`
4. `delete 削除`
5. confirm dialog `削除する`
6. 再度 `検索 -> search`
7. row が消えていることを確認

## representative pattern

### 状態整理型

- 目的
  - タグや表示状態をまとめて整える
- 主要操作
  - `タグ操作`
  - `対応マーク・表示操作`
- 向いている場面
  - 分岐前の状態統一
  - 既存シナリオの停止前整理

### 送信トリガー型

- 目的
  - 1回の action 呼び出しで案内を出す
- 主要操作
  - `テキスト送信`
  - `テンプレート送信`
- 向いている場面
  - 流入経路やリッチメニューからの即時案内
  - 回答後や予約後の単発送信

### シナリオ制御型

- 目的
  - 既存シナリオを止める、切り替える、次のシナリオへ渡す
- 主要操作
  - `シナリオ操作`
  - `タグ操作`
- 向いている場面
  - 二重配信防止
  - ステータス移行

## 最小 live test

最初の 1 本は `テキスト送信` だけで閉じる。

1. `アクション管理`
2. `新しいアクション`
3. `テキスト送信`
4. `保存するアクション名`
5. `メッセージ`
6. `この条件で決定する`
7. 一覧 `検索 -> search`
8. row 確認
9. テスト用なら `delete 削除 -> 削除する`

最初から `タグ操作 + テンプレート送信 + シナリオ操作` の複合にしない。まず `1 動作` で save / search / delete が通るかを見る。

## 複合 action を足す順

複合化は、次の順で 1 つずつ足す。

1. `テキスト送信` または `テンプレート送信`
2. `タグ操作`
3. `シナリオ操作`
4. `対応マーク・表示操作`

`送る -> 状態を作る -> 既存導線を止める` の順で足すと、壊れた時にどこでおかしくなったかを切りやすい。

## 最小 downstream smoke

action は保存できたかだけで終わらせない。

1. intended な action 名
2. intended な 1 動作
3. intended な downstream 変化
  - `テンプレート送信 / タグ変化 / シナリオ変化`
4. 再利用する場所を 1 つ言えるか

何が変わる action か 1 文で言えない時は、複合化しない。

## 最小複合 pattern

複合の最初の 1 本は、次で止める。

- `テンプレート送信`
- `タグ操作`

ここで save / search / 呼び出し元確認まで通ってから、`シナリオ操作` や `表示制御` を足す。最初から 4 動作以上を 1 action にしない。

## 最初の複雑パターン

最初に live で詰める複雑パターンは、次の 3 動作だけで閉じる。

1. `タグ解除`
2. `タグ追加`
3. `テンプレート送信`

順番
1. `新しいアクション`
2. `タグ操作`
3. `保存するアクション名`
4. `削除するタグ` 1件
5. `追加するタグ` 1件
6. `テンプレート送信` 1件
7. `この条件で決定する`
8. `検索 -> search`
9. 呼び出し元を 1 つだけ決めて smoke
10. exploratory なら `delete 削除 -> 削除する`

最初から
- `シナリオ操作`
- `対応マーク・表示操作`
- `イベント予約操作`
を同時に足さない。

## 最小 live change

既存 current を壊さずに exact 性を上げる時は、最初の 1 回を `1 action 1変更` で閉じる。

最初に触る候補
- `保存するアクション名`
- `1. テキスト送信` の本文
- `タグ操作` の追加先または削除先 1 件

順番
1. row を 1 つに固定する
2. 主役の動作を 1 つ決める
3. 1 フィールドだけ変える
4. `この条件で決定する`
5. 一覧へ戻り `検索 -> search`
6. intended な row を再確認する
7. 呼び出し元があるなら 1 箇所だけ smoke する
8. exploratory なら元に戻すか削除する

最初からやらないこと
- `テンプレート送信 + タグ操作 + シナリオ操作` を同時に変える
- selector 系を含む action を初手で触る
- 呼び出し元未確認のまま複合 action を増やす

## 最小 rollback

exploratory に action を作った時は、一覧検索で消えるところまで戻す。

1. 一覧 `検索`
2. `search`
3. 対象 row 右端 `more_vert`
4. `delete 削除`
5. confirm dialog `削除する`
6. 再度 `検索 -> search`
7. row が消えたことを確認

既存 current の 1 field 変更なら、まず同じ field を元の値へ戻す。row ごと削除するのは exploratory に限る。

## 変更後の最小 smoke

複合 action は save できたかでは閉じない。

1. intended な `保存するアクション名`
2. intended な主役動作
3. intended な `タグ操作 / テンプレート送信 / シナリオ操作 / 表示制御`
4. 呼び出し元 1 箇所
5. downstream の state 変化

`何が先に起き、何が後で変わるか` を順番どおり言えない時は、複合 action を増やさない。

## representative pattern を読む時の問い

- この `アクション管理` は、何を 1 つに束ねるために存在するか
- `送信 / 状態変更 / シナリオ制御 / 表示制御` のどこまでを 1 action に入れるべきか
- これは reusable にする価値があるか、それとも inline で足りるか
- action 名だけで、役割と呼び出し場面が分かるか
- `タグ操作` が主役なのか、`テンプレート送信` が主役なのか、`シナリオ操作` が主役なのか
- current の良い例と比べて、責務を詰め込みすぎていないか
- 保存後に `検索 -> search` で確実に辿れるか
- この action を呼ばれた時に、次の state が 1 文で説明できるか

## 複雑パターンで先に切ること

- `reusable にする理由`
- `1 action に何個の責務を持たせるか`
- `送信 / 状態変更 / シナリオ制御` のどこまで束ねるか
- `既存 action の拡張で足りるか`
- `selector 系` が入るか
- 呼び出し元が 2 箇所以上あるか
- `タグ操作` が主役なのか、`テンプレート送信` が主役なのか

この段階で `reusable にする価値` が言えないなら、inline で足りる可能性を先に疑う。

## 検証

最低でも次を確認する。
- action 名が役割を表している
- 何をまとめる action か一読で分かる
- 一覧検索で見つかる
- テスト用は削除済み

## 保存前の最小チェック

- 目的、対象、次行動を 1 文で言える
- 入力値や action の実体が確定している
- current の row やフォルダを取り違えていない
- テスト後に残すか削除するか決めている

## 保存後の最小チェック

- 一覧に想定どおり表示される
- 必要なら `テスト送信` や preview で実機確認した
- action や遷移先が意図どおり動く
- テスト用なら削除して row が消えたことを確認した

## 呼び出し後の確認順

1. intended な `テンプレート送信` または `テキスト送信` が出たか
2. intended な `タグ操作` が起きたか
3. intended な `シナリオ操作` が起きたか
4. intended な `対応マーク・表示操作` が起きたか

複合 action は、`呼び出せたか` だけで完了にしない。`何が順番どおり起きたか` まで 1 動作ずつ切って確認する。

## 症状から最初に疑う場所

- 保存できたのに呼び出し元で期待挙動が起きない
  - まず `保存するアクション名`
  - 次に呼び出し元の指定先
  - 最後に action 内の 1 動作目
- テンプレート送信だけ起きて、その後の state が変わらない
  - `タグ操作`
  - `シナリオ操作`
  - `対応マーク・表示操作`
  の順に疑う
- 想定より多くのことが起きる
  - 1 action に責務を詰め込みすぎていないかを先に疑う
  - reusable にする前に inline で足りないかへ戻る
- action 名は読めるが、中身の主役が分からない
  - `送信`
  - `状態変更`
  - `シナリオ制御`
  のどれが主役かを 1 つに切り直す
- 一覧検索では見つかるが current で使うべきか迷う
  - action 名の用途
  - 呼び出し元
  - downstream の state 変化
  の順に見て、1 文で説明できないなら legacy 候補として扱う

## 構築精度だけを見る時のチェック

1. この `アクション管理` が `状態整理 / 送信トリガー / シナリオ制御` のどれかで 1 文になっているか
2. `保存するアクション名` だけで用途と呼び出し場面が読めるか
3. 1 action に束ねる責務が多すぎないか
4. `タグ操作 / テンプレート送信 / シナリオ操作 / 表示制御` の主役を 1 つ決められているか
5. reusable にする理由を説明できるか
6. 一覧検索で確実に辿れているか
7. テスト用なら row が消えるまで削除確認したか
8. 呼び出し元が 1 つしかないのに、過剰に reusable 化していないか
9. `selector 系` が入るなら、最小構成で先に検証する順を切れているか

## ここで止めて確認する条件

- 何を変える設定か 1 文で言えない
- current / legacy の切り分けに迷う
- 新しい外部連携や本番影響の大きい分岐を足す
- 既存 high-quality な実例と大きく異なる構成にしたい
- `タグ操作` と `テンプレート送信` のどちらが主役か言えない
- 呼び出し元が 1 箇所しかなく、inline との差が説明できない
- 複合 action を作りたいが、まず `1 動作` で save / search / delete を通していない

## 完成条件

- 正しい入口から作成または編集できている
- 保存後に一覧と実挙動で確認できている
- テスト送信が必要な場合は `甲原 海人` で確認している
- テスト用の残骸が残っていない
- action を呼ばれた後の state 変化を 1 文で言える

## NG

- reusable にする必要がないものまで `アクション管理` に寄せる
- action 名が用途を表していない
- selector が絡む複雑な動作を、live 確認なしで正本化する
- テスト用 action を残す

## 正誤判断

正しい状態
- reusable にする理由がある
- action 名だけで用途が分かる
- 最小構成を exact に保存できる
- 一覧検索で確認し、不要なら削除できる

間違った状態
- inline で足りる処理を過剰に reusable 化する
- 動作を増やしすぎて action の責務が曖昧になる
- save できたかを一覧で確認しない
