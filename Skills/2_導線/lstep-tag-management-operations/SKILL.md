# lstep-tag-management-operations

## Overview

Lステップの `タグ管理` を、`新しいタグ` の作成だけでなく `一覧確認 -> 削除` まで exact に再現する skill です。

Addness 固有の命名規則、folder の読み方、current / legacy の判断は `Master/knowledge/lstep_structure.md` を正本にする。この skill は、会社をまたいで再利用できる exact 手順だけを持つ。

## ツールそのものの役割

`タグ管理` は、友だちの状態、属性、進行位置をラベルとして持つ機能です。

## アドネスでの役割

アドネスでは `タグ管理` を、誰に何を送るかを決める state system として使う。名前を付けること自体ではなく、次のテンプレート、シナリオ、予約導線を正しく分岐させるための基盤として扱う。

## 役割

`タグ管理` は、友だちの状態、属性、進行位置をラベルとして持つ機能です。

## ゴール

次を迷わずできる状態を作る。
- `タグ管理` の正しい入口を選ぶ
- 必要な folder を開く
- `新しいタグ`
- 一覧で確認する
- テスト用なら削除する

## 必要変数

- タグ名
- どの folder に置くか
- 何の状態を表すタグか
- テスト用か本番用か

## Workflow

1. `タグ管理`
2. 必要な folder を開く
3. `新しいタグ`
4. `タグ名`
5. `作成`
6. 一覧で確認
7. テスト用なら削除

## exact 手順

### create

1. `タグ管理`
2. 必要な folder を開く
3. `新しいタグ`
4. dialog `タグの新規作成`
5. field `タグ名`
6. `作成`

### 削除

1. 対象 row 右端 `more_vert`
2. `削除`
3. 一覧再読込後に row が消えていることを確認する

## 検証

最低でも次を確認する。
- folder が正しい
- タグ名が正しい
- 一覧に出る
- テスト用なら削除済み

## 実装前の最小チェック

- 何を実装する画面か 1 文で言える
- 現在触るべき account と current / legacy を切っている
- テスト送信や削除の要否を先に決めている
- 次の行動や状態変化を 1 つに絞っている

## 保存前の最小チェック

- 目的、対象、次行動を 1 文で言える
- 入力値や action の実体が確定している
- current の row やフォルダを取り違えていない
- テスト後に残すか削除するか決めている

## 保存後の最小チェック

- 一覧に想定どおり表示される
- 必要なら `テスト送信` や preview で実機確認した
- action や遷移先が意図どおり動く
- テスト用なら削除して row が消えたことを確認した

## ここで止めて確認する条件

- 何を変える設定か 1 文で言えない
- current / legacy の切り分けに迷う
- 新しい外部連携や本番影響の大きい分岐を足す
- 既存 high-quality な実例と大きく異なる構成にしたい

## 完成条件

- 正しい入口から作成または編集できている
- 保存後に一覧と実挙動で確認できている
- テスト送信が必要な場合は `甲原 海人` で確認している
- テスト用の残骸が残っていない

## NG

- folder を開かずに作って置き場所を誤る
- 命名をその場のメモで決める
- visible 一覧が空だから何も使っていないと決める
- テスト用タグを残す

## 正誤判断

正しい状態
- 何の状態を表すタグか説明できる
- 正しい folder で作っている
- 一覧で作成確認している
- テスト用は削除済み

間違った状態
- folder 無視で作る
- 用途の曖昧な名前を付ける
- 作っただけで action 側との関係を見ない
