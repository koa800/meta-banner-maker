# utage-product-detail-action-operations

## Overview

UTAGE の `商品管理` を、商品本体だけでなく `商品詳細管理`、`購入後アクション`、`会員サイトのバンドルコース` まで一気通貫で扱う skill です。

Addness 固有の current 商品例、group の置き方、会員サイトの実体は `Master/addness/utage_structure.md` を正本にする。この skill では、どの案件でも再利用できる exact 手順だけを持つ。

## 役割

UTAGE の `商品管理` は、価格や決済の見せ方を持つだけでなく、購入後に誰をどのシナリオへ流し、どの会員サイトを解放し、どの連携を実行するかを束ねる実行面です。

## ゴール

次を迷わずできる状態を作る。
- `商品管理 -> 商品追加`
- `商品詳細管理 -> 追加`
- `購入後アクション`
- `会員サイト -> バンドルコース`
を正しい順で見られる
- 何をもって別商品にするか説明できる
- `商品本体` だけ見て終わらず、`detail -> action -> bundle` まで確認できる

## 必要変数

- 商品名
- 価格
- 支払方法
- 支払回数
- 決済代行会社
- 表示名 / 表示価格
- 購入後シナリオ
- 購入後アクション
- 会員サイト解放先
- 外部連携の有無
  - webhook
  - スプレッドシート追記

## 判断フレーム

### 別商品に寄せる

次のどれか 1 つでも変わるなら、原則別商品に寄せる。
- 商品名
- 価格
- 支払方法
- 回数
- 流すシナリオ
- `購入後アクション`
- 会員サイト解放先

### 同一商品の中で吸収する

次の条件をすべて満たす時だけ、同一商品の中で吸収を検討する。
- 同じ商品と言い切れる
- 購入後の顧客体験も同じ
- 差分が表示名や軽微な表現だけ

## Workflow

1. `商品管理`
2. `商品追加`
3. 商品本体を保存
4. `商品詳細管理`
5. `追加`
6. detail で `支払方法 / 回数 / 金額 / 購入後動作` を決める
7. `購入後アクション` を開く
8. `アクション設定` で `バンドルコースへ登録 / webhook / Googleスプレッドシートへ追記` を確認する
9. `会員サイト > バンドルコース`
10. bundle の中身まで確認する
11. representative route と current 実値で、同じ判断を再現できるか確認する

## current の exact route

- `商品追加`
  - `/product/create`
- `商品詳細管理`
  - `/product/{product_id}/detail`
- `商品詳細の追加`
  - `/product/{product_id}/detail/create`
- `アクション設定`
  - `/action/{action_id}/edit`
- `会員サイト > バンドルコース`
  - `/site/{site_id}/bundle/{bundle_id}/edit`

current では、`商品本体 -> 商品詳細管理 -> 購入後アクション -> バンドルコース` を別画面で順に見る前提です。

## exact 手順

### create

1. `商品管理`
2. `商品追加`
3. `商品名 / 重複購入 / 販売上限 / 発行事業者`
4. 保存

current の `商品管理 > 商品一覧 > 行名 > 基本設定` で先に見える代表ラベル:
- `商品名`
- `重複購入`
- `メモ`
- `販売上限`
- `発行事業者`

### detail

1. 保存後の `商品詳細管理`
2. `追加`
3. detail で次を決める
   - `名称`
   - `支払方法`
   - `決済代行会社`
   - `決済連携設定`
   - `支払回数`
   - `金額`
   - `表示名`
   - `表示価格`
4. `購入後の動作設定`
   - `購入後シナリオ`
   - `購入後アクション`

current の detail edit で先に見える代表ラベル:
- `名称`
- `支払方法`
- `決済代行会社`
- `決済連携設定`
- `支払回数`
- `金額`
- `販売上限`
- `オーダーバンプ商品`
- `連携フォームへの表示`
- `商品名表記`
- `価格表記`
- `領収書の品名`
- `領収書の但し書き`
- `登録するシナリオ`
- `実行するアクション`
- `開放するバンドルコース`

current representative を読む時は、少なくとも
- `スキルプラス継続利用`
- `【クレカ】スキルプラス スタンダード ユーザー登録`
の 2 本を見比べる。単発販売と継続課金で、detail の意味がどう変わるかが分かりやすい。

さらに representative route の current 例
- `スキルプラス継続利用`
  - detail list route: `/product/ZqK5nHoXmlNo/detail`
- `【クレカ】スキルプラス スタンダード ユーザー登録`
  - detail row の中で
    - `登録するシナリオ`
    - `実行するアクション`
    - `開放するバンドルコース`
    を一体で読む

### action

`アクション設定` では少なくとも次を確認する。
- `バンドルコースへ登録`
- `webhook`
- `Googleスプレッドシートへ追記`

current の action edit で先に見える代表ラベル:
- `管理用名称`
- `種類`
- `URL`
- `name`
- `value`
- `バンドルコース`
- `商品`
- `ファネル`
- `Googleアカウント`
- `スプレッドシートURL`
- `シート`
- `列`
- `値`

current の `種類` 候補の代表:
- `webhook`
- `バンドルコースへ登録`
- `バンドルコースを停止`
- `継続課金を停止`
- `ファネルの共有ライセンスを発行`
- `ファネルの共有ライセンスを停止`
- `Googleスプレッドシートへ追記`

current representative の読み方
- `【クレカ】スキルプラス スタンダード ユーザー登録`
  - detail 側で `登録するシナリオ / 実行するアクション / 開放するバンドルコース` を 1 セットで見る
- `スキルプラス継続利用`
  - 継続課金 detail の `支払回数 / 価格表記 / 表示価格` の整合を先に見る

### bundle

`会員サイト > バンドルコース` を開き、bundle 名だけで終わらず、中に入る course まで確認する。

current の bundle edit で先に見える代表ラベル:
- `バンドルコース名`
- `追加するコース`
- `バンドルコース`

## 保存前後の最小チェックリスト

保存前
- `商品本体 / detail / 購入後アクション / バンドルコース` の役割を 1 文ずつ言える
- 別商品にする理由を説明できる
- 会員サイト解放先が正しいか確認した

保存後
- detail 一覧で `名称 / 支払方法 / 支払回数 / 価格 / 表示/非表示` を再確認
- `購入後アクション` の中で `webhook / Googleスプレッドシートへ追記 / バンドルコースへ登録` の有無を確認
- `会員サイト > バンドルコース` で、実際に解放される course 名まで確認

## exact に見る順番

1. `商品管理 > 商品一覧 > 行名`
2. `商品詳細管理`
3. detail 編集
4. `購入後アクション`
5. `アクション設定`
6. `会員サイト > バンドルコース`
7. representative route と current 実値を見直し、別商品に寄せる判断が再現できるか確認する

## 検証

最低でも次を確認する。
- 商品本体の意味が正しい
- detail の価格と支払方法が正しい
- `購入後シナリオ` が正しい
- `購入後アクション` が正しい
- bundle 名と解放先 course が正しい
- 連携があるなら `webhook / スプレッドシート追記` の有無も正しい

## NG

- `商品本体` だけ見て終わる
- detail の `購入後アクション` を見ない
- bundle 名だけ見て会員サイト側の中身を確認しない
- 解放先が違うのに同じ商品で吸収しようとする
- 同じ商品と言い切れないのに detail 差し替えで済ませる

## 正誤判断

正しい状態
- `商品本体 / detail / 購入後アクション / bundle` を分けて説明できる
- 別商品にする理由を説明できる
- 購入後の顧客体験まで含めて同一かどうか判断できる
- representative route を開いて current 実値と照らし、同じ判断を再現できる

間違った状態
- 商品本体しか見ていない
- 価格だけ見て判断する
- 会員サイト解放先の違いを軽視する
- 外部連携を後から確認する
