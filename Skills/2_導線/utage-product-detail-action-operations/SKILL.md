# utage-product-detail-action-operations

## Overview

UTAGE の `商品管理` を、商品本体だけでなく `商品詳細管理`、`購入後アクション`、`会員サイトのバンドルコース` まで一気通貫で扱う skill です。

Addness 固有の current 商品例、group の置き方、会員サイトの実体は `Master/addness/utage_structure.md` を正本にする。この skill では、どの案件でも再利用できる exact 手順だけを持つ。

## ツールそのものの役割

UTAGE の `商品管理` は、販売条件、決済条件、購入後の動作、会員サイト解放を束ねる機能です。

## アドネスでの役割

アドネスでは `商品管理` を、販売物そのものというより `顧客体験をどこへつなぐかを決める接続面` として使う。したがって、商品本体より `商品詳細管理 / 購入後アクション / バンドルコース` を重く見る。

## 役割

UTAGE の `商品管理` は、価格や決済の見せ方を持つだけでなく、購入後に誰をどのシナリオへ流し、どの会員サイトを解放し、どの連携を実行するかを束ねる実行面です。

## ゴール

次を迷わずできる状態を作る。
- `商品管理 -> 商品追加`
- `商品詳細管理 -> 追加`
- `購入後アクション`
- `会員サイト -> バンドルコース`
を正しい順で見られる
- 何をもって別商品にするか説明できる
- `商品本体` だけ見て終わらず、`detail -> action -> bundle` まで確認できる

## 必要変数

- 商品名
- 価格
- 支払方法
- 支払回数
- 決済代行会社
- 表示名 / 表示価格
- 購入後シナリオ
- 購入後アクション
- 会員サイト解放先
- 外部連携の有無
  - webhook
  - スプレッドシート追記

## 依頼を受けたら最初に埋める項目

- `商品名`
- `商品管理` で新規か既存か
- `商品詳細管理` で detail を増やすか
- `支払方法 / 支払回数 / 金額`
- `購入後シナリオ`
- `購入後アクション`
- `開放するバンドルコース`
- webhook や Sheets 連携の有無
- exploratory か、本線修正か

## 10秒判断

- `新規商品`
  - 商品名、価格、シナリオ、action、会員サイト解放先のどれかが変わる
- `既存商品内の商品詳細管理`
  - 同じ商品と言い切れる
  - 顧客体験も同じ
  - 決済条件差分の吸収で足りる
- `既存アクション修正`
  - 商品や detail はそのまま
  - downstream の処理だけ変える

この 10 秒判断が言えない時は、`商品追加` や `商品詳細管理 > 追加` を押さない。

## 実装前の最小チェック

- この販売物を別商品に切る理由を説明できる
- `商品本体` と `商品詳細管理` の差分を分けて考えている
- `購入後アクション` と `会員サイト解放先` が確定している
- webhook やスプレッドシート追記の有無を先に切っている
- 既存商品への追加で吸収するか、新規商品にするかを先に判断している

## 入口選択の exact な問い

- 今やりたいのは `販売物そのものを増やす / 分ける` か
  - なら `商品管理`
- 今やりたいのは `同じ商品内で価格や決済条件を分ける` か
  - なら `商品詳細管理`
- 今やりたいのは `購入後に何を起こすか` を変えることか
  - なら `購入後アクション`
- 今やりたいのは `受講できる講義や固定ページを変える` か
  - なら `会員サイト > バンドルコース`

つまり、この skill では `商品本体 / 商品詳細 / 購入後アクション / バンドルコース` のどこが主因かを先に切る。

## 判断フレーム

### 別商品に寄せる

次のどれか 1 つでも変わるなら、原則別商品に寄せる。
- 商品名
- 価格
- 支払方法
- 回数
- 流すシナリオ
- `購入後アクション`
- 会員サイト解放先

### 同一商品の中で吸収する

次の条件をすべて満たす時だけ、同一商品の中で吸収を検討する。
- 同じ商品と言い切れる
- 購入後の顧客体験も同じ
- 差分が表示名や軽微な表現だけ

### 既存 detail を流用してよい条件

- `支払方法`
- `支払回数`
- `金額`
- `登録するシナリオ`
- `実行するアクション`
- `開放するバンドルコース`
が全部同じ

このどれか 1 つでも違うなら、既存 detail の流用ではなく `新規商品` か `商品詳細管理 > 追加` を先に疑う。

## Workflow

1. `商品管理`
2. `商品追加`
3. 商品本体を保存
4. `商品詳細管理`
5. `追加`
6. detail で `支払方法 / 回数 / 金額 / 購入後動作` を決める
7. `購入後アクション` を開く
8. `アクション設定` で `バンドルコースへ登録 / webhook / Googleスプレッドシートへ追記` を確認する
9. `会員サイト > バンドルコース`
10. bundle の中身まで確認する
11. representative route と current 実値で、同じ判断を再現できるか確認する

### save を伴う最小 live test の条件

次をすべて満たす時だけ、最小 live save を許可する。
- `テスト` か `ZZ_TEST` と分かる隔離商品がある

隔離商品がない時は、UTAGE の商品系 live test は `読む / downstream を確認する` までで止める。

## 最小 live change

既存 current 商品を触る時は、最初の 1 回を `1 field だけ変える` に制限する。

1. 対象 `商品管理 > 行名`
2. `商品詳細管理`
3. 対象 row を 1 つ固定
4. 変更対象を 1 つに固定
   - `表示名`
   - `価格表記`
   - `領収書の品名`
   のような downstream 影響が軽い field を優先
5. 保存
6. detail 一覧へ戻る
7. `購入後シナリオ / 購入後アクション / 開放するバンドルコース` が変わっていないことを確認
8. 必要なら元に戻す

最初から `支払方法 / 支払回数 / 実行するアクション / 開放するバンドルコース` を同時に変えない。最初は `軽い display field 1 つ` で change-safe な edit を通す。

## 最小 rollback

exploratory に変更した時は、`商品一覧の行名 / 商品詳細管理 row / 変更した field` を先に固定する。

1. 対象 `商品管理 > 行名`
2. `商品詳細管理`
3. 対象 row
4. 変更した field を 1 つに固定
5. 同じ画面で元の値へ戻す
6. detail 一覧へ戻る
7. `購入後シナリオ / 購入後アクション / 開放するバンドルコース` が元に戻っていることを確認
8. 必要なら `アクション設定` と `会員サイト > バンドルコース` まで再確認

`購入後アクション` や `開放するバンドルコース` を exploratory に変えた時は、detail 画面の表示だけ戻して終わらせない。downstream の `アクション設定` と `bundle` まで元の接続に戻っていることを確認して rollback 完了とする。

exploratory product を新規作成した時の rollback は次で固定する。

1. `商品管理`
2. `商品一覧 > 行名`
3. row action の `削除`
4. row form の `_method=DELETE` で submit
5. 商品一覧で intended row が `0件` になったことを確認

`作成できた` では完了にしない。商品一覧から intended row が消えたところまで見て rollback 完了とする。

## representative pattern

- `単発販売型`
  - `商品本体 -> detail -> 購入後シナリオ -> アクション -> bundle`
    を 1 本で追う
  - 主に `ユーザー登録` や単発購入後解放で使う
- `継続課金型`
  - `支払回数 = 継続課金`
  - `表示価格 / 領収書表記 / 継続停止条件`
    の整合が重要
- `会員サイト直解放型`
  - `開放するバンドルコース` が主役
  - どの course が bundle 配下にあるかまで確認する
- `外部連携併用型`
  - `webhook` や `Googleスプレッドシートへ追記` が混在する
  - 顧客体験と裏側連携を分けて読む

## representative pattern を読む時の問い

- これは `単発販売` か `継続課金` か
- `detail` が顧客体験をどう変えているか
- `購入後アクション` は会員サイト解放か、外部連携か、その両方か
- `bundle 名` だけでなく `bundle 配下の course` まで一致しているか

## ベストな活用場面

- 新しく販売物を作り、購入後の顧客体験まで一気通貫で設計したい時
- 既存商品を見直し、どこを変えたら別商品にすべきか判断したい時
- `商品詳細管理 / 購入後アクション / バンドルコース` の downstream まで事故なく確認したい時

## よくあるエラー

- 商品本体だけ見て、detail と `購入後アクション` を見ない
- `表示価格` と実際の `金額` や `支払回数` がズレる
- `バンドルコース名` だけ確認して、配下 course を見ない
- 解放先が違うのに既存商品へ押し込もうとする

## エラー時の切り分け順

1. それが `単発販売 / 継続課金 / 会員サイト直解放 / 外部連携併用` のどれかを切る
2. `商品本体` と `商品詳細管理` の差分を確認する
3. `購入後アクション` の downstream を確認する
4. `会員サイト > バンドルコース` で bundle 配下 course を確認する
5. 同一商品に吸収できるか、別商品に切るべきかを再判定する

### representative chain を 1 本で読む

代表例として、少なくとも次の 2 本を end-to-end で読む。

- `【クレカ】スキルプラス スタンダード ユーザー登録`
  - `商品管理`
  - `商品詳細管理`
  - `購入後シナリオ`
  - `購入後アクション`
  - `開放するバンドルコース`
  まで 1 本で追う
- `スキルプラス継続利用`
  - 継続課金 detail
  - 表示価格
  - 継続利用の顧客体験が既存商品内で閉じているか
  を追う

この 2 本を分けて読めない時点で、`単発販売` と `継続課金` の判断はまだ甘いと扱う。

### bundle

`会員サイト > バンドルコース` を開き、bundle 名だけで終わらず、中に入る course まで確認する。

current の bundle edit で先に見える代表ラベル:
- `バンドルコース名`
- `追加するコース`
- `バンドルコース`

## 保存前の最小チェック

- `商品本体 / detail / 購入後アクション / バンドルコース` の役割を 1 文ずつ言える
- 別商品にする理由を説明できる
- 会員サイト解放先が正しいか確認した

## 保存後の最小チェック

- detail 一覧で `名称 / 支払方法 / 支払回数 / 価格 / 表示/非表示` を再確認
- `購入後アクション` の中で `webhook / Googleスプレッドシートへ追記 / バンドルコースへ登録` の有無を確認
- `会員サイト > バンドルコース` で、実際に解放される course 名まで確認

## 構築精度だけを見る時のチェック

1. これは `単発販売 / 継続課金 / 会員サイト直解放 / 外部連携併用` のどれかで 1 文になっているか
2. `商品本体 / detail / 購入後アクション / バンドルコース` を別物として説明できるか
3. 別商品にする理由が `設定差分` まで下りているか
4. `支払方法 / 支払回数 / 金額 / 表示価格` の整合が取れているか
5. `登録するシナリオ / 実行するアクション / 開放するバンドルコース` を 1 回で言えるか
6. `会員サイト > バンドルコース` で bundle 配下の course まで確認したか
7. 連携があるなら `webhook / Googleスプレッドシートへ追記` の downstream まで説明できるか

## 新規販売設定の最終確認順

1. `商品管理 > 商品一覧 > 行名`
2. `商品詳細管理`
3. detail の `名称`
4. `支払方法`
5. `支払回数`
6. `金額`
7. `登録するシナリオ`
8. `実行するアクション`
9. `開放するバンドルコース`
10. `アクション設定`
11. `会員サイト > バンドルコース`
12. bundle 配下の course

この順で見れば、`価格は合っているが解放先が違う`、`シナリオは合っているが action が違う` を保存前に潰せる。

## ここで止めて確認する条件

- `商品本体` は同じだが、`購入後アクション` か `開放するバンドルコース` が違う
- `商品詳細管理` に複数 row があり、どの row を current で使っているか説明できない
- `表示名 / 表示価格` と実際の販売条件がずれている
- `バンドルコースへ登録` と `Googleスプレッドシートへ追記` が混在していて、どちらが必須か説明できない
- 会員サイト解放先の bundle 名は分かるが、配下 course を確認していない
- 既存 detail を流用すると、別商品の顧客体験が混ざりそう
- `購入後シナリオ / 購入後アクション / 開放するバンドルコース`
  のうち、どれが主因で別商品に寄せるのか言えない

## `detail` を増やす前の exact チェック

1. 差分が
   - `支払方法`
   - `支払回数`
   - `価格`
   - `表示価格`
   の範囲に本当に収まる
2. `購入後シナリオ / 購入後アクション / 開放するバンドルコース`
   が既存 detail と同じ
3. 顧客体験として同一商品と言い切れる
4. 会員サイト解放先が変わらない

## 最初の複雑パターン

最初に exact に増やす 1 本は、次のどれか 1 つに絞る。

- `既存商品本体 + detail 1本追加`
- `商品詳細管理 1行 + 購入後アクション 1本 + バンドルコース 1本` の downstream 確認

最初から
- 商品本体変更
- detail 追加と action 差し替えの同時実行
- bundle 変更と会員サイト本文変更の同時実行
をやらない。まず `どの差分が主因か` を 1 つに絞る。

## 完成条件

- `商品本体 / detail / 購入後アクション / bundle` を分けて説明できる
- 別商品にする理由を説明できる
- 購入後の顧客体験まで current 実値で追える

## exact に見る順番

1. `商品管理 > 商品一覧 > 行名`
2. `商品詳細管理`
3. detail 編集
4. `購入後アクション`
5. `アクション設定`
6. `会員サイト > バンドルコース`
7. representative route と current 実値を見直し、別商品に寄せる判断が再現できるか確認する

## 検証

最低でも次を確認する。
- 商品本体の意味が正しい
- detail の価格と支払方法が正しい
- `購入後シナリオ` が正しい
- `購入後アクション` が正しい
- bundle 名と解放先 course が正しい
- 連携があるなら `webhook / スプレッドシート追記` の有無も正しい

## 症状から最初に疑う場所

- 価格は正しいのに購入後の体験が違う
  - `商品本体` ではなく `商品詳細管理`
  - `購入後アクション`
  - `開放するバンドルコース`
  の順で疑う
- 同じ商品だと思ったのに downstream が変わる
  - `detail` 追加で吸収すべき案件か
  - 最初から `別商品`
  に切るべき案件かを再判定する
- 会員サイトは開くが講義が違う
  - bundle 名だけでなく bundle 配下の course を疑う
- webhook や Sheets 連携だけズレる
  - `購入後アクション`
  の `種類`
  - `URL`
  - `シート`
  - `列`
  の順で疑う
- 継続課金まわりが不自然
  - `支払回数`
  - `価格表記`
  - `表示価格`
  - `領収書の品名`
  の整合を先に疑う

## NG

- `商品本体` だけ見て終わる
- detail の `購入後アクション` を見ない
- bundle 名だけ見て会員サイト側の中身を確認しない
- 解放先が違うのに同じ商品で吸収しようとする
- 同じ商品と言い切れないのに detail 差し替えで済ませる

## 正誤判断

正しい状態
- `商品本体 / detail / 購入後アクション / bundle` を分けて説明できる
- 別商品にする理由を説明できる
- 購入後の顧客体験まで含めて同一かどうか判断できる
- representative route を開いて current 実値と照らし、同じ判断を再現できる

間違った状態
- 商品本体しか見ていない
- 価格だけ見て判断する
- 会員サイト解放先の違いを軽視する
- 外部連携を後から確認する
