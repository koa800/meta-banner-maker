# UTAGE 商品・detail・action exact メモ

## current でよく使う route

- 商品一覧: `/product`
- 商品追加: `/product/create`
- 商品編集: `/product/{product_id}/edit`
- 商品詳細管理: `/product/{product_id}/detail`
- detail 追加: `/product/{product_id}/detail/create`
- detail 編集: `/product/{product_id}/detail/{detail_id}/edit`
- action 編集: `/action/{action_id}/edit`
- 会員サイト一覧: `/site`
- バンドルコース一覧: `/site/{site_id}/bundle`
- bundle 編集: `/site/{site_id}/bundle/{bundle_id}/edit`

## 保存前後の最小チェックリスト

### 保存前

- `商品本体 / 商品詳細管理 / 購入後アクション / バンドルコース` を別物として説明できる
- 何が変わるから別商品にするのか説明できる
- 会員サイト解放先が正しい bundle まで言える

### 保存後

- detail 一覧に `名称 / 支払方法 / 支払回数 / 価格 / オーダーバンプ / 表示/非表示` が意図通り出ている
- `購入後アクション` の中で `webhook / Googleスプレッドシートへ追記 / バンドルコースへ登録` のどれが使われているか確認する
- bundle 編集画面で、解放したい course 名まで確認する
- 可能なら購入後の到達先シナリオ名まで 1 本で確認する

## 完成条件

次を 1 本で説明できた時だけ完成扱いにする。
- 何を変えたから `別商品` にしたのか
- `商品本体`
- `detail`
- `購入後アクション`
- `バンドルコース`
の役割差が説明できる
- 購入後に
  - どのシナリオへ入るか
  - どのアクションが走るか
  - どのバンドルコースが解放されるか
 まで言える
- `会員サイト解放先が違うなら別商品` の原則に反していない

## current で迷いやすい差分

- `商品本体` を作っただけでは販売設定は終わっていない
- 実務上の本体は `detail + 購入後アクション`
- `表示価格` と `実際の金額` は役割が違う
- `購入後シナリオ` を変えるだけで済むのか、別商品に寄せるべきかを混同しやすい
- `バンドルコース` 名だけで判断すると事故る
  - 必ず bundle の中の course まで見る

## ここで止めて確認する条件

- `商品本体` だけで吸収できるか、別商品に切るべきか曖昧
- `detail` 追加で足りるか、新規商品が必要か曖昧
- `購入後アクション` の変更が既存購入者へ影響しそう
- 解放先 `バンドルコース` が複数候補あって迷う
- `商品名 / 価格 / シナリオ / 設定構成 / 会員サイト解放先` のどこが変わるか整理できていない
- `購入後シナリオ` を変えるだけで済むのか、別商品に寄せるべきか曖昧
- `detail` 差分で吸収してよい条件か、別商品へ切る条件かを 1 文で言えない

## `detail` 追加前の最小チェック

1. 差分が `価格 / 支払方法 / 支払回数 / 表示価格` の範囲に収まる
2. `購入後シナリオ / 購入後アクション / 開放するバンドルコース`
   は既存 detail と同じ
3. 会員サイト解放先が変わらない
4. 顧客体験として同一商品と言い切れる

## Addness 側で見るべき補足

- どの group の商品か
- current / legacy の読み分け
- 会員サイト解放先が変わるなら原則別商品
- `detail + action` が本体

## 保存前の最小チェック

- page type や商品の役割を 1 文で言える
- 主CTA、遷移先、action の主従が切れている
- 実 click で確認する前提になっている

## 保存後の最小チェック

- preview または公開ページで実挙動を確認した
- 商品、detail、action、bundle の接続を確認した
- CTA の最終遷移先が意図どおり
