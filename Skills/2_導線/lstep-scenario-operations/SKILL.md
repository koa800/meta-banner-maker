# lstep-scenario-operations

## Overview

Lステップの `シナリオ配信` を、一覧の見方だけでなく `新しいシナリオ -> step 追加 -> テンプレート / アクション設定 -> 保存 -> テスト -> 後片付け` まで exact に扱う skill です。

Addness 固有の account、命名、current / legacy、representative は `Master/knowledge/lstep_structure.md` を正本にする。この skill では、どの案件でも再利用できる exact 手順だけを持つ。

## 役割

`シナリオ配信` は、登録や条件達成を起点に、順番とタイミングを持って message を出し分ける機能です。単発送信ではなく、`いつ / 誰に / 何を / その後どうするか` を step 単位で制御します。

## ゴール

次を迷わずできる状態を作る。
- `シナリオ配信` の正しい入口から入る
- `新しいシナリオ` を作る
- step ごとに `配信タイミング / テンプレート / アクション` を設定する
- step の役割を `認識変換 / 行動誘導 / 状態更新` のどれかで説明できる
- テスト用なら削除まで後片付けする

## 必要変数

- シナリオ名
- シナリオフォルダ
- 誰に始めるか
- 何を認識させたいか
- 何を行動させたいか
- 何通に分けるか
- step ごとの配信タイミング
- 使うテンプレート
- step ごとのアクション有無
- 最終コンテンツ配信後の処理
- テスト送信先
- テスト後に残すか削除するか

## 使い分け

- `シナリオ配信`
  - 登録後や条件達成後に、順番と時間差を持って配信したい
- `一斉配信`
  - 今この瞬間にまとめて送りたい
- `テンプレート`
  - 単体 message の部品として作りたい

`シナリオ配信` を選ぶのは、
- step ごとに役割を分けたい
- 直後 / 1日後 / 3日後 のように時間差が必要
- 途中で `タグ追加 / シナリオ停止 / テンプレート送信` などを混ぜたい
時です。

## Workflow

1. `シナリオ配信`
2. 必要ならフォルダを切り替える
3. `新しいシナリオ`
4. `シナリオ名`
5. 必要なら `シナリオフォルダ`
6. `状態`
7. `対象の絞り込み`
8. `コンテンツ`
9. step ごとに
   - `配信タイミング`
   - `テンプレート`
   - `アクション`
   を入れる
10. `最終コンテンツ配信後の処理`
11. 保存
12. `テスト`
13. 実機確認
14. テスト用なら `削除`

## exact 手順

### 一覧と create 入口

- 一覧 URL
  - `/line/content/group`
- current API
  - `GET /api/content-group-groups`
  - `GET /api/content-groups?group={group_id}&page=1`
- 一覧で見るべきラベル
  - `新しいフォルダ`
  - `新しいシナリオ`
  - `並び替え`
  - `検索`

### 新規シナリオ

1. `シナリオ配信`
2. 必要なら対象フォルダを開く
3. `新しいシナリオ`
4. create 画面で次を確認
   - `シナリオ名`
   - `シナリオフォルダ`
   - `状態`
   - `対象の絞り込み`
   - `コンテンツ`
   - `最終コンテンツ配信後の処理`

### 既存シナリオ edit

- current edit URL pattern
  - `/line/content/{content_group_id}?group={group_id}`
- row controls
  - `編集`
  - `挿入`
  - `コピー`
  - `プレビュー`
  - `別窓`
  - `テスト`
  - `削除`

### step 編集 modal

step の主戦場は `配信メッセージ日時編集` modal です。current で見えている主要ラベルは次です。

- `配信タイミング`
  - `購読開始直後`
  - `経過時間指定`
- `step`
- `time`
- `order`
- `開始当日`
- `pause`
  - `シナリオを一時停止しない`
  - `シナリオを一時停止する`
- `送信先を絞り込む`
- `tag_limit`
- `tag_id`
- `テンプレート詳細`
- `テンプレート名`
- `テンプレート種別`
- `テンプレートフォルダ`
- `テンプレートを編集する`
- `テンプレートプレビュー`
- `アクションを設定しない`
- `アクションを設定する`
- `アクションを設定`
- `設定解除`

### step の基本設計

step ごとに先に役割を切る。

- 認識変換 step
  - 何が起きているか
  - 何を理解させるか
- 行動誘導 step
  - 何を押させるか
  - どこへ送るか
- 状態更新 step
  - タグ
  - アクション
  - 次シナリオ

悪い作り方は、1 step にこの 3 役割を詰め込みすぎることです。

### テンプレート接続

step で `テンプレート` を使う時は、次を先に確認する。

1. その step の役割に合う message 種か
2. `テンプレート名`
3. `テンプレート種別`
4. `テンプレートフォルダ`
5. 必要なら `テンプレートプレビュー`

テンプレートの exact 作業は、必要に応じて別 skill を使う。
- `lstep-standard-message-operations`
- `lstep-carousel-message-operations`
- `lstep-flex-message-operations`
- `lstep-template-pack-operations`

### アクション接続

step で `アクションを設定する` を使う時は、
- reusable にしたい複合処理なら `アクション管理`
- その step に閉じた軽い処理なら inline
で切る。

現在の exact 判断は、
- 使い回すなら action
- その場限りなら inline
です。

## representative の読み方

代表例を読む時は、次の順番で分解する。

1. このシナリオは、どの状態の人に始まるか
2. シナリオ全体で、何を何に変えたいか
3. 各 step は
   - 認識変換
   - 行動誘導
   - 状態更新
   のどれか
4. どの step が main CTA か
5. 最終コンテンツ配信後の処理は何か

## 検証

最低でも次を確認する。
- 一覧で対象シナリオが見える
- step 数と順番が意図どおり
- テンプレートが正しい
- action が正しい
- `テスト` で実機確認できる
- 最終的に意図した行動へつながる

## 後片付け

テスト用シナリオは残さない。

1. 一覧の対象 row 右端
2. `削除`
3. confirm
4. 再読込
5. row が消えたことを確認

`コピー` や `下書き` を残して終わらない。

## NG

- `一斉配信` でよいものを `シナリオ配信` にする
- 1 step に認識変換と行動誘導と状態更新を全部詰め込む
- 役割が曖昧な step を増やす
- current / legacy を見ずに old シナリオを複製する
- テストせずに本番へ回す
- テスト用シナリオを残す

## 正誤判断

正しい状態
- 正しい入口から入っている
- シナリオ全体の目的が 1 文で言える
- 各 step の役割が明確
- テンプレートと action の接続理由を説明できる
- 実機でテストできる
- テスト用は削除済み

間違った状態
- step がただ増えているだけで、役割分担がない
- `テンプレート` と `アクション` を何となく付けている
- current / legacy の区別が曖昧
- 作った後のテストと削除を飛ばしている
