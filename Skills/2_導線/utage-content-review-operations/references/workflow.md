# UTAGE content review workflow

## 使うタイミング

- 既存の LP や thanks を benchmark として分解したい時
- 新規ページを作る前に current quality を読みたい時
- 作ったページが意図した顧客体験になっているか確認したい時

## 30秒レビューの順番

### 構築精度だけを見る時のチェック

内容の良し悪しと、構築の正しさは分けて見る。まずは次の点が合っているかを先に切る。

1. type の選択が正しいか
2. main CTA が 1 つに切れているか
3. hyperlink や action の実体が合っているか
4. 入力値や遷移先が実値で正しいか
5. テスト方法が決まっているか
6. 実 click / 実挙動で確認したか
7. 残すべきものと削除すべきものが分かっているか


1. page type を切る
2. 誰が来るページかを言う
3. promise を 1 文で言う
4. 変えたい認識を 1 文で言う
5. 主CTA を 1 つに切る
6. `form / action / 商品 / 遷移先` を確認する
7. 実 click で最終挙動を見る

## page type ごとの重点

### LP

- first view の promise
- 誰向けか
- 主CTA
- `form`
- `シナリオ`
- `アクション`
- 次ページ

### thanks

- 次に何をしてほしいか
- visible CTA か自動遷移か
- LINE や short.io への橋渡しが滑らかか

### content page

- 何を理解させたいか
- 動画やテキストの役割
- CTA の位置と強さ

### ユーザー登録

- `商品詳細管理`
- `購入後アクション`
- `開放するバンドルコース`
を強く確認する

## code 領域の見方

- `カスタムCSS`
  - 見た目補正
- `カスタムJS`
  - 計測、補助挙動、自動遷移
- `first_view_css`
  - LP の FV を強く作る時の主戦場
  - thanks や content page では、まず主因にしない

## 完成条件

次を全部言えた時だけ review 完了とする。
- この page は何の type か
- 誰が来るか
- 何の promise を出しているか
- 何の認識をどう変えたいか
- 次に何を行動させたいか
- `form / action / 商品 / 遷移先` がその目的に合っているか
- click 後の最終挙動が意図どおりか

## current で見誤りやすい点

- 見た目だけで LP quality を判断しない
  - `form / scenario / action / 遷移先` が本体
- thanks は visible CTA と自動遷移を混同しやすい
- content page は `動画がある = 教育できている` ではない
  - 何を理解させて、どこへ送るかが必要
- `first_view_css` が存在するだけで主因と決めない
  - LP の FV 以外では、まず visible 要素と `js_body_top` を先に疑う
- 同じ page でも `登録経路` が違えば分析上は別入口になる

## 最低限の output

- page type
- 目的
- promise
- 変えたい認識
- 起こしたい行動
- 良い点
- 弱い点
- 再現すべき点
- 変えるならどこか

## ここで止めて確認する条件

- `promise` は言えるが、`次に何を行動させたいか` が言えない
- 見た目は良いが `form / action / 商品 / 遷移先` の接続が確認できていない
- `登録経路` の違いで分析結果が変わるのに、ページだけ見て判断しようとしている

## 保存前の最小チェック

- page type や商品の役割を 1 文で言える
- 主CTA、遷移先、action の主従が切れている
- 実 click で確認する前提になっている

## 保存後の最小チェック

- preview または公開ページで実挙動を確認した
- 商品、detail、action、bundle の接続を確認した
- CTA の最終遷移先が意図どおり
