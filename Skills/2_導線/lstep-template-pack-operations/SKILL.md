# lstep-template-pack-operations

## Overview

Lステップの `テンプレートパック` を、`作成 -> step追加 -> テスト送信 -> 削除` まで exact に再現する skill です。

`テンプレートパック` は、単なる message の寄せ集めではありません。`認識を変える step` と `行動させる step` を順番つきで束ねる container として使うと強いです。

Addness 固有の account 選定、命名規則、current representative、テスト送信先は `Master/knowledge/lstep_structure.md` を正本にする。この skill は、会社をまたいで再利用できる exact 手順だけを持つ。

## 役割

`テンプレートパック` は、1通で全部を説明しきらず、複数 step に役割を分けて受信体験を滑らかにする message container です。

## ゴール

次を迷わずできる状態を作る。
- `テンプレート -> 新しいパック` の正しい入口を選ぶ
- pack 名を作る
- `本文` step を追加する
- 必要なら既存 `テンプレート` を pack に差し込む
- `テスト送信`
- テスト用なら `削除`

## 必要変数

- pack 名
- 何の認識を変えたいか
- 最終的に何を行動させたいか
- step 数
- 各 step の役割
- step で使う message 種
- テスト送信先
- テスト後に残すか削除するか

## 使い分け

先に message の粒度を切る。
- `標準メッセージ`
  - 1通で十分
- `カルーセルメッセージ(新方式)`
  - card を並べて選ばせたい
- `フレックスメッセージ`
  - 1画面で重要度差を見せたい
- `テンプレートパック`
  - 複数 message を順番つきで束ねたい
  - `認識変換 -> CTA`
  - `理由づけ -> 回答`
  - `告知 -> visual 訴求 -> 返信`
  のように step を分けたい

## Workflow

1. `テンプレート`
2. `新しいパック`
3. `テンプレート名`
4. `フォルダ`
5. `保存`
6. pack 画面で `メッセージ新規追加` または `テンプレートから追加`
7. step を必要数だけ作る
8. `テスト送信`
9. 実機で順番と action を確認
10. テスト用なら削除

## exact 手順

### create

1. `テンプレート`
2. `新しいパック`
3. `テンプレート名`
4. `フォルダ`
5. `保存`

代表 route
- 新規: `/line/eggpack/edit/new?group=0`
- 表示: `/line/eggpack/show/{pack_id}?group=0`

### step を足す

pack 画面の主入口は次です。
- `メッセージ新規追加`
- `テンプレートから追加`

`メッセージ新規追加` で見える current 種類。
- `テキスト`
- `スタンプ`
- `画像`
- `質問`
- `ボタン・カルーセル`
- `位置情報`
- `紹介`
- `音声`
- `動画`

### 最小の本文 step

最初に exact 化しておくべきなのは `テキスト` です。

1. `メッセージ新規追加`
2. `テキスト`
3. 本文を入れる
4. `保存`

重要
- visible `textarea` ではなく `.ProseMirror` が主入口になることがある
- `見えている本文` を基準に保存確認する

### 既存テンプレート差し込み

1. `テンプレートから追加`
2. modal `テンプレート選択`
3. 対象テンプレートを選ぶ
4. `追加`

current modal で見える主要ラベル。
- `テンプレート選択`
- `閉じる`
- `コピーして編集`
- `追加`

## pack を high-quality にする観点

`テンプレートパック` は、step ごとの役割が重複しない時に強いです。

見る順番
1. step 1 は何の認識を変えるか
2. step 2 は何を行動させるか
3. step 同士の役割が重複していないか
4. 受信順が自然か

良い例
- step 1 で urgency や事情説明
- step 2 で visual CTA

悪い例
- step 1 で全部説明
- step 2 でも同じ説明を繰り返す
- どれを押せばよいかがぼやける

## テスト送信

1. 一覧行右端 `...`
2. `テスト送信`
3. `テスト送信先選択`
4. 送信先行の左端 checkbox label を押して選択
5. `テスト`
6. 実機確認

Addness 固有のテスト送信先と重複禁止ルールは `Master/knowledge/lstep_structure.md` を正本にする。

## 検証

最低でも次を確認する。
- step の順番が自然
- 各 step の役割が重複していない
- visual CTA の行き先が正しい
- 実機で押して意図どおり動く
- 必要なら最終遷移先まで合っている

## 後片付け

テスト用なら、
1. 一覧行右端 `...`
2. `削除`
3. confirm dialog `削除する`
4. 一覧再読込

削除したつもりで終わらない。対象行が見えないことを確認する。

## NG

- `テンプレートパック` を message の倉庫として使う
- 各 step の役割を決めずに足す
- step 1 と step 2 で同じ説明を繰り返す
- `テスト送信` をせずに本番へ回す
- テスト用 item を残す
- Addness 固有ルールを skill 内に焼き込む

## 正誤判断

正しい状態
- 正しい入口から pack を作れている
- step ごとに役割が分かれている
- `本文` と `テンプレート` の役割が重複していない
- 実機で順番どおり自然に受け取れる
- テスト用は削除済み

間違った状態
- 1 step で全部を説明しようとする
- `本文` と `テンプレート` が同じことを言っている
- CTA を複数 step で散らしている
- current representative を見ずに雑に増やす
