# lstep-answer-form-operations

## Overview

Lステップの `回答フォーム` を、作成して終わりではなく `質問追加 -> 保存 -> テスト送信 -> 削除` まで exact に再現する skill です。

Addness 固有の命名、テスト送信先、current 例は `Master/knowledge/lstep_structure.md` を正本にする。この skill は、会社をまたいで再利用できる作成手順だけを持つ。

live で触る前に、必ず `python3 System/scripts/funnel_tool_live_readiness.py` を実行し、`lstep.auth_alive=true` に加えて `lstep.account_name` が intended な公式LINE account か確認する。`auth_alive=true` でも別 account 文脈を見ていることがある。

## ツールそのものの役割

`回答フォーム` は、LINE 上で structured data を回収し、後続の配信、営業判断、状態管理に使う入力を作る機能です。

## アドネスでの役割

アドネスでは `回答フォーム` を、単なるアンケートではなく `次の配信や営業判断に使う状態入力` として扱う。したがって、質問は情報収集のためではなく、`この回答を取ったあと何を変えるか` を先に決めて作る。

## 役割

`回答フォーム` は、ユーザーから structured data を回収し、次の配信や予約や営業判断の入力を作るための form type です。

## ゴール

次を迷わずできる状態を作る。
- `回答フォーム` の正しい入口を選ぶ
- フォーム名とフォームタイトルを作る
- 質問を追加する
- 保存する
- 必要なら `テスト送信`
- テスト用なら削除する
- 一覧の `スプレッドシート連携 / 回答状態 / 公開状態` を読み、運用上の意味まで判断できる

## 必要変数

- フォーム名
- フォームタイトル
- 回収したい項目
- 必須項目かどうか
- 回答後に何を判断したいか
- テスト送信するか
- テスト後に残すか削除するか

## 依頼を受けたら最初に埋める項目

- `フォーム名(管理用)`
- `タイトル`
- 何を回収したいか
- 回答後に何を変えるか
  - `タグ / 配信 / 営業判断 / 予約`
- 質問種別
  - `記述式(テキストボックス) / ラジオボタン / プルダウン / ファイル添付`
- `サンクスページURL` や `回答後アクション` の有無
- exploratory か、本線用か

## 実装前の最小チェック

- `lstep.account_name` が intended な公式LINE account と一致している
- 何の判断に使う `回答フォーム` か 1 文で言える
- 回答後に次の配信、営業、タグ判断のどれへつなぐか決まっている
- `記述式(テキストボックス)`、`ラジオボタン`、`プルダウン` のどれが最短か決めている
- テスト送信するか、テスト後に削除するか決めている

## 入口選択の exact な問い

- 今やりたいのは `情報を回収する` ことか
  - なら `回答フォーム`
- 今やりたいのは `情報を見せる / 押させる` ことか
  - それはこの skill ではなく `テンプレート`
- 今やりたいのは `回答後に状態をまとめて変える` ことか
  - それはこの skill ではなく `アクション管理`

つまり、この skill の入口は `回収が主目的か` で決まる。

## 10秒判断

- `回答フォーム`
  - 情報を回収したい
  - 回答後の判断材料を作りたい
- `テンプレート`
  - 情報を見せることが主目的
- `アクション管理`
  - 回答後の state 変更を reusable にまとめたい
- `記述式(テキストボックス)`
  - 自由記述が本当に必要
- `ラジオボタン / プルダウン`
  - 選択肢を早く回収したい

この 10 秒判断が言えない時は、`新しい回答フォーム` を押さない。

## Workflow

1. `回答フォーム`
2. `新しい回答フォーム`
3. `フォーム名`
4. `フォームタイトル`
5. 質問を追加
6. `保存`
7. 一覧に戻る
8. 必要なら `... -> テスト送信`
9. 実機確認
10. テスト用なら `... -> 削除`

## exact 手順

### create

1. `回答フォーム`
2. `新しい回答フォーム`
3. `フォーム名(管理用)`
4. `タイトル`
5. 質問追加
6. `保存`

### 質問追加

最初に exact 化しておくべきなのは `記述式(テキストボックス)` です。質問 block を追加し、質問名を入れて保存するところまでをまず固定する。

current で主要な要素。
- `小見出し`
- `中見出し`
- `記述式(テキストボックス)`
- `段落(テキストエリア)`
- `チェックボックス`
- `ラジオボタン`
- `プルダウン`
- `ファイル添付`
- `都道府県`
- `日付`
- `メール登録`

### 一覧で先に読む項目

- `フォーム名`
- `スプレッドシート連携`
- `回答状態`
- `登録日`
- `公開状態`

row に `呼び出しタグ` や `外部用` のラベルが見える時は、そのまま用途シグナルとして読む。

### フォーム設計で先に決めること

1. 何を回収したいか
2. その回答で何を判断したいか
3. 回答後に何をさせたいか
4. `テキストボックス / ラジオ / プルダウン / ファイル` のどれが最短か

### current の option 例

- `進むボタンテキスト`
- `送信ボタンテキスト`
- `送信確認ダイアログ`
- `回答期限`
- `先着数制限`
- `1人が回答できる回数`
- `サンクスページURL`
- `回答後アクション`
- `回答復元`
- `Googleスプレッドシート連携`

## representative pattern

### 状態入力型

- 目的
  - 回答内容を次の配信や営業判断に使う
- 主要要素
  - `記述式(テキストボックス)`
  - `ラジオボタン`
  - `プルダウン`
- 向いている場面
  - 個別相談前の事前確認
  - 希望コースや現在地の把握

### 申込補助型

- 目的
  - 申込前に必要最低限の確認を回収する
- 主要要素
  - `メール登録`
  - `都道府県`
  - `ラジオボタン`
- 向いている場面
  - 合宿やイベントの参加確認
  - 面談前の必須確認

### 証跡回収型

- 目的
  - ファイルや長文回答を回収して後続判断へ使う
- 主要要素
  - `ファイル添付`
  - `段落(テキストエリア)`
- 向いている場面
  - 本人確認
  - 課題提出
  - 申請系の補助

## 最小 live test

最初の 1 本は `質問 1 つ` だけで閉じる。

1. `新しい回答フォーム`
2. `フォーム名(管理用)`
3. `タイトル`
4. `記述式(テキストボックス)` を 1 問だけ追加
5. `保存`
6. 一覧で row を確認
7. 必要なら `... -> テスト送信`
8. 実機確認
9. テスト用なら `... -> 削除`

## 最小 downstream smoke

回答フォームは save して終わりにしない。

1. intended な `フォーム名`
2. intended な `タイトル`
3. intended な質問種別
4. intended な `サンクスページURL` または `回答後アクション`
5. 回答後に何が変わるかを 1 文で言えるか

回答のあとに何が起きるか説明できない時は、質問を増やさない。

複数質問や `スプレッドシート連携` は、その 1 本で save / open / delete が通ってから足す。

## 最小 live change

既存 current を壊さずに exact 性を上げる時は、最初の 1 回を `1フォーム 1変更` で閉じる。

最初に触る候補
- `フォーム名(管理用)`
- `タイトル`
- 既存質問 1 つの文言
- `送信ボタンテキスト`

順番
1. row を 1 つに固定する
2. 変更点を 1 つだけ決める
3. save
4. 一覧に戻って row を再確認する
5. 必要なら `... -> テスト送信`
6. 実機で intended どおりに見えるか確認する
7. exploratory なら元に戻すか削除する

最初からやらないこと
- 質問追加と `スプレッドシート連携` を同時に変える
- `サンクスページURL` と `回答後アクション` を同時に変える
- 2問以上の種別変更を一気にやる

## 複雑パターンを足す順

複雑化は、次の順で 1 つずつ足す。

1. `記述式(テキストボックス)` 1 問
2. `ラジオボタン` または `プルダウン`
3. `送信ボタンテキスト`
4. `サンクスページURL` または `回答後アクション`
5. `Googleスプレッドシート連携`

`質問を増やす` と `回答後処理を変える` を同時にやると、どこで downstream がズレたか切れなくなる。まず `質問`, 次に `回答後処理`, 最後に `外部連携` の順で足す。

## 最初の複雑パターン

最初に live で詰める複雑パターンは、`2問だけ` で閉じる。

1. `記述式(テキストボックス)` 1問
2. `ラジオボタン` 1問
3. `... -> テスト送信`
4. `甲原 海人` に送る
5. 回答後に intended な `サンクスページURL` または `回答後アクション` を確認
6. exploratory なら `... -> 削除`

最初から
- `ファイル添付`
- `Googleスプレッドシート連携`
- `回答期限`
- `先着数制限`
を同時に足さない。まず `2問だけ` で downstream が intended かを閉じる。

## 最小 rollback

exploratory に form を作った時は、一覧から確実に消えるところまで戻す。

1. 一覧へ戻る
2. `フォーム名`
3. row 右端 `...`
4. `削除`
5. confirm dialog `削除する`
6. 一覧で row が消えたことを確認

既存 current を 1 field だけ変えた時は、まず同じ field を元の値へ戻す。新規 form を作った時だけ row ごと削除する。

## 変更後の最小 smoke

変更後は `届くか` だけで閉じない。

1. intended な `フォーム名`
2. intended な `タイトル`
3. intended な質問種別
4. `送信ボタンテキスト`
5. `サンクスページURL` または `回答後アクション`
6. downstream の `タグ / 配信 / 営業判断 / 予約導線`

この 6 つを 1 文で説明できない時は、質問を追加しない。

## representative pattern を読む時の問い

- この `回答フォーム` は、何の判断材料を回収するために存在するか
- 回答後に変わるのは `タグ / 配信 / 営業判断 / 予約導線` のどれか
- いまの質問種別は、回答者の負荷を最小にしているか
- `記述式(テキストボックス)` にしている理由が本当にあるか
- `ラジオボタン / プルダウン` にした方が判断しやすくならないか
- `回答フォーム` 自体が主役なのか、次の action へ渡すための補助なのか
- current の良い例と比べて、質問数が増えすぎていないか
- 回答後に `サンクスページURL` や `回答後アクション` まで一貫しているか

## 複雑パターンで先に切ること

- `何を回収したいか`
- `その回答で何を判断したいか`
- `回答後に何をさせたいか`
- `1フォームで足りるか`
- `サンクスページURL` が主役か、`回答後アクション` が主役か
- `スプレッドシート連携` が必要か
- `自由記述` が本当に必要か
- `既存フォーム流用` と `新規作成` のどちらが自然か

この4つが切れていない時は、質問追加の前に止まる。

## 質問種別の主従判断

- `ラジオボタン / プルダウン`
  - state を速く切りたい時の第一候補
- `記述式(テキストボックス) / 段落(テキストエリア)`
  - 選択肢では落ちる情報だけを取りたい時に限定
- `ファイル添付`
  - 証跡そのものが必要な時だけ
- `メール登録`
  - 連絡先が未取得で、回答フォーム内で回収する必要がある時だけ

質問種別は `取りたい情報の豊かさ` ではなく、`次の判断に必要な最小入力` を優先する。

### テスト送信

行右端 `... -> テスト送信` を使う。送信先や重複送信ルールは `Master/knowledge/lstep_structure.md` を正本にする。

### 削除

行右端 `... -> 削除 -> 削除する` で消す。消えたことを一覧で確認する。

## 検証

最低でも次を確認する。
- フォーム名が正しい
- フォームタイトルが正しい
- 質問が表示される
- 必須 / 任意の意図がずれていない
- 回答者目線で違和感がない
- テスト送信から開ける

## 保存前の最小チェック

- 目的、対象、次行動を 1 文で言える
- 入力値や action の実体が確定している
- current の row やフォルダを取り違えていない
- テスト後に残すか削除するか決めている

## 保存後の最小チェック

- 一覧に想定どおり表示される
- 必要なら `テスト送信` や preview で実機確認した
- action や遷移先が意図どおり動く
- テスト用なら削除して row が消えたことを確認した

## テスト送信後の確認順

1. `甲原 海人` に届いているか
2. `フォームタイトル`
3. 質問順
4. 質問種別
5. 必須 / 任意
6. `送信ボタンテキスト`
7. 送信後の `サンクスページURL` または `回答後アクション`

`届いた` だけで完了にしない。`回答者視点で迷わず入力できるか` と `送信後の next action が intended か` を分けて確認する。

## 構築精度だけを見る時のチェック

1. この `回答フォーム` が `状態入力 / 申込補助 / 証跡回収` のどれかで 1 文になっているか
2. `フォーム名` と `フォームタイトル` だけで用途が読めるか
3. 質問種別が `記述式(テキストボックス) / ラジオボタン / プルダウン` の最短解になっているか
4. 回答後に `タグ / 配信 / 営業判断 / 予約導線` の何が変わるか説明できるか
5. `サンクスページURL` や `回答後アクション` があるなら downstream まで言えるか
6. `テスト送信` か preview で実機確認したか
7. テスト用なら row が消えるまで削除確認したか
8. `スプレッドシート連携` があるなら、回収項目名と downstream の列意味が一致しているか
9. `サンクスページURL` と `回答後アクション` を同時に使う時、どちらが主役か説明できるか

## ここで止めて確認する条件

- 何を変える設定か 1 文で言えない
- current / legacy の切り分けに迷う
- 新しい外部連携や本番影響の大きい分岐を足す
- 既存 high-quality な実例と大きく異なる構成にしたい
- `自由記述` を増やしたくなっているが、選択式で代替できるか判断できない
- `サンクスページURL` と `回答後アクション` の役割が混ざっている
- 回収後に downstream で何を変えるか未定のまま質問だけ増やそうとしている

## 完成条件

- 正しい入口から作成または編集できている
- 保存後に一覧と実挙動で確認できている
- テスト送信が必要な場合は `甲原 海人` で確認している
- テスト用の残骸が残っていない
- 回答後の `見える体験` と `裏で変わる state` を分けて説明できる

## NG

- 回収したい情報が曖昧なまま作る
- 質問文が短すぎて回答者が迷う
- 一覧で開けるだけで完成扱いにする
- テスト送信せずに本番へ回す
- テスト用フォームを残す
- Addness 固有ルールを skill に焼き込む

## 正誤判断

正しい状態
- 何を回収したいフォームか一読で分かる
- 質問の順番が自然
- 回答者が迷わない
- 実機で開ける
- テスト用は削除済み
- 一覧の `公開状態 / スプレッドシート連携` も意図どおり読めている

間違った状態
- フォーム名と用途がずれる
- 質問だけ作って送信確認していない
- 質問文が運用側のメモになっている
- current 一覧で消えているか確認していない
- 取得したいデータに対して質問種別が遠回り
