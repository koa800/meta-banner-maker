# Lステップ テンプレートレビュー exact メモ

## 使うタイミング

- 既存テンプレートを benchmark にしたい時
- 新規作成前に良い例を分解したい時
- 作ったテンプレートをレビューしたい時

## 30秒レビューの順番

1. `テンプレート種`
2. `誰に送るか`
3. `何の認識を変えるか`
4. `何を行動させるか`
5. `画像 / テキスト / ボタン / action` の役割
6. `前後の導線との接続`

### 構築精度だけを見る時のチェック

品質の好き嫌いと、構築の正しさは分けて見る。まずは次の 7 点が正しく組めているかを先に切る。

1. 形式選択が正しいか
   - `標準メッセージ / カルーセルメッセージ(新方式) / フレックスメッセージ / テンプレートパック` の選び方が目的に合っているか
2. main CTA が 1 つに切れているか
   - 複数あるなら主従を説明できるか
3. action の実体が合っているか
   - `URLを開く / 回答フォームを開く / テキストを送信 / タグを追加する / シナリオを動かす` のどれかを言えるか
4. action の入力値が合っているか
   - URL、フォーム、タグ、送信文言などが実値で入っているか
5. テスト方法まで決まっているか
   - `テスト送信` で何を見るかが決まっているか
6. 削除条件まで決まっているか
   - テスト用なら、残すか消すかを先に決めているか
7. 実機で最終挙動を確認したか
   - 見た目だけでなく、押した後の挙動まで確認したか

## `テンプレート種` の current 分け方

- `標準メッセージ`
- `カルーセルメッセージ(新方式)`
- `フレックスメッセージ`
- `テンプレートパック`

まず種別を固定しないと、良し悪しの基準がぶれる。

## current で最初に見る UI ラベル

- `テンプレート`
- `新しいテンプレート`
- `標準メッセージ`
- `カルーセルメッセージ(新方式)`
- `フレックスメッセージ`
- `新しいパック`

ここを actual UI の言葉で切ってから読まないと、`単体で作るべきか / パックにすべきか` がぶれやすい。

## 代表的な読み方

### `標準メッセージ`

- 1 通 1 目的になっているか
- text だけで意味が完結しているか
- URL を置くなら actual hyperlink が意図どおりか

### `カルーセルメッセージ(新方式)`

- panel ごとの役割が分かれているか
- main CTA panel が埋もれていないか
- panel 間で情報量が散りすぎていないか

### `フレックスメッセージ`

- 画像が `見せたい順番` で並んでいるか
- text が画像の補足として機能しているか
- button action が 1 meaning で読めるか
- 長尺でも `何を変えたいか` が途中でぶれていないか

current の exact smoke で最低限見る UI ラベル
- `＋ブロックを追加`
- `画像`
- `タイトル`
- `テキスト`
- `ボタン`
- `メッセージを保存`
- 一覧行右端 `...`
- `テスト送信`
- `削除`

### 単発 CTA 型

- 何のオファーか
- urgency は十分か
- CTA が 1 つに絞られているか

### 重要告知型

- 何が変わるか
- なぜ知る必要があるか
- CTA が不要なら無理に置いていないか

### pack 型

- step 1 は認識変換か
- step 2 は行動か
- 役割が重複していないか

### `テンプレートパック`

- step ごとに役割が重なっていないか
- `認識変換 -> 行動` の順が崩れていないか
- pack にする理由があるか
  - 1 通で足りるなら pack にしない

## 最低限見るべき UI

- `テンプレート`
- `プレビュー`
- `別窓`
- 行右端 `...`
- `テスト送信`

## デザインで最低限見ること

- 画像の順番
- text の量
- main CTA の強さ
- 余白と読み順
- `押したい順` と `見せたい順` が一致しているか

Addness では quality が 60 点の current 例も含む前提なので、
`既存にある = 再現すべき`
とは置かない。

## テスト送信時の確認

- 実際の見た目
- 押した時の挙動
- main CTA の明確さ
- 画像と text の順番
- action の意味が一読で分かるか
- 1 通の中で `何をさせたいか` が増えすぎていないか

## `フレックスメッセージ` の live smoke で固定している最小構成

- `画像1 / タイトル1 / テキスト1 / ボタン1`
- `画像`
  - 既存 `登録メディア` または upload から入れる
- `タイトル`
  - current UI では通常 input ではなく contenteditable を含む前提で扱う
- `ボタン`
  - `アクション設定` で実体を確定する
- 保存
  - editor 上部の `メッセージを保存`
- テスト
  - 一覧行右端 `... -> テスト送信`
- 削除
  - 一覧行右端 `... -> 削除`

つまり、benchmark にする `フレックスメッセージ` は、表示だけでなく `保存 / テスト送信 / 削除` まで current で閉じるものを優先する。

## 制作者意図を読む時の問い

- この message を受け取る直前の相手は、何が分からず止まっているか
- その迷いを、どの text や画像で消そうとしているか
- 押した後の行動ハードルを、どこで下げているか
- `今すぐ押す理由` が 1 行で切れるか

## output の最小形

- 目的
- 変えたい認識
- 起こしたい行動
- 良い点
- 悪い点
- 再現すべき点
- 変えるならどこか

## 完成条件

- `テンプレート種` を `標準メッセージ / カルーセルメッセージ(新方式) / フレックスメッセージ / テンプレートパック` のどれかで固定できている
- `誰に / 何の認識を変え / 何を行動させるか` を 1 文で言える
- `画像 / テキスト / ボタン / action` の役割を分けて説明できる
- `良い点 / 悪い点 / 再現すべき点 / 変えるならどこか` を出せる
- `既存にある = 再現すべき` と短絡せず、quality の良し悪しを分けて言える
- 必要なら `メッセージを保存 -> テスト送信 -> 削除` まで current UI の言葉で説明できる

## ここで止めて確認する条件

- `標準メッセージ` で済むのに `フレックスメッセージ` にしたくなっている
- `フレックスメッセージ` の画像量が多いが、何を変えたいかが 1 本に見えない
- action が複数あり、main CTA がどれか言えない
- 既存テンプレートを benchmark にしたいが、なぜ良いのか言えない

## 保存前の最小チェック

- 目的と次行動を 1 文で言える
- 入力値や action の実体が確定している
- テスト後に残すか削除するか決めている

## 保存後の最小チェック

- 一覧と実挙動の両方で確認した
- `テスト送信` があるものは `甲原 海人` で確認した
- テスト用は削除して残っていない
