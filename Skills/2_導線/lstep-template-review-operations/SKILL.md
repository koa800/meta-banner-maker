# lstep-template-review-operations

## Overview

Lステップの `テンプレート` を、単なる message の見た目ではなく `コンテンツ` として分解する skill です。

ここでいう `コンテンツ` は、`コミュニケーションの手段であり、ユーザーをこちらが意図した認識に変換するもの` です。したがって、レビューの中心は

- 何の認識を変えたいか
- 何を行動させたいか
- そのために画像 / テキスト / ボタン / アクションをどう使っているか

です。

Addness 固有の account、命名規則、current representative は `Master/knowledge/lstep_structure.md` を正本にする。この skill は、どの案件でも再利用できる `レビューの型` だけを持つ。

## 役割

既存テンプレートを分解して、

- 何が良いか
- 何が悪いか
- 何を再現すべきか
- 何を変えるべきか

を、実装前に判断できる状態を作る。

## ゴール

次を迷わず言語化できる状態を作る。

- このテンプレートは何のために存在するか
- 誰のどんな状態を変えたいか
- 画像 / テキスト / ボタン / アクションの役割は何か
- 1 message で十分か、`テンプレートパック` に分けるべきか
- 良い点を残し、悪い点を変えるならどこか

## 必要変数

- テンプレート種
  - `標準メッセージ`
  - `カルーセルメッセージ(新方式)`
  - `フレックスメッセージ`
  - `テンプレートパック`
- テンプレート名
- 送る相手の状態
- 変えたい認識
- 起こしたい次行動
- CTA の種類
- action の種類
- current か legacy か

## レビューの順番

1. `何のテンプレートか`
2. `誰に送るか`
3. `何の認識を変えるか`
4. `何を行動させるか`
5. `その役割に対して message 種が合っているか`
6. `画像 / テキスト / ボタン / action の役割が分かれているか`
7. `前後の導線と滑らかにつながるか`

## テンプレート種ごとの見る場所

### `標準メッセージ`

- 主に text の流れを見る
- 1通で何をどこまで説明しているかを見る
- CTA が text の文脈から自然かを見る

### `カルーセルメッセージ(新方式)`

- 各 panel の役割分担を見る
- 比較させたいのか、選ばせたいのか、順番に見せたいのかを見る
- panel ごとに CTA がぶれていないかを見る

### `フレックスメッセージ`

- first view の強さを見る
- 画像 / タイトル / テキスト / ボタンの優先順位を見る
- `重要告知 / main CTA / 状態更新連動 / 障害告知` のどれに近いかを切る

### `テンプレートパック`

- step の順番を見る
- `認識変換の step` と `行動 step` が分かれているかを見る
- pack を使う必然性があるかを見る

## 良い / 悪いの判断軸

### 良い

- 1 message 1 目的になっている
- `何を理解してほしいか` が一読で切れる
- CTA が 1 つに絞られているか、複数でも役割が分かれている
- action が押した後の体験と一致している
- 前後の導線と文脈がつながっている

### 悪い

- 1 message に告知、説明、営業、FAQ を詰め込む
- 画像が decoration でしかない
- ボタンが複数あるのに、何が main か分からない
- 押した後の挙動が message の promise とずれる
- 前の接点と次の接点の間で文脈が切れる

## 実際に見る観点

### 画像

- 画像が promise の補強になっているか
- 画像がなくても message の意味が成立するか
- 画像が message の役割と無関係ではないか

### テキスト

- 各 text block に役割があるか
- `共感 / 問題提起 / 事実 / urgency / CTA前の後押し`
  のどれを担うか切れるか
- 長すぎて main CTA の妨げになっていないか

### ボタン

- main CTA が明確か
- 2つ以上あるなら役割を説明できるか
- 文言が抽象的ではないか

### action

- `URLを開く`
- `回答フォームを開く`
- `テンプレートを送信`
- `タグを追加する`
- `テキストを送信する`
など、押した後の動きが意図と一致しているかを見る

## Pack を使うかの判断

`テンプレートパック` を使うのは、次のどれかがある時。

- 1通で全部言うと認識負荷が高い
- 先に理由づけをしてから CTA を出したい
- `本文` と `visual CTA` を分けた方が自然

逆に、1通で役割が完結するなら pack にしない。

## exact なレビュー手順

1. `テンプレート` 一覧で対象を開く
2. `プレビュー`
3. `別窓`
4. 必要なら row 右端 `... -> テスト送信`
5. 実機で
  - 見た目
  - 文言
  - action
  - 押下後の挙動
  を確認
6. 次の型でメモする
  - `テンプレート種`
  - `目的`
  - `変えたい認識`
  - `起こしたい行動`
  - `良い点`
  - `悪い点`
  - `再現すべき点`
  - `変えるならどこか`

## output の型

最低限、次の 8 行に落とす。

- テンプレート種
- 目的
- 変えたい認識
- 起こしたい行動
- 良い点
- 悪い点
- 再現すべき点
- 変えるならどこか

## NG

- `見た目が良い` だけで評価する
- action を見ずに判断する
- 前後の導線を見ずに message 単体で完結評価する
- Addness 固有の current 例を、そのまま一般 skill に焼き込む
- `current` と `legacy` を混ぜて benchmark にする

## 正誤判断

正しい状態

- テンプレートの役割が 1 文で言える
- 何の認識を変えたいかが言える
- 次の行動が 1 つに絞れている
- 見た目だけでなく action まで確認している
- 再現すべき点と捨てるべき点を分けて言える

間違った状態

- なんとなく良さそうで終わる
- 画像や色だけを真似る
- 何のためのテンプレートか説明できない
- current 例と legacy 例を混ぜる
