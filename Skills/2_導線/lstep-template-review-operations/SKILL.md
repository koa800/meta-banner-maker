# lstep-template-review-operations

## Overview

Lステップの `テンプレート` を、単なる message の見た目ではなく `コンテンツ` として分解する skill です。

ここでいう `コンテンツ` は、`コミュニケーションの手段であり、ユーザーをこちらが意図した認識に変換するもの` です。したがって、レビューの中心は

- 何の認識を変えたいか
- 何を行動させたいか
- そのために画像 / テキスト / ボタン / アクションをどう使っているか

です。

Addness 固有の account、命名規則、current representative は `Master/knowledge/lstep_structure.md` を正本にする。この skill は、どの案件でも再利用できる `レビューの型` だけを持つ。

## ツールそのものの役割

`テンプレート` review は、message type ごとの構成、action、見た目、役割を分解して評価する作業です。

## アドネスでの役割

アドネスでは `テンプレート` review を、見た目の好みではなく、状態に合った message か、次の行動に必要十分か、current 品質に達しているかを見極める判断面として使う。

## 役割

既存テンプレートを分解して、

- 何が良いか
- 何が悪いか
- 何を再現すべきか
- 何を変えるべきか

を、実装前に判断できる状態を作る。

## ゴール

次を迷わず言語化できる状態を作る。

- このテンプレートは何のために存在するか
- 誰のどんな状態を変えたいか
- 画像 / テキスト / ボタン / アクションの役割は何か
- 1 message で十分か、`テンプレートパック` に分けるべきか
- 良い点を残し、悪い点を変えるならどこか

## 必要変数

- テンプレート種
  - `標準メッセージ`
  - `カルーセルメッセージ(新方式)`
  - `フレックスメッセージ`
  - `テンプレートパック`
- テンプレート名
- 送る相手の状態
- 変えたい認識
- 起こしたい次行動
- CTA の種類
- action の種類
- current か legacy か

## レビュー前の最小チェック

- そのテンプレートが `今も使う current` か、`legacy / draft` かを先に切ったか
- 送り先の相手状態を 1 文で言えるか
- `この message で変えたい認識` を 1 つに絞れているか
- `次にしてほしい行動` を 1 つに絞れているか
- 実機確認が必要な action かどうかを先に切ったか

## レビューの順番

1. `何のテンプレートか`
2. `誰に送るか`
3. `何の認識を変えるか`
4. `何を行動させるか`
5. `その役割に対して message 種が合っているか`
6. `画像 / テキスト / ボタン / action の役割が分かれているか`
7. `前後の導線と滑らかにつながるか`

## 最小 live review

最初の 1 本は `フレックスメッセージ` か `標準メッセージ` のどちらか 1 つだけを end-to-end で読む。

1. テンプレート種を 1 つ選ぶ
2. テンプレート名
3. 画像の役割
4. テキストの役割
5. main CTA
6. action の実体
7. `テスト送信` の有無

最初から `カルーセルメッセージ(新方式)` と `テンプレートパック` を混ぜて比較しない。まず 1 テンプレートで `何を認識させ、何を行動させるか` を言えることを優先する。

### 構築精度だけを見る時のチェック

品質の好き嫌いと、構築の正しさは分けて見る。
まずは次の 7 点が正しく組めているかを先に切る。

1. 形式選択が正しいか
   - `標準メッセージ / カルーセルメッセージ(新方式) / フレックスメッセージ / テンプレートパック` の選び方が目的に合っているか
2. main CTA が 1 つに切れているか
   - 複数あるなら主従を説明できるか
3. action の実体が合っているか
   - `URLを開く / 回答フォームを開く / テキストを送信 / タグを追加する / シナリオを動かす` のどれかを言えるか
4. action の入力値が合っているか
   - URL、フォーム、タグ、送信文言などが実値で入っているか
5. テスト方法まで決まっているか
   - `テスト送信` で何を見るかが決まっているか
6. 削除条件まで決まっているか
   - テスト用なら、残すか消すかを先に決めているか
7. 実機で最終挙動を確認したか
   - 見た目だけでなく、押した後の挙動まで確認したか

## テンプレート種ごとの見る場所

### current で最初に固定する UI ラベル

- `テンプレート`
- `新しいテンプレート`
- `標準メッセージ`
- `カルーセルメッセージ(新方式)`
- `フレックスメッセージ`
- `新しいパック`

まず `新しいテンプレート` 配下のどれを使うかを切る。`見た目が派手だから` ではなく、`何の認識を変え、何を行動させるか` で種別を決める。

### `標準メッセージ`

- 主に text の流れを見る
- 1通で何をどこまで説明しているかを見る
- CTA が text の文脈から自然かを見る

### `カルーセルメッセージ(新方式)`

- 各 panel の役割分担を見る
- 比較させたいのか、選ばせたいのか、順番に見せたいのかを見る
- panel ごとに CTA がぶれていないかを見る

### `フレックスメッセージ`

- first view の強さを見る
- 画像 / タイトル / テキスト / ボタンの優先順位を見る
- `重要告知 / main CTA / 状態更新連動 / 障害告知` のどれに近いかを切る
- `＋ブロックを追加` から何を足しているかを見る
- `メッセージを保存` 後に一覧行右端 `... -> テスト送信` まで通るかを見る

### `テンプレートパック`

- step の順番を見る
- `認識変換の step` と `行動 step` が分かれているかを見る
- pack を使う必然性があるかを見る

## 良い / 悪いの判断軸

### 良い

- 1 message 1 目的になっている
- `何を理解してほしいか` が一読で切れる
- CTA が 1 つに絞られているか、複数でも役割が分かれている
- action が押した後の体験と一致している
- 前後の導線と文脈がつながっている

### 悪い

- 1 message に告知、説明、営業、FAQ を詰め込む
- 画像が decoration でしかない
- ボタンが複数あるのに、何が main か分からない
- 押した後の挙動が message の promise とずれる
- 前の接点と次の接点の間で文脈が切れる

## 実際に見る観点

### 画像

- 画像が promise の補強になっているか
- 画像がなくても message の意味が成立するか
- 画像が message の役割と無関係ではないか

### テキスト

- 各 text block に役割があるか
- `共感 / 問題提起 / 事実 / urgency / CTA前の後押し`
  のどれを担うか切れるか
- 長すぎて main CTA の妨げになっていないか

### ボタン

- main CTA が明確か
- 2つ以上あるなら役割を説明できるか
- 文言が抽象的ではないか

### action

- `URLを開く`
- `回答フォームを開く`
- `テンプレートを送信`
- `タグを追加する`
- `テキストを送信する`
など、押した後の動きが意図と一致しているかを見る

## Pack を使うかの判断

`テンプレートパック` を使うのは、次のどれかがある時。

- 1通で全部言うと認識負荷が高い
- 先に理由づけをしてから CTA を出したい
- `本文` と `visual CTA` を分けた方が自然

逆に、1通で役割が完結するなら pack にしない。

## representative pattern

- `即行動型`
  - いま何をすればよいかを即伝える
  - `標準メッセージ` や短い `フレックスメッセージ` が向く
  - operational な CTA が 1 つに切れる
- `visual CTA 型`
  - 画像や panel で選ばせる
  - `カルーセルメッセージ(新方式)` や `フレックスメッセージ` が向く
  - action は `URLを開く` や `回答フォームを開く` が主役
- `状態更新連動型`
  - 押した後に tag、text、scenario が変わる
  - 見た目だけでなく action の実体が主役
- `段階誘導型`
  - `テンプレートパック` で理由づけと CTA を分ける
  - 1通で詰め込むと重くなる時に使う

## representative pattern を読む時の問い

- このテンプレートは `即行動 / visual CTA / 状態更新 / 段階誘導` のどれか
- `画像 / テキスト / ボタン / action` のうち何が主役か
- 押した後に何が変わる設計かを 1 文で言えるか
- 1 通で十分か、`テンプレートパック` に分けるべきか

## 最初の複雑パターン

最初に exact に分解する 1 本は、次のどれか 1 つに絞る。

- `フレックスメッセージ`
  - `画像 1 + テキスト 2 + ボタン 1`
  - action は `URLを開く` か `回答フォームを開く`
- `カルーセルメッセージ(新方式)`
  - `panel 2 枚`
  - 各 panel の CTA は 1 つだけ
- `テンプレートパック`
  - `認識変換 1 通 + 行動誘導 1 通`

最初から
- 画像多数
- CTA 多数
- `状態更新連動`
- 長尺 `テンプレートパック`
を同時に分解しない。まず 1 本で `何を認識させ、何を行動させるか` を切れることを優先する。

## exact なレビュー手順

1. `テンプレート` 一覧で対象を開く
2. `プレビュー`
3. `別窓`
4. 必要なら row 右端 `... -> テスト送信`
5. 実機で
  - 見た目
  - 文言
  - action
  - 押下後の挙動
  を確認
6. 次の型でメモする
  - `テンプレート種`
  - `目的`
  - `変えたい認識`
  - `起こしたい行動`
  - `良い点`
  - `悪い点`
  - `再現すべき点`
  - `変えるならどこか`

## 症状から最初に疑う場所

- 見た目は整っているのに押されない
  - まず `main CTA が 1 つに切れているか`
  - 次に `ボタン文言`
  - その後に `画像 / テキスト量`
- 情報量は多いのに読まれにくい
  - まず `1 message 1目的` になっているか
  - 次に `テンプレートパック` に分けるべき案件ではないか
  - その後に `画像が装飾だけになっていないか`
- action は正しいのに意図した行動が起きない
  - まず `押した後の promise` と `action` が一致しているか
  - 次に `遷移先` や `回答フォーム`
  - 最後に message 本文の後押し不足を疑う
- 既存テンプレートを流用したら違和感が出る
  - まず `誰のどんな状態向けか`
  - 次に `変えたい認識`
  - その後に `テンプレート種` 自体が合っているか

### `フレックスメッセージ` の exact smoke で最低限見ること

- 作成入口
  - `テンプレート -> 新しいテンプレート -> フレックスメッセージ`
- block 追加
  - `＋ブロックを追加 -> 画像`
  - `＋ブロックを追加 -> タイトル`
  - `＋ブロックを追加 -> テキスト`
  - `＋ブロックを追加 -> ボタン`
- 保存
  - editor 上部の `メッセージを保存`
- テスト
  - 一覧行右端 `... -> テスト送信`
- 削除
  - 一覧行右端 `... -> 削除`

つまり、`フレックスメッセージ` を benchmark にする時は、見た目だけでなく `画像 / タイトル / テキスト / ボタン / action / 保存 / テスト送信 / 削除` まで一連で読めるものを優先する。

## 保存前後ではなくレビュー完了前の最小チェック

- `目的`
- `変えたい認識`
- `起こしたい行動`
- `良い点`
- `悪い点`
- `再現すべき点`
が全部 1 行以上で埋まっているか
- ボタンや画像があるなら、実挙動まで確認したか
- `見た目が良い` だけの感想で止まっていないか

## output の型

最低限、次の 8 行に落とす。

- テンプレート種
- 目的
- 変えたい認識
- 起こしたい行動
- 良い点
- 悪い点
- 再現すべき点
- 変えるならどこか

## 実装前の最小チェック

- 何を benchmark としてレビューするか切っている
- current と legacy を混ぜていない
- 変えたい認識と起こしたい行動を 1 つに絞って読める
- 構築精度と content quality を分けて判定する前提になっている

## Workflow

1. 対象を current / legacy で切る
2. 目的と相手状態を言語化する
3. 構成要素を分解する
4. CTA や action の実体を確認する
5. 良い点、弱い点、再現点、変更点に落とす
6. 必要なら実機や preview で確かめる

## exact 手順

- 一覧や対象行を固定して preview または別窓を開く
- main CTA、actual hyperlink、final destination を分けて確認する
- 構築精度と content quality を分けてメモする
- benchmark にする点と捨てる点を分けて残す

## 保存前の最小チェック

- レビュー結果をどこへ書き戻すか決まっている
- current / legacy の判定が済んでいる
- main CTA と補助要素の主従が切れている
- 実 click で確認すべき箇所を切っている

## 保存後の最小チェック

- 良い点、弱い点、再現点、変更点が 1 行以上で埋まっている
- action や hyperlink の実体まで確認している
- benchmark にする理由を説明できる
- レビュー結果が次の実装判断に使える形になっている

## 完成条件

- 何の認識を変えるコンテンツか 1 文で言える
- 良い点、弱い点、再現点、変更点を残している
- 構築精度と content quality を分けて判断している
- 次の実装判断に使えるレビューになっている
- `フレックスメッセージ` なら `画像 / タイトル / テキスト / ボタン / action` のどれが主役かを言える
- 必要なら `テスト送信` と `削除` まで含めて再現手順として説明できる

## NG

- `見た目が良い` だけで評価する
- action を見ずに判断する
- 前後の導線を見ずに message 単体で完結評価する
- Addness 固有の current 例を、そのまま一般 skill に焼き込む
- `current` と `legacy` を混ぜて benchmark にする

## 正誤判断

正しい状態

- テンプレートの役割が 1 文で言える
- 何の認識を変えたいかが言える
- 次の行動が 1 つに絞れている
- 見た目だけでなく action まで確認している
- 再現すべき点と捨てるべき点を分けて言える

間違った状態

- なんとなく良さそうで終わる
- 画像や色だけを真似る
- 何のためのテンプレートか説明できない
- current 例と legacy 例を混ぜる

## ここで止めて確認する条件

- そのテンプレートが current か legacy か切れない
- `画像` と `ボタン` のどちらが main CTA か判断できない
- button action の実体が `URLを開く` 以外で、押下後の挙動が読めない
- 1 message に複数目的が混ざっていて、何を benchmark にすべきか切れない
