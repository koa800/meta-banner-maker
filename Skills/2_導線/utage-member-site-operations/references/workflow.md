## Addness 側での重要解釈

- 会員サイト修正は `本文修正` ではなく `解放設計` の一部
- 商品連動案件では `商品 -> detail -> action -> バンドルコース` まで見て初めて確定
- category は current 代表 3 site では `0件` だった
- つまり current 運用では category より `bundle / course / lesson` が主設計

## 保存前の最小チェック

- 対象が
  - `バンドルコース`
  - `コース`
  - `レッスン`
  - `固定ページ`
  - `お知らせ`
  のどれか言える
- この変更が、受講生の何を変えるか言える
- 商品連動案件なら、どの `購入後アクション` から入るか言える

## 保存後の最小チェック

- `バンドルコース` で対象 course が正しく入っている
- `コース` で `ステータス / 受講対象者 / 開放日(開始日)` が意図どおり
- `レッスン` で `種類 / コンテンツ / ステータス` が意図どおり
- 必要なら受講者視点で表示順と開放状態を確認する
- `お知らせ` や `固定ページ` を触った時も、対象 site の intended な導線から実際に到達できるか確認する

## 変更後の影響確認順

1. 変更した `バンドルコース / コース / レッスン / 固定ページ / お知らせ` の一覧へ戻る
2. row の `タイトル / コース名 / レッスン名 / ステータス` を再確認する
3. `受講対象者` と `開放日(開始日)` が変わっていないか見る
4. 商品連動案件なら `商品管理 -> 商品詳細管理 -> 購入後アクション -> バンドルコース` を逆引きしてズレがないか見る
5. 受講者視点で intended な導線に 1 回入り、表示順と開放状態を確認する

## ここで止めて確認する条件

- 直したい対象が `バンドルコース` なのか `コース` なのか曖昧
- 商品連動なのに `商品 -> detail -> action -> バンドルコース` の線が切れている
- `固定ページ / お知らせ` で足りるのか、course / lesson を直すべきか曖昧
- 既存受講生への影響が読めない
- 開放条件は変えたくないのに、`受講対象者 / 開放日(開始日)` まで触る必要が出ている

## 完成条件

- 直した対象を `バンドルコース / コース / レッスン / 固定ページ / お知らせ` のどれかで固定できている
- 変更が `どの受講者の何を変えるか` を 1 文で言える
- 商品連動なら `商品 -> detail -> action -> バンドルコース` を線で説明できる
- 保存後に `表示順 / 開放状態 / 対象者` を確認している
- 既存受講生への影響がないか、または影響範囲を説明できる
