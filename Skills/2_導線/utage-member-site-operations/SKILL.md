# utage-member-site-operations

## Overview

UTAGE の `会員サイト` を、本文から先に触らず `バンドルコース -> コース -> レッスン` の順で扱う skill です。

Addness 固有の site 名、current representative、どの site を主に参照するかは `Master/addness/utage_structure.md` を正本にする。この skill では、どの案件でも再利用できる exact 手順だけを持つ。

## ツールそのものの役割

UTAGE の `会員サイト` は、購入済みまたは解放済みの受講者に対して、講義、資料、固定ページ、お知らせを届ける機能です。

## アドネスでの役割

アドネスでは `会員サイト` を、単に動画を置く場所ではなく `商品購入後の顧客体験を成立させる実行面` として使う。したがって、本文編集より先に `どのバンドルコースで、どのコースが、誰に開放されるか` を先に切る。

## 役割

`会員サイト` は、購入済みまたは解放済みの顧客に対して、講義、資料、固定ページ、お知らせを届ける実行面です。

## ゴール

次を迷わずできる状態を作る。
- `会員サイト` 一覧から正しい site を選ぶ
- `バンドルコース -> コース -> レッスン` の順で安全に確認する
- `固定ページ` と `お知らせ` を補助導線として読む
- `商品 -> detail -> action -> バンドルコース` のつながりを逆引きできる
- 本文だけ直して解放条件を壊す事故を避ける

## 必要変数

- 対象 site 名
- 何を変えたいか
  - 解放先
  - コース構成
  - レッスン本文
  - 固定ページ
  - お知らせ
- どの bundle で解放しているか
- どの course に属するか
- どの lesson に属するか
- 受講対象者や公開条件を変えるか

## 実装前の最小チェック

- 変えたい対象が `バンドルコース / コース / レッスン / 固定ページ / お知らせ` のどれか切れているか
- その変更が `解放先` に影響するかを先に切ったか
- 商品や action とつながっている案件かを先に確認したか
- 受講対象者や公開条件を変える必要があるかを先に決めたか

## 判断フレーム

### まず `バンドルコース` から見る時

次のどれかなら、本文より先に `バンドルコース` を見る。
- どの講義が解放されるか確認したい
- 商品や action と会員サイトのつながりを確認したい
- 受講対象者の範囲を変える可能性がある

### `コース` から見る時

次のどれかなら、`コース` を主対象にする。
- コース名
- コース画像
- ボタンテキスト
- リンク先URL
- 進捗率
- 動画視聴完了設定
を見直したい

### `レッスン` から見る時

次のどれかなら、`レッスン` を主対象にする。
- 本文
- 動画
- コメント機能
- 公開日
- 開放前表示
を変えたい

## Workflow

1. `会員サイト`
2. site 一覧から対象 site を開く
3. `バンドルコース`
4. `コース`
5. `レッスン`
6. 必要なら `固定ページ` と `お知らせ`
7. 必要なら `設定 / 課金連動 / URL設定`
8. 商品や action と接続している案件なら、`商品 -> detail -> action -> バンドルコース` まで逆引きする

## representative pattern

### 保管庫全開放型

- 目的
  - 購入後に講義保管庫を一括で解放する
- 先に見る場所
  - `バンドルコース`
  - `商品詳細管理`
  - `購入後アクション`
- 向いている場面
  - main 商品の購入後導線

### コース追加型

- 目的
  - 既存 site に新しい講義を足す
- 先に見る場所
  - `コース`
  - `レッスングループ`
  - `レッスン`
- 向いている場面
  - 新しい教材追加
  - 既存コースの増補

### 固定ページ補助型

- 目的
  - ログイン案内やFAQの補助導線を置く
- 先に見る場所
  - `固定ページ`
  - `URL設定`
- 向いている場面
  - 補助導線
  - 受講生向けお知らせの補足

## representative pattern を読む時の問い

- この site 操作の主目的は `全体解放 / コース追加 / 補助導線` のどれか
- `バンドルコース` が主役なのか、`コース` が主役なのか、`レッスン` が主役なのか
- 変更が `受講体験` だけでなく `解放条件` に影響していないか
- 商品や action とつながる案件なら、逆引き先を説明できるか
- 補助ページだけ見れば十分か、それとも `バンドルコース` 側まで見るべきか

## ベストな活用場面

- 既存受講生サイトに新しい講義や案内を足す時
- `商品管理 -> 商品詳細管理 -> 購入後アクション` の downstream を確認したい時
- 受講体験の調整と、解放条件の安全確認を同時にやりたい時

## よくあるエラー

- `レッスン` だけ触って、どの `バンドルコース` で解放されているか見ていない
- `固定ページ` の文言だけ直して、受講導線全体との整合を見ていない
- 同名に近い `コース` や `レッスン` を取り違える
- `受講対象者` や公開条件が意図せず変わる

## エラー時の切り分け順

1. 対象 site 名を確認する
2. `バンドルコース` で解放束を確認する
3. `コース` で intended なコース名と管理名称を確認する
4. `レッスン` で intended な group と lesson 名を確認する
5. 商品連動案件なら `商品管理 -> 商品詳細管理 -> 購入後アクション -> バンドルコース` まで逆引きする

## 30秒レビューの順番

1. 対象 site 名
2. `バンドルコース`
3. `コース`
4. `レッスン`
5. `固定ページ / お知らせ`
6. 必要なら `商品 -> detail -> action -> バンドルコース` の逆引き

## 検証

最低でも次を確認する。
- どの bundle がどの course を解放するか説明できる
- どの course のどの lesson を触っているか言える
- 本文変更なら、公開条件や受講対象者がずれていない
- 固定ページやお知らせを触る時も、会員サイト全体の導線と矛盾していない
- 商品連動案件なら `商品 -> detail -> action -> bundle` まで逆引きできる

## 保存前の最小チェック

- 今触っているのが `bundle / course / lesson / page / news` のどれか言えるか
- 変更が `解放条件` に影響しないか確認したか

## 保存後の最小チェック

- 一覧に戻って対象行が intended な状態か
- 公開条件や受講対象者が意図どおりか
- 商品連動案件なら downstream 側も矛盾していないか

## 完成条件

- `バンドルコース -> コース -> レッスン` の順で current を説明できる
- 何を変える時にどの画面を見るか言える
- 解放条件と受講体験の両方を確認できる

## NG

- いきなり lesson 本文から触る
- bundle を見ずに解放先を断定する
- course 名だけ見て判断する
- `未使用` や `コピー` を current 正本だと決め打ちする
- 会員サイト修正なのに product / action 連動を無視する

## 正誤判断

正しい状態
- `バンドルコース -> コース -> レッスン` の順で説明できる
- 何を変える時にどの画面を見るべきか言える
- 受講体験と解放条件の両方を見ている

間違った状態
- 本文だけ変えて満足する
- bundle 名だけで解放先を判断する
- お知らせや固定ページだけを見て会員サイト全体を理解した気になる

## ここで止めて確認する条件

- どの bundle が current 正本か切れない
- 同じ名前の course や lesson が複数あって、対象を 1 つに絞れない
- 商品や action と連動している可能性が高いのに、接続先が読めない
- 本文変更だけのつもりでも、受講対象者や公開条件が変わりそう
