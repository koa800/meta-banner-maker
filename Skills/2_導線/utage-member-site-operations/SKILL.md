# utage-member-site-operations

## Overview

UTAGE の `会員サイト` を、本文から先に触らず `バンドルコース -> コース -> レッスン` の順で扱う skill です。

Addness 固有の site 名、current representative、どの site を主に参照するかは `Master/addness/utage_structure.md` を正本にする。この skill では、どの案件でも再利用できる exact 手順だけを持つ。

## ツールそのものの役割

UTAGE の `会員サイト` は、購入済みまたは解放済みの受講者に対して、講義、資料、固定ページ、お知らせを届ける機能です。

## アドネスでの役割

アドネスでは `会員サイト` を、単に動画を置く場所ではなく `商品購入後の顧客体験を成立させる実行面` として使う。したがって、本文編集より先に `どのバンドルコースで、どのコースが、誰に開放されるか` を先に切る。

## 役割

`会員サイト` は、購入済みまたは解放済みの顧客に対して、講義、資料、固定ページ、お知らせを届ける実行面です。

## ゴール

次を迷わずできる状態を作る。
- `会員サイト` 一覧から正しい site を選ぶ
- `バンドルコース -> コース -> レッスン` の順で安全に確認する
- `固定ページ` と `お知らせ` を補助導線として読む
- `商品 -> detail -> action -> バンドルコース` のつながりを逆引きできる
- 本文だけ直して解放条件を壊す事故を避ける

## 必要変数

- 対象 site 名
- 何を変えたいか
  - 解放先
  - コース構成
  - レッスン本文
  - 固定ページ
  - お知らせ
- どの bundle で解放しているか
- どの course に属するか
- どの lesson に属するか
- 受講対象者や公開条件を変えるか

## 依頼を受けたら最初に埋める項目

- 対象 `会員サイト`
- 対象が
  - `バンドルコース / コース / レッスン / 固定ページ / お知らせ`
  のどれか
- 何を変えたいか
  - 解放先
  - 構成
  - 本文
  - 公開条件
- upstream の `商品 / 商品詳細管理 / 購入後アクション`
- exploratory か、本線修正か

## 実装前の最小チェック

- 変えたい対象が `バンドルコース / コース / レッスン / 固定ページ / お知らせ` のどれか切れているか
- その変更が `解放先` に影響するかを先に切ったか
- 商品や action とつながっている案件かを先に確認したか
- 受講対象者や公開条件を変える必要があるかを先に決めたか

## 入口選択の exact な問い

- 今やりたいのは `受講体験そのものを直す` ことか
  - なら `会員サイト`
- 今やりたいのは `商品や購入後動作を変える` ことか
  - それはこの skill ではなく `商品管理 / 商品詳細管理 / 購入後アクション`
- 今やりたいのは `動画ファイルや章構成を直す` ことか
  - それはこの skill ではなく `動画管理`

つまり、この skill の入口は `受講後の見せ方と解放先が論点か` で決まる。

## 判断フレーム

## 10秒判断

- `バンドルコース`
  - 何を解放するか
  - どの商品や action とつながるか
  を見たい
- `コース`
  - コース名
  - コース画像
  - ボタンテキスト
  - 公開条件
  を直したい
- `レッスン`
  - 本文
  - 動画
  - コメント
  - 開放日
  を直したい
- `固定ページ / お知らせ`
  - 受講補助の案内や告知を直したい

この 10 秒判断が言えない時は、会員サイト一覧を開いても本文編集へ進まず stop する。

### まず `バンドルコース` から見る時

次のどれかなら、本文より先に `バンドルコース` を見る。
- どの講義が解放されるか確認したい
- 商品や action と会員サイトのつながりを確認したい
- 受講対象者の範囲を変える可能性がある

### `コース` から見る時

次のどれかなら、`コース` を主対象にする。
- コース名
- コース画像
- ボタンテキスト
- リンク先URL
- 進捗率
- 動画視聴完了設定
を見直したい

### `レッスン` から見る時

次のどれかなら、`レッスン` を主対象にする。
- 本文
- 動画
- コメント機能
- 公開日
- 開放前表示
を変えたい

### 新規追加を先に疑う時

次のどれかなら、既存本文の修正より `新規追加` を先に疑う。
- 新しい講義を足す
- 既存コースの中に新しい章を追加する
- 既存レッスンを壊さずに別 lesson として持ちたい

## Workflow

1. `会員サイト`
2. site 一覧から対象 site を開く
3. `バンドルコース`
4. `コース`
5. `レッスン`
6. 必要なら `固定ページ` と `お知らせ`
7. 必要なら `設定 / 課金連動 / URL設定`
8. 商品や action と接続している案件なら、`商品 -> detail -> action -> バンドルコース` まで逆引きする

## live probe で確認済みの最小 create/delete

- `バンドルコース一覧 > 追加`
  - 必須: `バンドルコース名` と `追加するコース` の最小選択
  - current では `create -> delete` を再確認済み
- `コース一覧 > 追加`
  - 必須: `コース名` `管理名称` `リンク先URL`
  - current では `create -> delete` を再確認済み
- `レッスン一覧 > 追加`
  - 必須: `グループ` `レッスン名` `コンテンツ`
  - current では save 後に一覧へ `1件` 出ることと、row delete で `0件` に戻ることを再確認済み

## create 入口の actual ラベル

- `バンドルコース一覧 > 追加`
  - `バンドルコース名 必須`
  - `追加するコース`
- `コース一覧 > 追加`
  - `コース名 必須`
  - `管理名称`
  - `種類`
  - `リンク先URL 必須`
- `レッスン一覧 > 追加`
  - `グループ 必須`
  - `レッスン名 必須`
  - `種類 必須`
  - `コンテンツ`
  - `ステータス`
  - `コメント機能`
  - `受講対象者`
  - `開放日(開始日)`
  - `開放前の表示`

## 最小 downstream smoke

会員サイトの確認は、その場の本文だけで閉じない。

1. intended な `バンドルコース`
2. intended な `コース`
3. intended な `レッスン`
4. upstream の `購入後アクション`
5. 商品や detail から見ても同じ bundle に着くか

解放先が intended でない限り、本文が正しく見えていても完了にしない。

## 最小 live change

既存 current の会員サイトを触る時は、最初の 1 回を `1 field だけ変える` に制限する。

1. 対象 `会員サイト > site 行`
2. `コース` または `レッスン` の対象行を 1 つ固定
3. 変更対象を 1 つに固定
   - `コース名`
   - `ボタンテキスト`
   - `レッスン名`
   - `固定ページのタイトル`
   のような downstream 影響が軽い field を優先
4. 保存
5. 一覧へ戻る
6. `バンドルコース -> コース -> レッスン` の順で再読
7. 公開条件や受講対象者が変わっていないことを確認
8. 必要なら元に戻す

最初から `受講対象者`、`開放日(開始日)`、`未ログイン時の閲覧`、`バンドルコース` を同時に変えない。最初は `表示系 field 1 つ` で change-safe な edit を通す。

## 最小 rollback

exploratory に変更した時は、`site / course / lesson` のどこを戻すかを先に固定する。

1. 対象 `会員サイト`
2. 対象 `バンドルコース / コース / レッスン / 固定ページ / お知らせ`
3. 変更した 1 field
4. 同じ画面で元の値へ戻す
5. 一覧へ戻る
6. `バンドルコース -> コース -> レッスン`
   の順で再確認

`バンドルコース` や `受講対象者` を exploratory に変えた時は、本文だけ戻しても cleanup にならない。解放束から再確認して rollback 完了とする。

## 変更後の最小 smoke

本文が変わっただけで完了にしない。

1. intended な `バンドルコース`
2. intended な `コース`
3. intended な `レッスン`
4. intended な `固定ページ / お知らせ`
5. upstream の `商品 / 商品詳細管理 / 購入後アクション`

この 5 つを 1 本で言えない時は、会員サイト変更を広げない。

## 最小 live test

最初の 1 本は `バンドルコース -> コース -> レッスン` の読解を end-to-end で通す。

1. `会員サイト`
2. representative な site 行
3. `バンドルコース`
4. representative な bundle
5. `コース`
6. representative な course
7. `レッスン`
8. representative な lesson
9. 必要なら `固定ページ`
10. 必要なら `お知らせ`

最初から本文を保存しない。まず `どの bundle がどの course を解放し、その中でどの lesson が見えるか` を無迷いで追えることを優先する。

### save を伴う最小 live test の条件

次をすべて満たす時だけ、最小 live save を許可する。
- `テスト` か `ZZ_TEST` と分かる隔離 course / lesson がある
- current 受講中ユーザーへ影響しない
- 変更が `名称` や軽微な本文で済む
- 保存後に自分で戻せる

隔離 course / lesson がない時は、会員サイトの live test は `読む / preview 相当の確認` までで止める。

## ログイン異常時の見分け方

- `会員サイト` や `コース` の URL を開いたはずなのに `オペレーターログイン | UTAGE` が出る
- site 一覧から入り直しても login に戻される

この時は `会員サイト` の設定不備ではなく、UTAGE の session 切れと判断する。

## representative pattern

### 保管庫全開放型

- 目的
  - 購入後に講義保管庫を一括で解放する
- 先に見る場所
  - `バンドルコース`
  - `商品詳細管理`
  - `購入後アクション`

## 最初の複雑パターン

最初に live で詰める複雑パターンは、`バンドルコース 1本 + コース 1本 + レッスン 1本` の downstream 確認だけで閉じる。

1. `会員サイト`
2. representative な site を 1 つ固定
3. `バンドルコース設定`
4. representative な bundle を 1 つ固定
5. `コース`
6. representative な course を 1 つ固定
7. `レッスングループ`
8. representative な lesson を 1 つ固定
9. intended な `開放される講義束` を 1 文で言う
10. exploratory なら save せず一覧へ戻る

最初から
- course 新規追加
- lesson 新規追加
- 本文 save
を同時にやらない。
- 向いている場面
  - main 商品の購入後導線

### コース追加型

- 目的
  - 既存 site に新しい講義を足す
- 先に見る場所
  - `コース`
  - `レッスングループ`
  - `レッスン`
- 向いている場面
  - 新しい教材追加
  - 既存コースの増補

### 固定ページ補助型

- 目的
  - ログイン案内やFAQの補助導線を置く
- 先に見る場所
  - `固定ページ`
  - `URL設定`
- 向いている場面
  - 補助導線
  - 受講生向けお知らせの補足

### 新規講義追加型

- 目的
  - 既存の会員サイトに新しい講義体験を足す
- 先に見る場所
  - `コース`
  - `レッスングループ`
  - `レッスン`
  - 必要なら `バンドルコース`
- 向いている場面
  - 新しい教材追加
  - 新しい章の追加
  - 別動画を lesson として独立追加

## 既存を流用してよい条件

- 受講対象者が同じ
- 解放先の `バンドルコース` を変えない
- 既存 `コース` の promise を壊さない
- `レッスン` を足しても、既存受講者の導線が混乱しない

上のどれか 1 つでも怪しい時は、既存の本文修正より `新規追加` を先に疑う。

## representative pattern を読む時の問い

- この site 操作の主目的は `全体解放 / コース追加 / 補助導線` のどれか
- `バンドルコース` が主役なのか、`コース` が主役なのか、`レッスン` が主役なのか
- 変更が `受講体験` だけでなく `解放条件` に影響していないか
- 商品や action とつながる案件なら、逆引き先を説明できるか
- 補助ページだけ見れば十分か、それとも `バンドルコース` 側まで見るべきか

## ベストな活用場面

- 既存受講生サイトに新しい講義や案内を足す時
- `商品管理 -> 商品詳細管理 -> 購入後アクション` の downstream を確認したい時
- 受講体験の調整と、解放条件の安全確認を同時にやりたい時

## よくあるエラー

- `レッスン` だけ触って、どの `バンドルコース` で解放されているか見ていない
- `固定ページ` の文言だけ直して、受講導線全体との整合を見ていない
- 同名に近い `コース` や `レッスン` を取り違える
- `受講対象者` や公開条件が意図せず変わる

## エラー時の切り分け順

1. 対象 site 名を確認する
2. `バンドルコース` で解放束を確認する
3. `コース` で intended なコース名と管理名称を確認する
4. `レッスン` で intended な group と lesson 名を確認する
5. 商品連動案件なら `商品管理 -> 商品詳細管理 -> 購入後アクション -> バンドルコース` まで逆引きする

## 30秒レビューの順番

1. 対象 site 名
2. `バンドルコース`
3. `コース`
4. `レッスン`
5. `固定ページ / お知らせ`
6. 必要なら `商品 -> detail -> action -> バンドルコース` の逆引き

## 検証

最低でも次を確認する。
- どの bundle がどの course を解放するか説明できる
- どの course のどの lesson を触っているか言える
- 本文変更なら、公開条件や受講対象者がずれていない
- 固定ページやお知らせを触る時も、会員サイト全体の導線と矛盾していない
- 商品連動案件なら `商品 -> detail -> action -> bundle` まで逆引きできる

## 保存前の最小チェック

- 今触っているのが `bundle / course / lesson / page / news` のどれか言えるか
- 変更が `解放条件` に影響しないか確認したか

## 新規講義追加の exact 順

1. 対象 site を開く
2. `コース`
3. 既存 course を流用するか、新しい course を足すか切る
4. 既存 course 内で足すなら `レッスングループ`
5. `グループ追加`
6. `レッスン管理`
7. `レッスン追加`
8. `種類`
9. `コンテンツ`
10. 必要なら `動画管理` で動画を先に確認
11. 必要なら `メディア管理` で画像や資料を先に確認
12. `バンドルコース` に戻って、解放対象に含まれるか確認

新規追加では、`まず asset、次に lesson、最後に bundle` の順で見る。本文から作り始めて、最後に解放漏れに気づく流れを避ける。

## 保存後の最小チェック

- 一覧に戻って対象行が intended な状態か
- 公開条件や受講対象者が意図どおりか
- 商品連動案件なら downstream 側も矛盾していないか

## 変更後の影響確認順

1. `バンドルコース`
2. `コース`
3. `レッスン`
4. `固定ページ / お知らせ`
5. 必要なら `商品管理 -> 商品詳細管理 -> 購入後アクション`

本文を変えたつもりでも、必ず `解放束` 側から逆流して見る。これで `見た目は正しいが解放条件が崩れた` を先に潰せる。

## 構築精度だけを見る時のチェック

1. 今触っているのが `バンドルコース / コース / レッスン / 固定ページ / お知らせ` のどれか切れているか
2. `バンドルコース -> コース -> レッスン` のどこが主役か 1 文で言えるか
3. 本文変更だけのつもりでも、公開条件や受講対象者を崩していないか
4. 商品連動案件なら `商品管理 -> 商品詳細管理 -> 購入後アクション -> バンドルコース` まで逆引きできるか
5. 同名に近い course / lesson を取り違えていないか
6. 補助導線の fixed page だけを見て、会員サイト全体を理解した気になっていないか

## 完成条件

- `バンドルコース -> コース -> レッスン` の順で current を説明できる
- 何を変える時にどの画面を見るか言える
- 解放条件と受講体験の両方を確認できる

## 症状から最初に疑う場所

- 本文は合っているのに受講者体験が違う
  - まず `バンドルコース`
  - 次に `商品管理 -> 商品詳細管理 -> 購入後アクション`
  - 本文だけを主因にしない
- 受講者が見える講義が足りない / 多すぎる
  - まず current bundle
  - 次に bundle に入っている course
  - その後に受講対象者や公開条件
- lesson を直したのに intended な画面が変わらない
  - まず同名の course / lesson 取り違え
  - 次に fixed page やお知らせを見ていないか
  - その後に公開条件
- 新しい講義を足したいが既存へ追加か別 course か迷う
  - まず promise が同じか
  - 次に解放単位が同じか
  - どちらかが違うなら別 course を先に疑う
- fixed page だけ直して会員サイト修正が終わった気がする
  - まず `バンドルコース -> コース -> レッスン`
  - fixed page は補助導線として後から確認する

## NG

- いきなり lesson 本文から触る
- bundle を見ずに解放先を断定する
- course 名だけ見て判断する
- `未使用` や `コピー` を current 正本だと決め打ちする
- 会員サイト修正なのに product / action 連動を無視する

## 正誤判断

正しい状態
- `バンドルコース -> コース -> レッスン` の順で説明できる
- 何を変える時にどの画面を見るべきか言える
- 受講体験と解放条件の両方を見ている

間違った状態
- 本文だけ変えて満足する
- bundle 名だけで解放先を判断する
- お知らせや固定ページだけを見て会員サイト全体を理解した気になる

## ここで止めて確認する条件

- どの bundle が current 正本か切れない
- 同じ名前の course や lesson が複数あって、対象を 1 つに絞れない
- 商品や action と連動している可能性が高いのに、接続先が読めない
- 本文変更だけのつもりでも、受講対象者や公開条件が変わりそう
- 新規 lecture を追加したいが、既存 course に入れるべきか別 course にすべきか切れない
- 既存受講者に見える順番や promise が変わりそう
