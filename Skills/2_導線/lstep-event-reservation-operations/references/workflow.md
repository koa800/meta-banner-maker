# Lステップ イベント予約 workflow

## current で固定している事実

- 一覧 URL
  - `https://manager.linestep.net/line/reserve/event`
- 画面上部ボタン
  - `新しいフォルダ`
  - `新しいイベント`
- row action
  - `友だち予約URL`
  - `友だち予約履歴URL`
  - `予約枠一覧`
  - `CSV出力`
  - `予約者一覧`
  - `プレビュー`

## live で確認した exact 手順

1. `イベント予約`
2. `新しいイベント`
3. イベントの基本情報を入力して `登録`
4. 一覧で `友だち予約URL`
5. `友だち予約履歴URL`
6. `予約枠一覧`
7. `CSV出力`
8. `予約者一覧`
9. `プレビュー`
10. テスト用なら `more_vert -> 削除 -> 削除する`

## 読み方

- `イベント予約` は、単発イベントや合宿の予約受付を回す operational screen
- `友だち予約URL` は新規予約の入口
- `友だち予約履歴URL` は既予約者向けの履歴・確認導線
- current の Addness では、`スキルプラス【サポートLINE】` が告知口、`【予約窓口】スキルプラス合宿` が実務予約口に寄っている

## 30秒レビューの順番

1. このイベントの役割を 1 文で言う
2. `新規予約` と `既予約者確認` の導線を分けて考える
3. `友だち予約URL / 友だち予約履歴URL / 予約枠一覧 / 予約者一覧` の使い分けを見る
4. `プレビュー` で実際の見え方を確認する
5. テスト用なら削除まで行く

## 完成条件

- 対象イベントの役割を 1 文で言える
- `友だち予約URL / 友だち予約履歴URL / 予約枠一覧 / 予約者一覧` の使い分けを説明できる
- 作成後に `プレビュー` まで確認している
- テスト用なら削除まで終えている

## ここで止めて確認する条件

- `友だち予約URL` と `友だち予約履歴URL` の違いが曖昧
- 告知口と実務予約口の切り分けが言えない
- 予約枠を見ずに公開しようとしている

## 保存前の最小チェック

- 目的と次行動を 1 文で言える
- 入力値や action の実体が確定している
- テスト後に残すか削除するか決めている

## 保存後の最小チェック

- 一覧と実挙動の両方で確認した
- `テスト送信` があるものは `甲原 海人` で確認した
- テスト用は削除して残っていない
