# utage-tracking-operations

## Overview

UTAGE の `登録経路` を、`追加` だけでなく `データ(合算) / データ(日別)` の読み方まで含めて exact に扱う skill です。

Addness 固有の funnel 名、current representative、どの粒度で登録経路を切るかの運用判断は `Master/addness/utage_structure.md` を正本にする。この skill は、会社をまたいで再利用できる exact 手順だけを持つ。

## ツールそのものの役割

`登録経路` は、同じページでも流入元を分けて記録し、分析できる tracking unit です。

## アドネスでの役割

アドネスでは `登録経路` を、広告IDのためだけでなく、`どこから来たかを知りたい時に切る分析軸` として使う。ページ複製で吸収すべきか、同一ページのまま登録経路で分けるべきかを判断するための主道具です。

## 役割

`登録経路` は、同じページへの流入でも `どこから来たか` を分けて計測するための分析単位です。

## ゴール

次を迷わずできる状態を作る。
- `登録経路` の正しい入口を選ぶ
- `追加` で新しい登録経路を作る
- `グループ / 管理名称 / ファネルステップ / ページ` を正しく設定する
- `データ(合算)` と `データ(日別)` で確認する
- `グループ管理` と `表示順変更` の入口を把握する

## 必要変数

- 対象 funnel
- 何を見分けたいか
- グループ
- 管理名称
- ファネルステップ
- ページ
- 合算で見るか日別で見るか

## 判断フレーム

### `登録経路` を切る時

次を満たすなら、まず `登録経路` を疑う。
- 同じページを使う
- どこから来たかを分けて見たい
- 分析軸を増やしたい

### 新規ページを切る時

次を満たすなら、`登録経路` ではなく新規ページを疑う。
- ページの中身自体が変わる
- コンセプトが大きく変わる
- クリエイティブごとに最適化した別LPを作る

## 実装前の最小チェック

- 何を見分けたい登録経路か 1 文で言える
- 新規ページではなく `登録経路` を切る理由を説明できる
- `グループ / 管理名称 / ファネルステップ / ページ` を先に決めている

## Workflow

1. 対象 funnel を開く
2. `登録経路`
3. `追加`
4. `グループ / 管理名称 / ファネルステップ / ページ` を設定
5. 保存後に `データ(合算)` と `データ(日別)` で確認
6. 必要なら `グループ管理` と `表示順変更`

## exact 手順

### current 主要 route

- `データ(合算) = /funnel/{funnel_id}/data`
- `データ(日別) = /funnel/{funnel_id}/data/daily`
- `登録経路 = /funnel/{funnel_id}/tracking`
- `追加 = /funnel/{funnel_id}/tracking/create`
- `グループ管理 = /funnel/{funnel_id}/tracking/group`
- `表示順変更 = /funnel/{funnel_id}/tracking/sort`

### create

1. `登録経路`
2. `追加`
3. create 画面で次を設定する
   - `グループ`
   - `管理名称`
   - `ファネルステップ`
   - `ページ`
4. 保存

### live で確認できた current 実装

- temporary funnel 配下でも `create -> delete` が通る
- current create form の visible labels
  - `管理名称`
  - `ファネルステップ`
  - `ページ`
- current select name
  - `step_id`
  - `page_id`
- `管理名称` だけでは保存されない
- 少なくとも `ファネルステップ` は選択が必要
- temporary funnel 直後は `ページ = 指定しない` だけのことがある

つまり最初の exact 判断は、
- `step_id` に intended な step があるか
- `page_id` に intended な page があるか
を保存前に見ること

これが取れない時は、`登録経路` 追加より先に funnel/page 側の状態を疑う

### 作成後の確認

1. `データ(合算)` へ移動
2. `期間 / 集計方法 / 登録経路` の条件が visible か確認
3. `データ(日別)` でも同じく `登録経路` 条件を確認
4. 一覧で `登録経路名` と `接続ページ名` を確認

## 保存前の最小チェック

- `管理名称` を見て流入元が分かるか
- どの `ファネルステップ / ページ` に属するか切れているか
- 本当に分析したい軸があるか

## 保存後の最小チェック

- `データ(合算)` と `データ(日別)` の両方で条件が使えるか
- 一覧で `登録経路名` と `接続ページ名` が intended か
- `delete_action` を row 単位で見つけられるか

## 完成条件

- なぜ `登録経路` を増やしたか説明できる
- data 画面で intended な分析軸として読める

## ここで止めて確認する条件

- 本当に `登録経路` を増やすべきか、新規ページを切るべきか説明できない
- `管理名称` を見ても流入元や分析軸が読めない
- `ファネルステップ` と `ページ` のどちらかを current に 1 つへ絞れない
- data 画面で何を比較したいかが曖昧

## 検証

最低でも次を確認する。
- 何を見分けたい経路か説明できる
- `管理名称` が分析で見て分かる名前になっている
- `ファネルステップ` と `ページ` が意図どおり
- `データ(合算)` と `データ(日別)` の両方で `登録経路` 条件が使える

## NG

- ページの中身が同じなのに、分析のためだけにページを複製する
- `管理名称` を見ても流入元が分からない
- `ファネルステップ` と `ページ` を曖昧に選ぶ
- `データ(日別)` を見ずに終える
- `管理名称` だけ入れて保存し、step/page の結びつきを未確認のまま終える

## 正誤判断

正しい状態
- `登録経路` を切る理由が説明できる
- どの step とどの page に属するか言える
- 作成後に data 画面でも確認している

間違った状態
- 何を分けて見たいかが曖昧
- `登録経路` と新規ページの使い分けが崩れている
- 一覧で作っただけで分析画面を見ていない
