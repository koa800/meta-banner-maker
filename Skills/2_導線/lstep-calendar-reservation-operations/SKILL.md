# lstep-calendar-reservation-operations

## Overview

Lステップの `カレンダー予約` を、`新しいカレンダー` の作成だけでなく `予約枠 / コース`、`アクション`、`リマインダ / フォロー`、`外部サービス連携` の分離を意識して exact に扱う skill です。

Addness 固有の account 選定、命名規則、current representative は `Master/knowledge/lstep_structure.md` を正本にする。この skill は、会社をまたいで再利用できる exact 手順だけを持つ。

## 役割

`カレンダー予約` は、1on1、個別相談、面談、担当者ごとの枠管理を回す operational screen です。

## ゴール

次を迷わずできる状態を作る。
- `カレンダー予約` の正しい入口を選ぶ
- カレンダー、予約枠、コースの責務を分けて理解する
- `予約直後 / 前日 / 直前` の送信をコースごとに分岐する
- 表示条件と Google カレンダー連携を混同せずに設定する
- UI だけでなく API でも current state を検証する

## 必要変数

- 対象 account
- カレンダー名
- 予約枠名
- コース名
- 予約対象者の条件
  - タグ
  - コース
- 予約後に送る内容
  - 予約直後
  - 前日
  - 直前
- Google カレンダー連携先
  - 予約を書き込むカレンダー
  - シフト元にするカレンダー
- テスト用か本番用か

## 本質の切り分け

`カレンダー予約` は少なくとも次の 3 レイヤーで分けて扱う。

- `予約枠 / コース`
  - 何を予約させるか
- `アクション / リマインダ / フォロー`
  - 予約後に何を送るか
- `外部サービス連携`
  - どの Google カレンダーに予定を書き、どの Google カレンダーをシフトとして使うか

一覧名だけで送信対象を判断しない。実際の送信対象は条件設定内の `カレンダー予約 / 予約枠 / コース` で分岐している。

## Workflow

1. 対象 account と対象カレンダーを固定する
2. `予約枠 / コース` で何を予約させるか決める
3. `アクション` で予約直後メッセージを決める
4. `リマインダ / フォロー` で前日 / 直前の送信を分ける
5. 必要なら表示条件を設定する
6. `外部サービス連携` で Google カレンダー連携を設定する
7. UI と API の両方で current state を確認する

## exact 手順

### 入口

1. `カレンダー予約`
2. 対象カレンダー row を開く
3. 左側で少なくとも次のタブを見分ける
   - `予約一覧`
   - `シフト`
   - `予約設定`
4. `予約設定` 配下で少なくとも次を見分ける
   - `予約受付`
   - `予約枠 / コース`
   - `予約画面`
   - `アクション`
   - `リマインダ / フォロー`
   - `外部サービス連携`

### 新しいコースや面談を足す時

1. `予約枠 / コース` で対象の `予約枠` と `コース` を確認する
2. 既存カレンダーに足すか、新しいカレンダーを作るかを決める
3. 同じカレンダー内に別コースを共存させる時は、後続の `アクション` と `リマインダ / フォロー` で必ず `コース` 条件まで切る

### 予約直後メッセージ

1. `アクション`
2. 対象 action を新規作成または編集する
3. 条件設定で少なくとも次を固定する
   - `カレンダー予約`
   - `予約枠`
   - `コース`
4. `予約直後` に送りたい text または template を設定する

### 前日 / 直前リマインド

1. `リマインダ / フォロー`
2. 対象 step を新規作成または編集する
3. timing を決める
   - 前日
   - 1時間前
   - 15分前
4. step に紐づく action の条件設定で `コース` まで絞る

### 表示条件

1. `誰に見せるか` を先に固定する
   - タグで切るか
   - コースで切るか
2. `表示されること` と `送信されること` を混同しない
   - 表示条件は予約ページの露出
   - action 条件は送信対象

### Google カレンダー連携

1. `外部サービス連携`
2. `Googleカレンダー連携設定を追加する`
3. modal で次を選ぶ
   - `Googleアカウント名`
   - `カレンダーID`
   - `予約枠`
   - `連携対象`
4. `連携対象` は次から選ぶ
   - `すべて`
   - `予約をGoogleカレンダーに連携`
   - `Googleカレンダーをシフトに連携`

重要
- `外部サービス連携` は `コース` 単位ではなく `予約枠` 単位
- 同じ予約枠の `予約 -> Googleカレンダー` を複数 Google カレンダーへ向ける保存は通らない
- 予約だけ別カレンダーに出したい時は、既存連携を整理するか `reservation-only / shift-only` で役割を分ける

## API での確認

UI だけで終わらせず、必要に応じて次を確認する。

- `GET /api/salon/{salon_id}`
  - episode steps と action_id を見る
- `GET /api/action/data/{action_id}`
  - action の条件設定を見る
- `GET /api/salon/{salon_id}/googlecalendars`
  - Google カレンダー association を見る
- `POST /api/salon/{salon_id}/googlecalendars`
  - 連携追加
  - body 例
    - `{"salon_resource_id":109117,"google_token_id":16079,"google_calendar_id":"koa800sea.nifs@gmail.com","link_target":"all"}`
- `DELETE /api/salon/{salon_id}/googlecalendars/{google_calendar_primary_id}/{purpose}/{association_id}`
  - 連携解除

## 検証

最低でも次を確認する。
- 対象コースだけに予約直後メッセージが当たる
- 前日 / 直前リマインドが別コースに漏れていない
- 表示条件が意図どおりに絞れている
- Google カレンダー連携先が正しい
- 必要なら `reservation` と `shift` の association が両方ある

## NG

- `表示条件` と `送信条件` を同じものとして扱う
- 一覧名だけ見て、送信対象が分かっているつもりになる
- `コース` を増やしただけで Google カレンダーも分かれると思い込む
- `予約 -> カレンダー` と `カレンダー -> シフト` を混同する
- UI の見た目だけで完了扱いにし、API や実挙動を確認しない

## 正誤判断

正しい状態
- `誰に見せるか / 何を送るか / どこへ同期するか` を別々に説明できる
- action 条件を `コース` まで切っている
- Google カレンダー連携が `予約枠` 単位だと説明できる
- current state を UI と API の両方で確認している

間違った状態
- コース名だけ見て送信対象が分かれていると思い込む
- Google カレンダー設定を変えたのに `reservation / shift` の両方を確認しない
- 一時テストの連携変更で予約反映先が空になる
