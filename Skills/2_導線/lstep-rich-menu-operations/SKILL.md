# lstep-rich-menu-operations

## Overview

Lステップの `リッチメニュー` を、`作成方法の選択 -> 編集 -> 保存` まで exact に再現する skill です。

Addness 固有の current representative、account 選定、current / legacy の見分けは `Master/knowledge/lstep_structure.md` を正本にする。この skill は、会社をまたいで再利用できる exact 手順だけを持つ。

## 役割

`リッチメニュー` は、LINE トーク画面下部の固定ナビとして、ユーザーを次のテンプレート、フォーム、URL、action へ分岐させる機能です。

## ゴール

次を迷わずできる状態を作る。
- `リッチメニュー` の正しい入口を選ぶ
- 作成方法の分岐を選ぶ
- 画像、タイトル、フォルダ、表示設定を入れる
- ボタンごとに action を割り当てる
- `保存`

## 必要変数

- メニュー名
- 使う画像
- どの account で出すか
- 誰に見せるか
- 初期表示か切替用か
- ボタン数
- 各ボタンの役割
- 各ボタンの action 種別

## 使い分け

`リッチメニュー` は、すべてを直接詰め込む場所ではありません。

基本思想
- `リッチメニュー` で入口を見せる
- 押下後に `テンプレート送信 / 回答フォーム / URL / action`
  へ分業する

## Workflow

1. `リッチメニュー`
2. `新しいリッチメニュー`
3. 作成方法を選ぶ
4. 画像と基本情報を入れる
5. 各ボタンの action を設定する
6. `保存`
7. 必要なら current representative と役割を比較する

## exact 手順

### create 入口

1. `リッチメニュー`
2. `新しいリッチメニュー`
3. 作成方法を選ぶ

作成方法の current 分岐
- `画像をアップロードして作成`
- `テンプレートをベースに作成`

### editor で見える主要ラベル

- `画像`
- `タイトル`
- `フォルダ`
- `トークルームメニュー`
- `メニューの初期状態`
- `プレビュー`
- `テンプレート`
- `コンテンツ設定`
- `一覧へ戻る`
- `保存`

### ボタン action

各ボタンで切り替えられる current 種類
- `URL`
- `TEL`
- `ユーザーメッセージ`
- `アクション`
- `回答フォーム`
- `その他`

必要に応じて `アクション設定` を開き、テンプレート送信や action を割り当てる。

## 検証

最低でも次を確認する。
- 画像が正しい
- ボタンの役割が重複していない
- どのボタンを押すべきか一読で分かる
- 保存前に current representative と比べて責務過多になっていない

## NG

- `リッチメニュー` 自体に全部の説明を詰め込む
- current / old / 作成中 を名前だけで判定する
- 画像だけ入れて action を確認しない
- すべてのボタンを同じ役割にする

## 正誤判断

正しい状態
- 入口としての役割が明確
- ボタンごとの役割が分かれている
- `テンプレート送信 / 回答フォーム / URL / action` の分業ができている
- current representative と比べても責務が自然

間違った状態
- 説明も行動も全部リッチメニュー自体に押し込む
- current / legacy の混在を読まずに古い menu を編集する
- ボタン action を設定せず保存する
