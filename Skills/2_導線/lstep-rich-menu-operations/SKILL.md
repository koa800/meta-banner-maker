# lstep-rich-menu-operations

## Overview

Lステップの `リッチメニュー` を、`作成方法の選択 -> 編集 -> 保存` まで exact に再現する skill です。

Addness 固有の current representative、account 選定、current / legacy の見分けは `Master/knowledge/lstep_structure.md` を正本にする。この skill は、会社をまたいで再利用できる exact 手順だけを持つ。

live で触る前に、必ず `python3 System/scripts/funnel_tool_live_readiness.py` を実行し、`lstep.auth_alive=true` に加えて `lstep.account_name` が intended な公式LINE account か確認する。`auth_alive=true` でも別 account 文脈を見ていることがある。

## ツールそのものの役割

`リッチメニュー` は、LINE トーク画面下部の固定ナビとして、ユーザーを特定の action、URL、テンプレート、回答フォームへ分岐させる機能です。

## アドネスでの役割

アドネスでは `リッチメニュー` を、情報を全部載せる場所ではなく、`今この状態の人をどこへ進ませるか` を分岐させる入口として使う。説明の主役ではなく、次のコンテンツや action へ渡すナビです。

## 役割

`リッチメニュー` は、LINE トーク画面下部の固定ナビとして、ユーザーを次のテンプレート、回答フォーム、URL、action へ分岐させる機能です。

## ゴール

次を迷わずできる状態を作る。
- `リッチメニュー` の正しい入口を選ぶ
- 作成方法の分岐を選ぶ
- 画像を確定し、タイトル、フォルダ、表示設定を入れる
- ボタンごとに action を割り当てる
- final `登録`
- テスト用 menu を削除できる

## 必要変数

- メニュー名
- 使う画像
- どの account で出すか
- 誰に見せるか
- 初期表示か切替用か
- ボタン数
- 各ボタンの役割
- 各ボタンの action 種別

## 依頼を受けたら最初に埋める項目

- `タイトル`
- 対象 account
- 誰に見せるか
- `初期表示` か `切替用` か
- 使う画像
- ボタン数
- 各ボタンの役割
- 各ボタンの action 種別
  - `URL / アクション / 回答フォーム / ユーザーメッセージ / TEL`
- exploratory か、本線用か

## 10秒判断

- `リッチメニュー`
  - 常設ナビが必要
  - 入口を固定したい
- `テンプレート`
  - 単発で message を送りたい
- `回答フォーム`
  - 回収が主目的
- `アクション管理`
  - 押下後の複合処理を reusable にしたい

この 10 秒判断が言えない時は、`新しいリッチメニュー` を押さない。

## 実装前の最小チェック

- `lstep.account_name` が intended な公式LINE account と一致している
- この `リッチメニュー` が誰に見えるか決まっている
- 初期表示か切替用か決まっている
- 各ボタンの役割が重複していない
- ボタンごとの action 種別を先に決めている
- 画像だけ先に作って、役割が未定のまま進めていない

## 入口選択の exact な問い

- 今やりたいのは `常設の入口を置く` ことか
  - なら `リッチメニュー`
- 今やりたいのは `単発 message を送る` ことか
  - それはこの skill ではなく `テンプレート`
- 今やりたいのは `情報を回収する` ことか
  - それはこの skill ではなく `回答フォーム`
- 今やりたいのは `押下後の複合処理を reusable にする` ことか
  - それはこの skill ではなく `アクション管理`

つまり、この skill の入口は `常設ナビが必要か` で決まる。

## 使い分け

`リッチメニュー` は、すべてを直接詰め込む場所ではありません。

基本思想
- `リッチメニュー` で入口を見せる
- 押下後に `テンプレート送信 / 回答フォーム / URL / action`
  へ分業する

## Workflow

1. `リッチメニュー`
2. `新しいリッチメニュー`
3. 作成方法を選ぶ
4. 画像を確定する
5. 1回目の `登録` で editor 状態へ入る
6. 基本情報と各ボタンの action を設定する
7. final `登録`
8. テスト用 menu を削除する

## exact 手順

### create 入口

1. `リッチメニュー`
2. `新しいリッチメニュー`
3. 作成方法を選ぶ

作成方法の current 分岐
- `画像をアップロードして作成`
- `テンプレートをベースに作成`

`画像をアップロードして作成` の exact
- route は `/line/richmenu/new?group=0`
- 最初の画面で見えるのは
  - `画像`
  - `メニュー画像選択`
  - footer の `登録`
  だけ
- `メニュー画像選択` の modal は
  - `登録メディア一覧`
  - `新規アップロード`
  - `アップロード済み`
  - `ファイルを選択する`
  - `決定`
- file 条件
  - `1MBまで`
  - `jpg,png`
  - `2500×1686` または `2500×843`
- upload 後に row を選び `決定` を押すと、最初の `登録` が enabled になる
- 最初の `登録` は final save ではなく editor 状態への遷移

### editor で見える主要ラベル

- `画像`
- `タイトル`
- `フォルダ`
- `トークルームメニュー`
- `メニューの初期状態`
- `プレビュー`
- `テンプレート`
- `コンテンツ設定`
- `一覧へ戻る`
- `登録`

### ボタン action

各ボタンで切り替えられる current 種類
- `URL`
- `TEL`
- `ユーザーメッセージ`
- `アクション`
- `回答フォーム`
- `その他`

必要に応じて `アクション設定` を開き、テンプレート送信や action を割り当てる。

### テンプレート選択時の注意

- current の先頭テンプレートを選ぶと
  - `ボタン1URL`
  - `ボタン2URL`
- `ボタン3URL`
  がすべて必須
- URL が不足すると 422 になる

## representative pattern

### 入口分岐型

- 目的
  - `今この人はどこへ進むべきか` を 1 画面で分岐させる
- よく使う action
  - `URL`
  - `回答フォーム`
  - `アクション`
- 向いている場面
  - サポート導線
  - 予約導線
  - イベント一覧への分岐

### テンプレート送信型

- 目的
  - 押した直後に説明や案内を出す
- よく使う action
  - `アクション`
  - `テンプレート送信`
- 向いている場面
  - 使い方案内
  - 参加前案内
  - メニューから詳細説明を出したい時

### 回答回収型

- 目的
  - ボタン押下後すぐに入力を取りたい
- よく使う action
  - `回答フォーム`
- 向いている場面
  - 予約前確認
  - 申請導線

## 最小 live test

最初の検証は `1 画像 + 1 ボタン` で閉じる。

1. `リッチメニュー`
2. `新しいリッチメニュー`
3. `画像をアップロードして作成`
4. 画像を 1 枚確定
5. 1 回目の `登録`
6. `タイトル`
7. ボタンを 1 箇所だけ設定
8. final `登録`
9. preview で 1 ボタンの挙動を確認
10. テスト用なら `削除`

## 最小 downstream smoke

リッチメニューは登録できたかだけで終わらせない。

1. intended な `タイトル`
2. intended な画像
3. intended なボタン位置
4. intended な action 種別
5. intended な遷移先や次メッセージ

1 ボタンでも intended な downstream に着かない時は、他ボタンを増やさない。

最初から 3 ボタン以上や、`テンプレート送信 / 回答フォーム / URL / アクション` の混在を作らない。まず 1 ボタンで疎通を取る。

## 最小 live change

既存 current を壊さずに exact 性を上げる時は、最初の 1 回を `1 menu 1変更` で閉じる。

最初に触る候補
- `タイトル`
- 1 ボタンだけの action 種別
- 1 ボタンだけの遷移先

順番
1. row を 1 つに固定する
2. 触るボタンを 1 箇所だけ決める
3. 変更点を 1 つだけ入れる
4. final `登録`
5. preview または intended な確認導線で 1 ボタンだけ押す
6. 他ボタンに副作用がないか見る
7. exploratory なら削除する

最初からやらないこと
- 複数ボタンの責務を同時に変える
- 画像差し替えと action 差し替えを同時にやる
- `初期表示` と `切替用` の両方を一度にいじる

## 複数ボタンを足す順

複雑化は、次の順で 1 つずつ足す。

1. `URL`
2. `回答フォーム`
3. `アクション`
4. `ユーザーメッセージ` または `その他`

最初に `URL` で疎通を取り、次に `回答回収`、その後に `テンプレート送信や状態変更を含む アクション` を足す。`入口 / 回収 / 制御` の順で増やすと、どのボタンが事故源かを切りやすい。

## 最初の複雑パターン

最初に live で詰める複雑パターンは、`2ボタン mixed pattern` だけで閉じる。

1. ボタン1
   - `URL`
2. ボタン2
   - `回答フォーム`

順番
1. `リッチメニュー`
2. `新しいリッチメニュー`
3. `画像をアップロードして作成`
4. 画像を 1 枚確定
5. 1 回目の `登録`
6. `タイトル`
7. ボタン1 を `URL`
8. ボタン2 を `回答フォーム`
9. final `登録`
10. preview で 2 ボタンとも intended な downstream か確認
11. exploratory なら `削除`

最初から
- `ユーザーメッセージ`
- `アクション`
- `TEL`
を同時に混在させない。

## 最小 rollback

exploratory に menu を作った時は、一覧から確実に消えるところまで戻す。

1. 一覧へ戻る
2. 対象 row 右端 `more_vert`
3. `削除`
4. `この操作は取り消せません。本当に削除しますか？`
5. `削除する`
6. 一覧で row が消えたことを確認

既存 current の 1 ボタン変更なら、まず同じボタンを元の action に戻す。row ごと削除するのは exploratory に限る。

## 変更後の最小 smoke

`登録` が通っただけで閉じない。

1. intended な `タイトル`
2. intended な画像
3. touched button の位置
4. intended な action 種別
5. intended な遷移先か次メッセージ
6. 他ボタンに副作用がないか

1 ボタンの意図を 1 文で説明できない時は、複数ボタンを増やさない。

## representative pattern を読む時の問い

- この `リッチメニュー` は、誰に見せる何の入口か
- `初期表示` なのか、特定状態向けの切替用なのか
- 各ボタンは `URL / 回答フォーム / アクション / テンプレート送信` のどれに渡すべきか
- メニュー自体で説明しすぎず、次のコンテンツへ適切に分業できているか
- ボタンごとの役割が重複していないか
- current の良い例と比べて、1画面に責務を載せすぎていないか
- 画像の見た目ではなく、押した後の挙動まで 1 本ずつ説明できるか
- `テンプレートをベースに作成` か `画像をアップロードして作成` かの選択理由があるか

## ベストな活用場面

- LINE トーク下部で、次の行動への入口を固定したい時
- `テンプレート送信 / 回答フォーム / URL / アクション` への分岐をまとめたい時
- 受講後サポートや予約系で、よく使う動線を迷わず押せるようにしたい時

## よくあるエラー

- `リッチメニュー` 自体に説明を詰め込みすぎる
- 各ボタンの役割が重複して、どこを押せばいいか分からなくなる
- 1回目の `登録` を final save だと誤認する
- テンプレート起点で required URL が足りず 422 になる

## エラー時の切り分け順

1. その menu が `初期表示 / 切替用` のどちらかを切る
2. `テンプレートをベースに作成` か `画像をアップロードして作成` かを切る
3. 各ボタンの役割を `URL / 回答フォーム / アクション / テンプレート送信` で切る
4. 1回目の `登録` 後に editor 状態へ入れているか確認する
5. final `登録` 後に一覧と intended な挙動を確認する

## 複雑パターンで先に切ること

- 初期表示か切替用か
- 入口だけ見せるか、押下後に説明まで出すか
- `リッチメニュー` で完結させるか、次の template / form / URL に渡すか
- 1画面に何個まで役割を載せるか
- `テンプレート送信 / 回答フォーム / URL / アクション` のどれが主役か
- 画像を先に作るのか、接続先を先に確定するのか
- `テンプレートをベースに作成` で required URL を埋めきれるか

## ボタン責務の切り方

- `URL`
  - 外部ページや short.io へそのまま送る
- `回答フォーム`
  - 情報回収が主目的
- `アクション`
  - 押下後に複数処理やテンプレート送信を起こしたい
- `テンプレート送信`
  - 押下直後に説明を出し、次の入口へつなぎたい

1ボタン1責務を原則にする。`URL も説明も回収も同時に持たせる` ような設計にしない。

### 削除

1. 一覧 row 右端 `more_vert`
2. `削除`
3. `この操作は取り消せません。本当に削除しますか？`
4. `削除する`

## 検証

最低でも次を確認する。
- 画像が正しい
- ボタンの役割が重複していない
- どのボタンを押すべきか一読で分かる
- final `登録` が通る
- 保存前に current representative と比べて責務過多になっていない
- テスト用 menu を削除できる

## 保存前の最小チェック

- 目的、対象、次行動を 1 文で言える
- 入力値や action の実体が確定している
- current の row やフォルダを取り違えていない
- テスト後に残すか削除するか決めている

## 保存後の最小チェック

- 一覧に想定どおり表示される
- 必要なら `テスト送信` や preview で実機確認した
- action や遷移先が意図どおり動く
- テスト用なら削除して row が消えたことを確認した

## 押下後の確認順

1. どのボタンを押したか
2. intended な `URL / 回答フォーム / アクション / テンプレート送信` が起きたか
3. 押した後の next content や next action が intended か
4. 別ボタンの責務と重複していないか

`押したら何か起きた` では完了にしない。各ボタンが `誰を / どこへ / 何のために` 渡しているかを 1 本ずつ確認する。

## 症状から最初に疑う場所

- 画像は正しいのに押した先が違う
  - まずそのボタンの action 種別
  - 次に intended な `URL / 回答フォーム / アクション / テンプレート送信`
  - 最後に押下後 downstream
- どのボタンを押せばいいか分かりづらい
  - ボタン責務の重複を先に疑う
  - `初期表示 / 切替用` の目的と一致しているかを見直す
- final `登録` は通るのに current で使いにくい
  - メニュー自体に説明を載せすぎていないか
  - 次コンテンツへの分業が弱くないかを疑う
- `テンプレートをベースに作成` で保存できない
  - `ボタン1URL`
  - `ボタン2URL`
  - `ボタン3URL`
  の required 項目不足を先に疑う
- 切替用メニューを作ったが、初期表示との違いが弱い
  - `何の状態の人にだけ見せるか`
  - `何を引き算したか`
  の 2 点を先に見直す

## 切替用メニューで先に見ること

- 何の状態の人にだけ見せる切替か
- 初期表示メニューから何を引き算するか
- `テンプレート送信 / 回答フォーム / URL / アクション` の配分が切替理由と一致するか

切替用で初期表示と同じ責務を並べるなら、メニューを分ける意味が弱い。まず切替理由を 1 文にする。

## 構築精度だけを見る時のチェック

1. この `リッチメニュー` が `初期表示 / 切替用` のどちらかで 1 文になっているか
2. `テンプレートをベースに作成` と `画像をアップロードして作成` の選択理由があるか
3. 各ボタンが `URL / 回答フォーム / アクション / テンプレート送信` のどれに渡すか整理できているか
4. 1画面に責務を載せすぎていないか
5. `画像` と action の組み合わせを 1 ボタンずつ説明できるか
6. final `登録` 後に一覧と preview で確認したか
7. テスト用 menu を row が消えるまで削除確認したか
8. `テンプレートをベースに作成` の required URL を全部説明できるか
9. 画像を先に作った場合でも、接続先の主従が崩れていないか

## ここで止めて確認する条件

- 何を変える設定か 1 文で言えない
- current / legacy の切り分けに迷う
- 新しい外部連携や本番影響の大きい分岐を足す
- 既存 high-quality な実例と大きく異なる構成にしたい
- `テンプレート送信 / 回答フォーム / URL / アクション` の主役が決まっていない
- `テンプレートをベースに作成` なのに required URL を埋め切れない
- 最初の検証なのに 1 ボタンで閉じず、複数 action を同時に足そうとしている

## 完成条件

- 正しい入口から作成または編集できている
- 保存後に一覧と実挙動で確認できている
- テスト送信が必要な場合は `甲原 海人` で確認している
- テスト用の残骸が残っていない
- menu を押した先の next content / next action を 1 ボタンずつ説明できる

## NG

- `リッチメニュー` 自体に全部の説明を詰め込む
- current / old / 作成中 を名前だけで判定する
- 画像だけ入れて action を確認しない
- すべてのボタンを同じ役割にする
- 1回目の `登録` を final save だと誤認する
- テンプレートごとの required URL 数を確認せずに進める

## 正誤判断

正しい状態
- 入口としての役割が明確
- ボタンごとの役割が分かれている
- `テンプレート送信 / 回答フォーム / URL / action` の分業ができている
- current representative と比べても責務が自然
- 不要なテスト menu が残っていない

間違った状態
- 説明も行動も全部リッチメニュー自体に押し込む
- current / legacy の混在を読まずに古い menu を編集する
- ボタン action を設定せず保存する
