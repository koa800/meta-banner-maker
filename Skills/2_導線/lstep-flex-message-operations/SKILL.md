# lstep-flex-message-operations

## Overview

Lステップの `フレックスメッセージ` を、作成して終わりではなく `保存 -> テスト送信 -> 実挙動確認 -> 削除` まで一気通貫で扱う skill です。

Addness 固有の account 選定、命名規則、テスト送信先は `Master/knowledge/lstep_structure.md` を正本にする。この skill では、どの案件でも再利用できる exact 手順だけを持つ。

## 役割

`フレックスメッセージ` は、視覚階層を強く作り、重要告知や main CTA を 1 画面で見落とされにくく伝えるための message type です。

## ゴール

次を迷わずできる状態を作る。
- `テンプレート -> 新しいテンプレート -> フレックスメッセージ` の正しい入口を選ぶ
- `画像 / タイトル / テキスト / ボタン / アクション` を必要最小限で組む
- 一覧行右端 `... -> テスト送信` で実機確認する
- テスト用なら `... -> 削除` まで後片付けする
- 既存の良い `フレックスメッセージ` を、何を認識させて何を押させる構造かまで分解できる

## 必要変数

- テンプレート名
- 何を認識させたいか
- 何を行動させたいか
- 画像の有無
- タイトルの有無
- テキストの有無
- ボタンの有無
- ボタンのアクション種別
- テスト送信先
- テスト後に残すか削除するか

## 使い分け

先に message 種を切る。
- `標準メッセージ`
  - 文字説明が主役
  - simple な CTA で足りる
- `カルーセルメッセージ(新方式)`
  - 比較や選択肢を並べたい
- `フレックスメッセージ`
  - 重要告知
  - main CTA
  - 視覚階層を強く出したい
- `テンプレートパック`
  - message 群を container として持ちたい

`フレックスメッセージ` を選ぶのは、
- 1画面で重要度の差を見せたい
- テキストだけだと見落とされやすい
- ボタン action を明確に 1つか数個に絞りたい
時です。

## Workflow

1. `テンプレート`
2. `新しいテンプレート`
3. `フレックスメッセージ`
4. `テンプレート名`
5. 必要なら `PC版・通知欄での代替テキスト`
6. `edit メッセージを作成`
7. editor 内で必要な block を追加
8. `メッセージを保存`
9. route が `/line/template/edit_v3/{id}?editMessage=0` に切り替わる
10. 外側の `保存`
11. 一覧に戻る
12. 行右端 `... -> テスト送信`
13. 送信先を選ぶ
14. 実機で見た目とタップ先を確認
15. テスト用なら `... -> 削除`

## 先に決めること

UI を触る前に、最低でも次を決める。
1. 何の認識を変えたいか
2. 次に 1 つだけ何を行動させたいか
3. `単発CTA型 / 2CTA案内型 / 状態更新連動型 / 重要告知型` のどれに近いか
4. 画像が本当に必要か
5. ボタンが 1 つで足りるか

## Editor の exact 手順

### block の追加

`＋ブロックを追加` は helper text と button の 2 箇所に見えることがある。押すのは button 側です。

### 画像

`画像`
1. `＋ブロックを追加 -> 画像`
2. inline block の `なし` を押す
3. modal `画像を追加`
4. modal 内で `新規アップロード` か `登録メディア`
5. file を入れるだけで終わらない
6. `登録メディア` では card 全体ではなく filename 行を押して `選択中のファイル数: 1 / 1`
7. modal の `保存`

### テキスト / タイトル

入力は通常 input ではなく contenteditable 系になることがある。表示されている入力枠にそのまま入れる前提で考える。

### ボタン

`ボタン`
1. `アクション設定`
2. `種類`
3. 種別を選ぶ
4. 種別ごとの必要値を入れる
5. `保存する`

current で確認できている主な種類。
- `URLを開く`
- `回答フォームを開く`
- `電話をかける`
- `LINEアカウントを友だち追加する`
- `メールを送る`
- `その他`

注意
- UI 上のラベルが `Linyアクション` でも、実体が必ず URL とは限らない
- current では少なくとも
  - `URLを開く`
  - `タグを追加する`
  - `テキストを送信する`
  の複合を `action_description` で持つ例がある
- したがって representative を読む時は、`見た目の種類` ではなく `action_description` まで確認する

## representative の exact 分類

### 単発CTA型

- 例
  - `3月 スキルプラス合宿 集客`
  - `2月 スキルプラス合宿 集客`
- 構成
  - `画像 1`
  - `テキスト 3-4`
  - `ボタン 1`
- 読む順
  - 重要告知
  - 補足
  - urgency
  - main CTA

### 2CTA案内型

- 例
  - `20250228 コース移行のお知らせ`
- 構成
  - `タイトル 1`
  - `テキスト 3`
  - `ボタン 2`
- 読む順
  - 何が起きているか
  - 何をすべきか
  - `登録`
  - `ガイド確認`

### 状態更新連動型

- 例
  - `対象者に告知（1on1予約→実施でアマギフ1,000円）`
  - `ゴール設定1on1予約案内【2026/01/12：アマギフ訴求】`
- 構成
  - `画像 1`
  - `テキスト 1`
  - `ボタン 1`
- 読む順
  - 誰向けか
  - 何が得か
  - CTA
  - `action_description`

### 重要告知型

- 例
  - `2月16日スキルプラス 大幅アップデートのお知らせ`
  - `スキルプラス 大幅アップデートのお知らせ`
  - `SL.AIアップデートのお知らせ`
- 構成
  - `動画 + 画像 + テキスト`
  - または `動画 + タイトル + テキスト`
- 注意
  - CTA を置かないこと自体が正しい場合がある
  - 認識合わせを優先する

### 運用変更告知型

- 例
  - `日報提出方法変更のお知らせ`
- 構成
  - `動画 1`
  - `タイトル 1`
  - `テキスト 4`
- 読む順
  - 何が変わるか
  - 旧運用と新運用の差
  - どこで実行するか
  - 締切や開始日
- 注意
  - すぐ行動させるより、運用ルールを誤読なく伝えることを優先する
  - CTA が不要な時は置かない

### 専用窓口誘導型

- 例
  - `12/14 スキルプラス合宿 予約専用LINE誘導１`
- 構成
  - `画像 1`
  - `テキスト 3`
  - `ボタン 1`
- 読む順
  - 何の連絡か
  - なぜ別 LINE なのか
  - 期限
  - 送る先
- 注意
  - `Linyアクション` の実体が URL ではなく `テキスト送信` の場合がある
  - 予約情報の再登録不要など、余計な不安を消す文が入っているかを見る

### 障害告知型

- 例
  - `【障害報告】外部システム障害の影響に関するお知らせ`
- 構成
  - `テキスト 3`
- 読む順
  - 何が起きているか
  - 影響範囲
  - 待機でよいのか
  - 謝辞 / お詫び
- 注意
  - 不要な CTA を足さない
  - 不安軽減と問い合わせ抑制が主目的

## デザインと構造の判断

- 画像は装飾ではなく文脈補強として使う
- ボタンは `何を押すか` を 1 行で言い切る
- 2CTA にするなら、役割が明確に分かれている時だけ
- CTA が 1 つで足りるなら、無理に増やさない
- `動画 + 画像 + テキスト` のような混在は、まず `何を認識させる message か` を切ってから使う

## テスト送信

1. 一覧行右端 `...`
2. `テスト送信`
3. `テスト送信先選択`
4. 送信先行の左端 checkbox label を押して選択
5. `テスト`
6. 実機確認

Addness ではテスト送信先と重複禁止ルールがある。必ず `Master/knowledge/lstep_structure.md` の current ルールに従う。

## 検証

最低でも次を確認する。
- 見た目が崩れていない
- 画像が出る
- テキストが切れていない
- ボタンが押せる
- action が意図どおり動く
- URL 系なら最終遷移先まで正しい

## 後片付け

テスト用なら、
1. 一覧行右端 `...`
2. `削除`
3. confirm dialog `削除する`
4. 一覧再読込

削除したつもりで終わらない。対象行が見えないことを確認する。

## NG

- `標準メッセージ` で十分なのに、なんとなく `フレックスメッセージ` を選ぶ
- block を増やしすぎて main CTA がぼやける
- 画像を upload しただけで保存した気になる
- `テスト送信` をせずに本番へ回す
- テスト用 item を残す
- Addness 固有ルールを skill 内に焼き込む
- `メッセージを保存` だけで終わったと誤認する
- 外側の `保存` を押さずに一覧へ戻る

## 正誤判断

正しい状態
- 正しい入口から入っている
- 画像 / テキスト / ボタン / action の役割が分かれている
- editor 上部 `メッセージを保存` の後に外側の `保存` まで完了している
- 実機で main CTA を押して意図どおり動く
- テスト用は削除済み
- 既存 good 例のどの型に近いか説明できる

間違った状態
- 見た目だけ整えて action が未設定
- 画像とボタンの役割が重複している
- テスト送信先を毎回変える
- current account や current ルールを見ずに流用する
- 2CTA にしたのに役割分担が曖昧
