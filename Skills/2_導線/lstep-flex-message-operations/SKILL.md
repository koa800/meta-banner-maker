# lstep-flex-message-operations

## Overview

Lステップの `フレックスメッセージ` を、作成して終わりではなく `保存 -> テスト送信 -> 実挙動確認 -> 削除` まで一気通貫で扱う skill です。

Addness 固有の account 選定、命名規則、テスト送信先は `Master/knowledge/lstep_structure.md` を正本にする。この skill では、どの案件でも再利用できる exact 手順だけを持つ。

## ツールそのものの役割

`フレックスメッセージ` は、画像、テキスト、ボタンを組み合わせて、視覚階層を強く持つ message type です。

## アドネスでの役割

アドネスでは `フレックスメッセージ` を、重要告知、main CTA、状態別の案内を 1 画面で見落とされにくく伝えるために使う。見た目を作ること自体が目的ではなく、認識と次の行動を 1 つに絞って渡すことを優先する。

## 役割

`フレックスメッセージ` は、視覚階層を強く作り、重要告知や main CTA を 1 画面で見落とされにくく伝えるための message type です。

## ゴール

次を迷わずできる状態を作る。
- `テンプレート -> 新しいテンプレート -> フレックスメッセージ` の正しい入口を選ぶ
- `画像 / タイトル / テキスト / ボタン / アクション` を必要最小限で組む
- 一覧行右端 `...` に出る `コピー / 名前を変更 / テスト送信 / 一斉配信を作成 / 削除` を前提に実機確認する
- テスト用なら `... -> 削除` まで後片付けする
- 既存の良い `フレックスメッセージ` を、何を認識させて何を押させる構造かまで分解できる

## 必要変数

- テンプレート名
- 何を認識させたいか
- 何を行動させたいか
- 画像の有無
- タイトルの有無
- テキストの有無
- ボタンの有無
- ボタンのアクション種別
- テスト送信先
- テスト後に残すか削除するか

## 使い分け

先に message 種を切る。
- `標準メッセージ`
  - 文字説明が主役
  - simple な CTA で足りる
- `カルーセルメッセージ(新方式)`
  - 比較や選択肢を並べたい
- `フレックスメッセージ`
  - 重要告知
  - main CTA
  - 視覚階層を強く出したい
- `テンプレートパック`
  - message 群を container として持ちたい

`フレックスメッセージ` を選ぶのは、
- 1画面で重要度の差を見せたい
- テキストだけだと見落とされやすい
- ボタン action を明確に 1つか数個に絞りたい
時です。

## Workflow

1. `テンプレート`
2. `新しいテンプレート`
3. `フレックスメッセージ`
4. `テンプレート名`
5. 必要なら `PC版・通知欄での代替テキスト`
6. `edit メッセージを作成`
7. editor 内で必要な block を追加
8. `メッセージを保存`
9. route が `/line/template/edit_v3/{id}?editMessage=0` に切り替わる
10. 外側の `保存`
11. 一覧に戻る
12. 行右端 `...`
13. current の候補 `コピー / 名前を変更 / テスト送信 / 一斉配信を作成 / 削除` を確認
14. `テスト送信`
15. 送信先を選ぶ
16. 実機で見た目とタップ先を確認
17. テスト用なら `... -> 削除`

## current の exact route

- 作成開始:
  - `/line/template/edit_v3/new?group=0&editMessage=0`
- `edit メッセージを作成` 後:
  - `/line/template/edit_v3/new?group=0&editMessage=1`
- `メッセージを保存` 後:
  - `/line/template/edit_v3/{id}?editMessage=0`

つまり current では、
- 先に `テンプレート名`
- 次に `edit メッセージを作成`
- その後に message editor
の 2 段階です。

## 先に決めること

UI を触る前に、最低でも次を決める。
1. 何の認識を変えたいか
2. 次に 1 つだけ何を行動させたいか
3. `単発CTA型 / 2CTA案内型 / 状態更新連動型 / 重要告知型` のどれに近いか
4. 画像が本当に必要か
5. ボタンが 1 つで足りるか

## Editor の exact 手順

### block の追加

`＋ブロックを追加` は helper text と button の 2 箇所に見えることがある。押すのは button 側です。

### 画像

`画像`
1. `＋ブロックを追加 -> 画像`
2. inline block の `なし` を押す
3. modal `画像を追加`
4. modal 内で `新規アップロード` か `登録メディア`
5. file を入れるだけで終わらない
6. `登録メディア` では card 全体ではなく filename 行を押して `選択中のファイル数: 1 / 1`
7. modal の `保存`
8. 画像が入った後も、外側の `メッセージを保存` までは確定していない

### テキスト / タイトル

入力は通常 input ではなく contenteditable 系になることがある。表示されている入力枠にそのまま入れる前提で考える。

### ボタン

`ボタン`
1. `アクション設定`
2. `種類`
3. 種別を選ぶ
4. 種別ごとの必要値を入れる
5. `保存する`

current で確認できている主な種類。
- `URLを開く`
- `回答フォームを開く`
- `電話をかける`
- `LINEアカウントを友だち追加する`
- `メールを送る`
- `その他`

注意
- UI 上のラベルが `Linyアクション` でも、実体が必ず URL とは限らない
- current では少なくとも
  - `URLを開く`
  - `タグを追加する`
  - `テキストを送信する`
  の複合を `action_description` で持つ例がある
- したがって representative を読む時は、`見た目の種類` ではなく `action_description` まで確認する

## representative の exact 分類

### 単発CTA型

- 例
  - `3月 スキルプラス合宿 集客`
  - `2月 スキルプラス合宿 集客`
- 構成
  - `画像 1`
  - `テキスト 3-4`
  - `ボタン 1`
- 読む順
  - 重要告知
  - 補足
  - urgency
  - main CTA

### 2CTA案内型

- 例
  - `20250228 コース移行のお知らせ`
- 構成
  - `タイトル 1`
  - `テキスト 3`
  - `ボタン 2`
- 読む順
  - 何が起きているか
  - 何をすべきか
  - `登録`
  - `ガイド確認`

### 状態更新連動型

- 例
  - `対象者に告知（1on1予約→実施でアマギフ1,000円）`
  - `ゴール設定1on1予約案内【2026/01/12：アマギフ訴求】`
- 構成
  - `画像 1`
  - `テキスト 1`
  - `ボタン 1`
- 読む順
  - 誰向けか
  - 何が得か
  - CTA
  - `action_description`

### 重要告知型

- 例
  - `2月16日スキルプラス 大幅アップデートのお知らせ`
  - `スキルプラス 大幅アップデートのお知らせ`
  - `SL.AIアップデートのお知らせ`
- 構成
  - `動画 + 画像 + テキスト`
  - または `動画 + タイトル + テキスト`
- 注意
  - CTA を置かないこと自体が正しい場合がある
  - 認識合わせを優先する

### 運用変更告知型

- 例
  - `日報提出方法変更のお知らせ`
- 構成
  - `動画 1`
  - `タイトル 1`
  - `テキスト 4`
- 読む順
  - 何が変わるか
  - 旧運用と新運用の差
  - どこで実行するか
  - 締切や開始日
- 注意
  - すぐ行動させるより、運用ルールを誤読なく伝えることを優先する
  - CTA が不要な時は置かない

### 専用窓口誘導型

- 例
  - `12/14 スキルプラス合宿 予約専用LINE誘導１`
- 構成
  - `画像 1`
  - `テキスト 3`
  - `ボタン 1`
- 読む順
  - 何の連絡か
  - なぜ別 LINE なのか
  - 期限
  - 送る先
- 注意
  - `Linyアクション` の実体が URL ではなく `テキスト送信` の場合がある
  - 予約情報の再登録不要など、余計な不安を消す文が入っているかを見る

### 障害告知型

- 例
  - `【障害報告】外部システム障害の影響に関するお知らせ`
- 構成
  - `テキスト 3`
- 読む順
  - 何が起きているか
  - 影響範囲
  - 待機でよいのか
  - 謝辞 / お詫び
- 注意
  - 不要な CTA を足さない
  - 不安軽減と問い合わせ抑制が主目的

## ベストな活用場面

- 重要告知を 1 画面で見落とされにくく届けたい時
- `標準メッセージ` だと CTA が弱く、視覚階層を足したい時
- 画像、短いテキスト、1 つの main CTA で十分な時
- 状態別の案内を、`画像 + 補足 + ボタン` で friction なく渡したい時

## よくあるエラー

- `メッセージを保存` だけ押して、外側の `保存` を押し忘れる
- 画像を upload しただけで、modal の `保存` を押していない
- `URLを開く` だと思って設定した action が、実際は別種別になっている
- テスト送信前に main CTA の final destination を確認していない
- `標準メッセージ` で足りる内容なのに、要素を盛りすぎて `フレックスメッセージ` を使う

## エラー時の切り分け順

1. 一覧に対象 row が存在するか
2. row 右端 `...` から `テスト送信` が開けるか
3. editor 上部の `メッセージを保存` と外側 `保存` の両方を押したか
4. ボタンの `アクション設定` で `種類` と設定値が intended か
5. 実機で押した時の final destination が intended か

## デザインと構造の判断

- 画像は装飾ではなく文脈補強として使う
- ボタンは `何を押すか` を 1 行で言い切る
- 2CTA にするなら、役割が明確に分かれている時だけ
- CTA が 1 つで足りるなら、無理に増やさない
- `動画 + 画像 + テキスト` のような混在は、まず `何を認識させる message か` を切ってから使う

## quality を読む時の前提

- current の representative は、まず `構築方法` と `action 設計` の benchmark として読む
- そのまま `見た目や訴求の品質上限` とみなさない
- Addness の current には、構築精度は高いが content quality は 60 点台のものも混ざる前提で扱う
- したがって review では
  - `正しく組めているか`
  - `良い content になっているか`
  を分けて判定する

## テスト送信

1. 一覧行右端 `...`
2. `テスト送信`
3. `テスト送信先選択`
4. 送信先行の左端 checkbox label を押して選択
5. `テスト`
6. 実機確認

Addness ではテスト送信先と重複禁止ルールがある。必ず `Master/knowledge/lstep_structure.md` の current ルールに従う。

## 完成条件

次を全部満たした時だけ完成扱いにする。
- `テンプレート -> 新しいテンプレート -> フレックスメッセージ` の正しい入口から入っている
- `画像 / タイトル / テキスト / ボタン / アクション` の各要素に役割がある
- `メッセージを保存` の後に外側の `保存` まで終わっている
- 一覧行右端 `... -> テスト送信` で `甲原 海人` に送っている
- 実機で `見た目 / タップ先 / 認識させたいこと / 行動させたいこと` を確認している
- テスト用なら `... -> 削除` まで終わっている

## current で迷いやすい差分

- `画像` は `＋ブロックを追加 -> 画像` だけで完了しない
  - inline block の `なし` を押して modal を開く
- `画像を追加` modal は `登録メディア` で card 全体ではなく filename 行を押して選択状態にする
- `Linyアクション` は UI 上の見え方であって、実体が `URLを開く` とは限らない
- `メッセージを保存` と外側 `保存` は別
- `テンプレートパック` の step 内 message と、単体 `フレックスメッセージ` は入口も保存単位も違う

## 削除

1. 一覧行右端 `...`
2. `削除`
3. confirm modal `削除する`
4. 一覧再読込後に行が消えていることを確認

## 検証

最低でも次を確認する。
- 見た目が崩れていない
- 画像が出る
- テキストが切れていない
- ボタンが押せる
- action が意図どおり動く
- URL 系なら最終遷移先まで正しい

## 後片付け

テスト用なら、
1. 一覧行右端 `...`
2. `削除`
3. confirm dialog `削除する`
4. 一覧再読込

削除したつもりで終わらない。対象行が見えないことを確認する。

## 実装前の最小チェック

- 何を実装する画面か 1 文で言える
- 現在触るべき account と current / legacy を切っている
- テスト送信や削除の要否を先に決めている
- 次の行動や状態変化を 1 つに絞っている

## exact 手順

- まず UI 上の正しい入口を開く
- visible ラベルに従って入力する
- 保存後は一覧で row の存在を確認する
- `テスト送信` がある機能は `甲原 海人` で実機確認する
- テスト用は一覧から削除し、残っていないことまで確認する

## 保存前の最小チェック

- 目的、対象、次行動を 1 文で言える
- 入力値や action の実体が確定している
- current の row やフォルダを取り違えていない
- テスト後に残すか削除するか決めている

## 保存後の最小チェック

- 一覧に想定どおり表示される
- 必要なら `テスト送信` や preview で実機確認した
- action や遷移先が意図どおり動く
- テスト用なら削除して row が消えたことを確認した

## ここで止めて確認する条件

- 何を変える設定か 1 文で言えない
- current / legacy の切り分けに迷う
- 新しい外部連携や本番影響の大きい分岐を足す
- 既存 high-quality な実例と大きく異なる構成にしたい

## NG

- `標準メッセージ` で十分なのに、なんとなく `フレックスメッセージ` を選ぶ
- block を増やしすぎて main CTA がぼやける
- 画像を upload しただけで保存した気になる
- `テスト送信` をせずに本番へ回す
- テスト用 item を残す
- Addness 固有ルールを skill 内に焼き込む
- `メッセージを保存` だけで終わったと誤認する
- 外側の `保存` を押さずに一覧へ戻る

## 正誤判断

正しい状態
- 正しい入口から入っている
- 画像 / テキスト / ボタン / action の役割が分かれている
- editor 上部 `メッセージを保存` の後に外側の `保存` まで完了している
- 実機で main CTA を押して意図どおり動く
- テスト用は削除済み
- 既存 good 例のどの型に近いか説明できる

間違った状態
- 見た目だけ整えて action が未設定
- 画像とボタンの役割が重複している
- テスト送信先を毎回変える
- current account や current ルールを見ずに流用する
- 2CTA にしたのに役割分担が曖昧
