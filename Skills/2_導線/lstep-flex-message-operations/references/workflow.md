# Lステップ フレックスメッセージ exact メモ

## current 入口

- `テンプレート`
- `新しいテンプレート`
- `フレックスメッセージ`

代表 route
- 新規開始: `/line/template/edit_v3/new?group=0&editMessage=0`
- editor 展開後: `/line/template/edit_v3/new?group=0&editMessage=1`
- 保存後: `/line/template/edit_v3/{id}?editMessage=0`

## current で見えている主要ラベル

- `テンプレート名`
- `PC版・通知欄での代替テキスト`
- `edit メッセージを作成`
- `＋ブロックを追加`
- `メッセージを保存`
- 外側の `保存`
- `テンプレート一覧へ戻る`
- `edit メッセージを編集`

## block 追加候補

- `画像`
- `タイトル`
- `テキスト`
- `ボタン`
- `動画`

## 画像 block の current ラベル

- `画像`
- `なし`
- `画像を追加`
- `新規アップロード`
- `登録メディア`
- `ここにファイルをドロップ`
- `ファイルを選択する`
- `閉じる`
- `保存`

重要
- `＋ブロックを追加 -> 画像` の直後に modal は開かない
- 先に inline block の `なし` を押して modal `画像を追加` を開く
- upload だけでは不十分
- `登録メディア` では card 全体ではなく filename 行を押して `選択中のファイル数: 1 / 1` にしてから保存する

## ボタン block の current ラベル

- `アクション設定`
- `種類`
- `閉じる`
- `保存する`

確認済みの action 種類
- `URLを開く`
- `回答フォームを開く`
- `電話をかける`
- `LINEアカウントを友だち追加する`
- `メールを送る`
- `その他`

重要
- UI 上のラベルが `Linyアクション` でも、実体が必ず URL とは限らない
- representative を読む時は `action_description` まで見る
- current では少なくとも
  - `URLを開く`
  - `タグを追加する`
  - `テキストを送信する`
  の複合例を確認済み

## representative の読み方

### 単発 CTA 型

- 例: `3月 スキルプラス合宿 集客`
- 例: `2月 スキルプラス合宿 集客`
- 構成
  - 画像 1
  - テキスト 3-4
  - ボタン 1
- 読み方
  - `重要告知 -> 補足 -> main CTA`
- 実値
  - alt text: `【3月スキルプラス合宿】のご案内`
  - button action: `URLを開く`
  - URL: `https://liff.line.me/2008652187-VzGy7821/landing?follow=%40909refvl&lp=HBTqth&liff_id=2008652187-VzGy7821`

### 2CTA案内型

- 例: `20250228 コース移行のお知らせ`
- 構成
  - タイトル 1
  - テキスト 3
  - ボタン 2
- 読み方
  - `何が起きているか -> 何をすべきか -> 登録 -> ガイド確認`
- 注意
  - 2つの button の役割が分かれていないなら bad

### segmentation 型

- 例: `選べる3種のPR`
- 構成
  - 画像 4
  - テキスト 4
  - 3 panel
- 読み方
  - `どれを売るか` より `どの入口を選ばせるか`

### 状態更新連動型

- 例: `対象者に告知（1on1予約→実施でアマギフ1,000円）`
- 例: `ゴール設定1on1予約案内【2026/01/12：アマギフ訴求】`
- 構成
  - 画像 1
  - テキスト 1
  - ボタン 1
- 読み方
  - `誰向けか -> 何が得か -> CTA -> action_description`
- 注意
  - `Linyアクション` の中身を URL と決めつけない
- 実値
  - alt text: `[姓]さんへ本日限定のお知らせ`
  - button action: `タグ[アマギフ_オプト]を追加` + `テキスト[こちらから...]を送信`

### 重要告知型

- 例: `2月16日スキルプラス 大幅アップデートのお知らせ`
- 例: `スキルプラス 大幅アップデートのお知らせ`
- 例: `SL.AIアップデートのお知らせ`
- 構成
  - `動画 + 画像 + テキスト`
  - または `動画 + タイトル + テキスト`
- 読み方
  - `何が変わったか -> なぜ変わったか -> 自分にどう影響するか`
- 注意
  - CTA を置かないこと自体が正しい場合がある
- 実値
  - `2月16日スキルプラス 大幅アップデートのお知らせ`
  - `video 1 / image 1 / text 4`
  - `何が変わるか -> 使いやすさの改善 -> メンテナンス日時` の順

### 運用変更告知型

- 例: `日報提出方法変更のお知らせ`
- 構成
  - `動画 1`
  - `タイトル 1`
  - `テキスト 4`
- 読み方
  - `何が変わるか -> 旧運用と新運用の差 -> 実行場所 -> 開始日`
- 注意
  - リッチメニュー変更や提出方法変更のような、運用ルール差分を誤読なく伝える
  - CTA を足すより運用理解を優先する

### 専用窓口誘導型

- 例: `12/14 スキルプラス合宿 予約専用LINE誘導１`
- 構成
  - `画像 1`
  - `テキスト 3`
  - `ボタン 1`
- 読み方
  - `何の連絡か -> なぜ別 LINE なのか -> 期限 -> 送る先`
- 注意
  - `Linyアクション` の実体が URL ではなく `テキスト送信` の場合がある
  - 予約情報の再登録不要など、誤行動を防ぐ説明が入っているかを見る

### 障害告知型

- 例: `【障害報告】外部システム障害の影響に関するお知らせ`
- 構成
  - `テキスト 3`
- 読み方
  - `何が起きているか -> 影響範囲 -> 待機でよいか -> お詫び`
- 注意
  - 不安軽減と問い合わせ抑制が主目的
  - 不要な CTA は置かない

## 30秒レビューの順番

1. この `フレックスメッセージ` の目的を 1 文で言う
2. `単発CTA / 2CTA案内 / segmentation / 状態更新連動 / 重要告知 / 運用変更告知 / 専用窓口誘導 / 障害告知` のどれに近いか切る
3. `画像 / タイトル / テキスト / ボタン / 動画` の役割を分ける
4. main CTA がどれか決める
5. `テスト送信` で実機の見え方と挙動を確認する

## Addness 側で見るべき補足

- どの account で作るか
- テスト送信先
- current / legacy の判定
- 命名規則

これらは `Master/knowledge/lstep_structure.md` を正本にする

## 完成条件

- `画像 / タイトル / テキスト / ボタン / 動画` のうち必要 block だけを使えている
- `action_description` を含めて button action の意味を説明できる
- `メッセージを保存` と外側の `保存` を両方通している
- `テスト送信` で `甲原 海人` 宛ての実機確認をしている
- テスト用なら削除まで終えている

## ここで止めて確認する条件

- 何を変えたいメッセージか 1 文で言えない
- button が複数あるのに main CTA が決まっていない
- `Linyアクション` の中身を確認せず URL だと決めつけている
- 画像量は多いが、読み順と行動順が一致していない

## 保存前の最小チェック

- 目的と次行動を 1 文で言える
- 入力値や action の実体が確定している
- テスト後に残すか削除するか決めている

## 保存後の最小チェック

- 一覧と実挙動の両方で確認した
- `テスト送信` があるものは `甲原 海人` で確認した
- テスト用は削除して残っていない
