# lstep-landing-operations

## Overview

Lステップの `流入経路分析` を、`新しい流入経路` の作成だけでなく `URL / QR の確認 -> 削除` まで exact に再現する skill です。

Addness 固有の命名規則、どの account で作るか、current representative は `Master/knowledge/lstep_structure.md` を正本にする。この skill は、会社をまたいで再利用できる exact 手順だけを持つ。

## ツールそのものの役割

`流入経路分析` は、友だち追加の入口ごとに URL、QR、初期 action、計測単位を持つ entry management 機能です。

## アドネスでの役割

アドネスでは `流入経路分析` を、どこから来たかを切る入口であり、同時に最初のタグや action を起動する起点として使う。集客と state 変換の最初の接点です。

## 役割

`流入経路分析` は、どこから友だち追加されたかを追跡しつつ、登録直後の action を起動する landing 機能です。

## ゴール

次を迷わずできる状態を作る。
- `流入経路分析` の正しい入口を選ぶ
- `新しい流入経路` を作る
- URL と QR を確認する
- 登録時 action の有無を判断する
- テスト用なら削除する

## 必要変数

- 流入経路名
- QRコード表示用テキスト
- どこから来たか
- action を付けるかどうか
- `友だち追加時設定のアクション` を無視するかどうか
- action の実行タイミング
- URL と QR の保管先

## 命名ルール

- 基本形は `集客媒体_ファネル名_イベント名`
- SNS は先頭に `（アカウント名）` を付ける
  - 例: `（スキルプラス公式TV）YouTube _スキルプラス_直リストイン`
- `直リストイン`
  - 直接 LINE 登録したもの
- `リストイン`
  - メール登録後に LINE 登録したもの
- 命名は `既存 family に似せる` から始めない
- 先に
  - `流入前オファー`
  - `流入時の状態`
  - `開始シナリオ`
  を固定して、その後で名前を決める

## Workflow

1. `流入経路分析`
2. `新しい流入経路`
3. 基本情報を入れる
4. 必要なら action を設定する
5. `登録`
6. 一覧で `URLコピー / QR / 広告挿入タグ` を確認する
7. テスト用なら削除する

## 実案件で一番ズレにくい順

実案件では、次の順が最もズレにくい。

1. Lステップで `流入経路分析` を作る
2. 実際の `流入URL` を確定する
3. short.io を作る
4. URL管理シートを更新する
5. 必要な相手に共有する

先に short.io を作らない。`流入経路名 / 開始シナリオ / 実流入URL` が固まる前に short.io を作ると、タイトル、台帳、共有文面がずれやすい。

## exact 手順

### create

1. `流入経路分析`
2. `新しい流入経路`
3. create 画面で次を入れる
   - `流入経路名`
   - `QRコード表示用テキスト`
   - 必要なら `有効期間(開始)` / `有効期間(終了)`
   - 必要なら `アクション`
4. `友だち追加時設定のアクション`
5. `アクションの実行`
6. `登録`

### action 未設定時

action を入れないまま `登録` を押すと確認 dialog が出る。

- message
  - `アクションが設定されていません。本当に登録しますか？`
- button
  - `キャンセル / OK`

### 作成後の確認

一覧では少なくとも次を確認する。
- `URLコピー`
- `QR`
- `広告挿入タグ`

QR を使う施策では、URL と QR を両方控える。

### 削除

1. 一覧 row 右端 `more_vert`
2. `削除`
3. 一覧再読込後に row が消えていることを確認する

## 検証

最低でも次を確認する。
- 流入経路名が正しい
- 必要なら action が入っている
- URL が発行されている
- QR が開ける
- テスト用なら削除済み

## 実装前の最小チェック

- 何を実装する画面か 1 文で言える
- 現在触るべき account と current / legacy を切っている
- テスト送信や削除の要否を先に決めている
- 次の行動や状態変化を 1 つに絞っている

## 保存前の最小チェック

- 目的、対象、次行動を 1 文で言える
- 入力値や action の実体が確定している
- current の row やフォルダを取り違えていない
- テスト後に残すか削除するか決めている

## 保存後の最小チェック

- 一覧に想定どおり表示される
- 必要なら `テスト送信` や preview で実機確認した
- action や遷移先が意図どおり動く
- テスト用なら削除して row が消えたことを確認した

## ここで止めて確認する条件

- 何を変える設定か 1 文で言えない
- current / legacy の切り分けに迷う
- 新しい外部連携や本番影響の大きい分岐を足す
- 既存 high-quality な実例と大きく異なる構成にしたい

## 完成条件

- 正しい入口から作成または編集できている
- 保存後に一覧と実挙動で確認できている
- テスト送信が必要な場合は `甲原 海人` で確認している
- テスト用の残骸が残っていない

## NG

- 何を分けて取りたいか決めずに作る
- URL だけ控えて QR を見ない
- action 未設定 dialog を見落とす
- `友だち追加時設定のアクション` の挙動を確認しない
- テスト用 landing を残す

## 正誤判断

正しい状態
- 何を分けて計測したい landing か説明できる
- 必要な action だけ入っている
- URL と QR の両方を確認している
- テスト用は削除済み

間違った状態
- 何のための landing か曖昧
- URL だけ作って action や QR を見ない
- test 作成後に row を残す
