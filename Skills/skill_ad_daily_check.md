### 確認する指標

| 指標 | 計算式 | 見る粒度 |
|------|--------|----------|
| CPA | 広告費 / 集客数 | 媒体別・日別・月別 |
| ROAS | 売上 ÷ 広告費 | 媒体別・日別・月別 |
| CPO | 広告費 / 個別予約数 | 月別 |
| 集客数 | LINE登録数等 | 日別・媒体別 |
| 個別予約数 | 個別相談予約 | 日別 |
| 広告費 | 媒体別費用 | 日別・媒体別 |
| 売上 | 着金売上 | 日別 |

## 手順

### Step 1: 数値の読み取り（AIが自動実行）

kpi_summary.json から以下を抽出してサマリを作成する。

**日次サマリ（直近3日 + 前7日平均との比較）:**
- 全体: 集客数 / 個別予約数 / 売上 / 広告費 / CPA / ROAS
- 媒体別: 同上を媒体ごとに分解
- 変化率: 前7日平均との差分（%）

**月次進捗:**
- 月間目標に対する進捗率（日数経過割合との比較）
- 着地予測（現在ペースで月末までに達成するか）

**異常フラグ:**
- kpi_anomaly_detector の結果を添付

### Step 2: 判断の提案（AIが生成 → 甲原が承認）

数値を読んで以下のフォーマットで判断案を出す。

```
【今日の広告サマリ】

■ 全体
- 昨日の集客: XX人（前7日平均比 +X%）
- 昨日のCPA: X,XXX円（目標: X,XXX円）
- 月間進捗: 売上XX% / 広告費XX%（日数経過XX%）

■ 媒体別で気になる点
- [媒体名]: CPAがXX%上昇。原因仮説: [CR疲弊 / 配信面ズレ / CPM高騰]

■ 判断提案
1. [媒体名] の [CR/オーディエンス/予算] について [継続 / 一時停止 / 予算変更]を提案
   理由: [数値根拠]
2. ...

■ アクション不要の項目
- [媒体名]: 正常範囲（CPA X,XXX円、ROAS XXX%）
```

**判断基準:**

最上位の目的は **粗利の最大化**（= 売上 − 広告費）。
広告費 = SNS広告媒体で使った金額（人件費は含まない）。

**最重要指標はROASとCPO。CPAは参考値。**

> CPAだけで判断しない。質の高い顧客層はCPAが高くても、後ろの数値（予約率・成約率・LTV）が良い。
> 逆に浅い超潜在層はCPAが安くても後ろが悪い。CPAが安い＝良いではない。
> 判断は必ずROAS・CPOとセットで行う。

#### ROAS判断（最重要 / 粗利最大化の視点）

| ROAS | 状態 | アクション |
|------|------|-----------|
| 350%超 | 余力あり | ガンガンアクセルを踏む。予算増を提案 |
| 300〜350% | 目標レンジ | 全力で踏みつつ、350%に近づける動きをする |
| 300%付近 | 限界ライン | 300%を切らないよう調整。原因特定+改善提案 |
| 300%未満 | 赤字リスク | **絶対に切らない**。固定費+中途解約返金で赤字になる。即座に調整提案 |

> **ROAS高すぎ = 良いとは限らない。** ROAS500%でも「もっと踏めるのにセーブしている」なら粗利は最大化していない。ROASを目標レンジ内に保ちながら広告費を最大化することが粗利最大化の本質。

#### CPA上昇時の診断フロー

```
CPA上昇 → CR疲弊 / 配信面ズレ / CPM高騰のどれかを確認
ROAS低下 → CPA上昇 or 成約率低下 or LTV低下を切り分け
集客数急減 → 広告予算縮小・配信制限・CR停止の可能性
売上急減   → 集客数減 or 成約率低下 or 単価変動を確認
広告費急増 → CPM高騰 or 予算設定ミスを確認
```

**重要な判断ルール（甲原の原則）:**
- 単日の数値で停止判断しない。日別は「検知」にすぎない
- 判断は合計値（3日以上の合算）で行う
- 新規CRの検証は最低3日（1日3,000円×3日）見る
- 管理画面CVと実CVの乖離は2〜3倍まで正常

### Step 3: 甲原が確認・判断

- サマリを見て、提案を承認 / 修正 / 却下
- 追加の指示があればAIに伝える

### Step 4: 実行（承認後）

- 承認された判断に基づいてアクションを記録
- 必要に応じてメンバー（おり・てる・亮太等）への指示を作成

## アウトプット

- 毎朝のLINE通知: 広告サマリ + 判断提案
- 異常時の即時通知（12:05の異常検知と連動）
- 週次での振り返りデータ蓄積（広告レポート作成スキルへの入力）

## 承認ポイント

| ポイント | 内容 |
|---------|------|
| Step 2の判断提案 | 停止・予算変更等のアクションを伴う判断は甲原が承認 |
| 承認率の目標 | 現在0% → まず50%（修正なしで使える率）→ 100%で承認不要に |
| 承認率の計測 | 「提案がそのまま使えた回数 / 全提案数」を週次で記録 |

## 既存の仕組みとの接続

| 既存の仕組み | このスキルでの活用 |
|-------------|-------------------|
| kpi_summary.json | Step 1のデータソース |
| kpi_anomaly_detector.py | Step 1の異常フラグ |
| kpi_daily_import（12:00） | データ更新のトリガー |

## エラー時の対処

| エラー | 原因 | 対処 |
|--------|------|------|
| kpi_summary.json が古い | kpi_cache_build が失敗 | 手動で `kpi_cache_build` を実行 |
| CSVデータが欠損 | Lookerセッション切れ | `looker_session_keepalive` を手動実行 |
| 媒体別データが取れない | CSV形式の変更 | `csv_sheet_sync.py` のカラムマッピングを確認 |

## 実装ステップ

### Phase 1: レポート生成（Lv.1 → 数値を見やすくまとめる）
- kpi_summary.json を読んでサマリを自動生成
- 毎朝LINE通知に載せる
- 甲原がサマリを見て自分で判断する

### Phase 2: 判断提案（Lv.2 → AIが提案、甲原が承認）
- 判断基準をルール化
- 数値に基づいて「こうした方がいい」を提案
- 承認率を計測開始

### Phase 3: 自律判断（Lv.3 → AIが判断、甲原は事後確認）
- 承認率が安定して80%以上になったら移行
- アクション実行まで自動化（メンバーへの指示含む）
